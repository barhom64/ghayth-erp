أمر فحص شامل ونهائي لنظام غيث — Technical Full Verification

تصرّف بصفتك مهندسًا معماريًا أول، ومدقق جودة برمجية، ومهندس DevOps، ومراجع أمن، ومحلل نظم تشغيلية، ونفّذ فحصًا شاملًا فعليًا لنظام غيث بالكامل، وليس فحصًا نظريًا أو شكليًا.

المطلوب ليس إضافة ميزات جديدة الآن، بل التحقق أن النظام الحالي يعمل بشكل صحيح ومترابط وآمن وقابل للتشغيل من الصفر.

1) التحقق من التشغيل من الصفر

نفّذ تشغيلًا كاملًا للنظام من بيئة نظيفة تمامًا:

Clone جديد للمستودع.
تثبيت الاعتمادات.
تشغيل PostgreSQL جديد فارغ.
استيراد db/schema.sql.
تشغيل جميع migrations.
تشغيل seed أو bootstrap.
تشغيل API.
تشغيل Frontend.
الدخول للنظام بمستخدم فعلي.
إنشاء شركة/فرع/مستخدم/صلاحيات.
التأكد أن النظام يعمل بدون الاعتماد على قاعدة بيانات قديمة أو إعدادات غير موثقة.

يجب تسليم:

أوامر التشغيل كاملة.
ملف .env.example محدث.
تقرير بأي متغير ناقص.
Screenshot أو Log يثبت نجاح التشغيل.
توثيق واضح لأي خطوة فشلت وطريقة إصلاحها.
2) فحص قاعدة البيانات

راجع قاعدة البيانات كاملًا:

db/schema.sql

توافق أسماء الأعمدة بين SQL والكود.
وجود companyId / branchId / createdAt / updatedAt / deletedAt حسب الحاجة.
دعم multi-tenant وعدم تسرب بيانات الشركات.

3) فحص API والمسارات

راجع جميع Routes في API:

كل Route له auth.
كل Route له permission مناسب.
كل mutation له validation.
كل mutation له audit.
كل mutation له event عند الحاجة.
كل mutation يحترم lifecycle/state.
كل mutation يحترم system governor/guards إذا كان ماليًا أو تشغيليًا حساسًا.
لا يوجد Route مفتوح بلا داعي.
لا يوجد endpoint تجريبي مكشوف في production.
لا يوجد استدعاء مباشر يخترق حدود المسارات.

المطلوب تقرير:

Route	Method	Module	Auth	Permission	Validation	Audit	Event	Guard	ملاحظات
