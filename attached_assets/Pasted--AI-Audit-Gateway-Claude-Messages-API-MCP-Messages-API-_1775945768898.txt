تبني AI Audit Gateway داخل غيث، وتربطه بـ Claude عبر Messages API أو عبر MCP. الـ Messages API يدعم tool use، وMCP معيار مفتوح لربط الوكلاء بالأدوات والأنظمة الخارجية. Claude Code أيضًا يدعم الاتصال بأدوات خارجية عبر MCP.

هنا تعطيه أدوات مثل:
scan_codebase_structure
list_broken_imports
check_api_frontend_alignment
find_unprotected_routes
find_missing_audit_hooks
find_translation_gaps
check_env_dependencies
run_tests
read_logs
inspect_build_errors
inspect_database_migrations
compare_routes_to_menu
find_dead_pages
find_unreachable_components

Claude لا يروح بنفسه للملفات أو الداتابيس بعشوائية؛
هو يطلب الأداة، وأنت تنفذها وترجع له النتيجة. هذا هو نمط tool use الرسمي.

أهم قاعدة

لا تجعله “يستخدم كل شيء” دفعة واحدة.
خله “يرى كل شيء” بالتدريج، و”ينفذ” على مراحل.

أنت نظامك كبير ومتشعب، فالمراجعة العامة ما تكفي.
لازم تعطيه checklists متخصصة.

مثال Checklist للواجهة
كل صفحة في القائمة لها route
كل route له component فعلي
كل component له data source
كل form له submit فعلي
كل button له action حقيقي
لا يوجد placeholder production
لا يوجد fake stats
لا يوجد text hardcoded غير مترجم
RTL سليم

Checklist للـ API
كل endpoint له validation
كل endpoint له auth
كل endpoint له permission check
كل endpoint له error handling
كل endpoint له audit إن لزم
العقود موحدة
response schema ثابت
Checklist للحوكمة
لا يوجد bypass للصلاحيات
لا يوجد تعديل بدون سجل
لا يوجد اعتماد نهائي بلا موافقة
لا يوجد cross-module breach

Checklist للتكامل
frontend field names = backend contract
backend logic = database columns
migrations = actual schema
reports = real data

نموذج دور واضح له

اكتب له دور داخلي قريب من هذا:

أنت رقيب تقني وتشغيلي لنظام غيث.
مهمتك تحليل الكود والربط والبنية والصلاحيات والتدفقات التشغيلية واكتشاف الأخطاء والنواقص والتعارضات والمخاطر.
لا تنفذ أي تعديل مباشر إلا من خلال أدوات مصرّح بها.
عند اكتشاف مشكلة، صنفها حسب الخطورة، واشرح السبب والأثر والملفات المتأثرة والإصلاح المقترح.
لا تفترض اكتمال أي ميزة لمجرد وجود واجهة لها.
اعتبر أي صفحة بلا بيانات فعلية أو مسار أو إجراء أو صلاحية أو تسجيل تدقيقي ميزة ناقصة.
أي إجراء إصلاحي يجب أن يكون بصيغة patch أو PR مقترح ما لم تصدر موافقة صريحة.

الخلاصة

نعم، تقدر تخليه الآن رقيب شامل للنظام، وهذا هو الاستخدام الأنسب في مرحلة بناء نظام جديد.
والتنفيذ الصحيح يكون هكذا:

Claude Code + GitHub لمراجعة الكود وPRs
Messages API + tool use لوكيل داخلي يقرأ ويحلل أدوات نظامك
MCP إذا تبغى توسيع الربط مع أدواتك ومصادرك بطريقة موحدة
ابدأ Read-Only
ثم تقارير
ثم اقتراحات إصلاح
ثم PRs بموافقة
ثم لاحقًا منع الدمج إذا اكتشف أخطاء حرجة
