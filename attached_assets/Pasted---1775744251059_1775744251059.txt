التقييم العام
التاسك صحيح ومطلوب، لكن فيه تفاوت كبير بين بنوده — بعضها ميكانيكي بحت وبعضها له قيمة حقيقية. إليك تفصيل كل بند:

ملاحظات بند بند
البند 1: رسائل الخادم الـ266
المشكلة الحقيقية ليست الرقم. الفاليدشن errors (400) في الكود ممتازة أصلاً:
"رصيد الإجازة غير كافٍ"
"يشترط مدة خدمة لا تقل عن X شهر"
"لقد حصلت على هذا النوع مسبقاً"
الـ500 catch blocks هي الجزء المشكلة وكلها بنفس الشكل:
typescript} catch (err) {
  res.status(500).json({ error: "خطأ في الخادم" })
}
المطلوب الحقيقي ليس 266 موضع — المطلوب شيء واحد: middleware مركزي لمعالجة الأخطاء يحدد نوع الخطأ تلقائياً من err.code:
23505 → تكرار (unique violation)
23503 → قيد مرجعي
23502 → حقل فارغ إجباري
connection → انقطاع قاعدة البيانات
بدلاً من تعديل 266 موضع يدوياً، middleware واحد يحل الكل في 30 سطر.
البند 2: الـ catch الفارغة
في dashboard.ts الـcatch (_e) {} غالباً متعمدة — بعض الجداول قد لا تكون موجودة في بيئات مختلفة فيمر الـendpoint بدون crash. هذا ليس خطأ بل defensive programming. لا تكسره، فقط أضف console.warn.
البنود الإشكالية الفعلية هي الـ.catch(() => {}) على العمليات الجانبية (إرسال إشعار، إنشاء مهمة) — هذه تُسقط أخطاء صامتة حقيقية.
البند 3: الواجهة الأمامية — هذا أسهل بند وأكبر قيمة
apiFetch يرمي new Error(err.error) بالفعل. يعني كل صفحة عندها:
typescriptonError: () => toast({ title: "حدث خطأ أثناء إنشاء المشروع" })
يكفي تغييره إلى:
typescriptonError: (err) => toast({ title: "حدث خطأ", description: err.message })
هذا تغيير ميكانيكي على 50+ صفحة لكنه يحل 80% من مشكلة تجربة المستخدم. المستخدم سيرى "رصيد الإجازة غير كافٍ" بدلاً من "حدث خطأ".
البند 4: Error Boundary
مطلوب ومهم ولكن ليس له أي خطأ في الكود الحالي يستدعيه. النظام لا يستخدم patterns تسبب runtime crashes في React (لا مكونات تقرأ undefined بدون optional chaining). هذا preventive وليس تصحيحي، ولا يُعطى وزناً كبيراً.
البند 5: معالجة أخطاء الشبكة
apiFetch لا يلتقط TypeError: Failed to fetch. هذا خطأ حقيقي — إذا انقطع الإنترنت يرى المستخدم خطأ تقني غير مفهوم. الإصلاح 5 أسطر في api.ts.

تحسينات مقترحة غير موجودة في الخطة
أولاً: error middleware مركزي في Express
أضفه في app.ts كآخر middleware:
typescriptapp.use((err, req, res, next) => {
  if (err.code === '23505') return res.status(409).json({ error: 'هذا السجل موجود مسبقاً' });
  if (err.code === '23503') return res.status(400).json({ error: 'لا يمكن الحذف: يوجد بيانات مرتبطة' });
  res.status(500).json({ error: 'خطأ داخلي في الخادم' });
});
هذا يحل البند 1 بشكل حقيقي بدلاً من تعديل 266 موضع.
ثانياً: تفريق نوع الخطأ في الـ toast
بدلاً من كل الأخطاء بنفس الأيقونة الحمراء، الخطأ يحتاج context:

validation error (400) → أصفر "تحقق من البيانات"
conflict (409) → أصفر "سجل مكرر"
network → رمادي "انقطع الاتصال"
server (500) → أحمر "خطأ في الخادم"

ثالثاً: retry تلقائي على أخطاء الشبكة
useApiQuery عنده retry: 1 — كافي. لكن useApiMutation لا يعيد المحاولة. الإجراءات (save, update) لا يجب أن تُعاد تلقائياً إلا على network timeout.

تعديل مقترح على بنية التاسك
احذف البند 1 كما هو واستبدله بـ:

إنشاء Express error middleware مركزي في app.ts يعالج أكواد قاعدة البيانات (23505, 23503, 23502) وأخطاء الشبكة وأخطاء الصلاحيات بدلاً من تعديل 266 catch block يدوياً.

اجمع البند 2 مع البند 5 في "تنظيف الأخطاء الصامتة" — هي نفس الفكرة.
أعط البند 3 الأولوية الأولى — أسرع تأثير وأوضح قيمة للمستخدم.

الخلاصة
التاسك كما هو يحتاج تعديلاً على نقطتين:
البند 1 يُعاد صياغته بالـ middleware بدلاً من 266 تعديل يدوي. والأولويات تُعاد ترتيبها: البند 3 أولاً ثم البند 5 ثم البند 4 ثم الـ middleware.
كل ما تبقى صحيح ومنطقي ويستحق التنفيذ.
