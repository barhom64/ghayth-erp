الصورة الكاملة الآن. الملاحظات مرتبة من الأكثر خطورة إلى الأقل:

المستوى الأول — ثغرات تُوقف النظام أو تُفسد البيانات
1. الرواتب ليست في Transaction — خطر مالي حقيقي

// الكود الحالي في hr.ts سطر 1457:
const { insertId: runId } = await rawExecute(`INSERT INTO payroll_runs...`);
await Promise.all(lines.map(l => rawExecute(`INSERT INTO payroll_lines...`)));
await rawExecute(`UPDATE attendance_deductions...`).catch(() => {}); // ← صامت


إذا فشل أي INSERT في المنتصف: runId موجود، بعض الموظفين مُسجَّلون، باقيهم لا. والـ catch(() => {}) على تحديث الخصومات يعني الموظف يُخصم مرة ثانية في الشهر التالي. الحل: كل عملية الرواتب داخل withTransaction واحدة.
2. بدلات السكن والنقل مُثبَّتة بالكود — تتجاهل عقود الموظفين

// hr.ts سطر 1409 — لكل موظف في كل شركة:
const housingAllowance = basic * 0.25;  // ← ثابت للجميع
const transportAllowance = basic * 0.10; // ← ثابت للجميع


جدول salary_components موجود ومُعبَّأ لكن لا يُستخدم في حساب الرواتب أبداً. موظف براتب 20,000 يأخذ 5,000 سكن تلقائياً بغض النظر عن عقده. الحل: الرواتب تقرأ من salary_components وتحترم نوع كل بند (fixed / percentage / conditional).
3. الحذف المادي على سجلات محاسبية

DELETE FROM invoices WHERE id = $1        -- hr.ts سطر 2931
DELETE FROM payroll_runs WHERE id = $1    -- hr.ts سطر 2272
DELETE FROM journal_entries WHERE id = $1 -- finance.ts سطر 2970


حذف فاتورة أو مسير رواتب لا يترك أثراً — القيد المحاسبي المقابل يبقى معلقاً بدون مستند. هذا يكسر مبدأ Audit Grade الذي يفترضه النظام. الحل: deletedAt على كل كيان مالي، لا حذف أبداً.

المستوى الثاني — ثغرات أمنية جوهرية
4. CORS مفتوح لأي موقع

app.use(cors()); // ← يقبل أي origin


يعني أي موقع خارجي يقدر يرسل طلبات للـ API باسم المستخدم. الحل: cors({ origin: ['https://hr.door.sa'] }).
5. لا rate limiting — الـ API مكشوف للـ brute force
لا يوجد أي حماية على /auth/login. شخص يقدر يجرب ملايين كلمات مرور بدون توقف. express-rate-limit غائب تماماً من app.ts.
6. JWT Secret يتغير عند كل restart

const SECRET = process.env.JWT_SECRET || (() => {
  return crypto.randomBytes(64).toString("hex"); // ← جديد عند كل restart
})();


كل إعادة تشغيل للسيرفر = كل المستخدمين يُطرَدون فوراً. في بيئة Replit يحدث هذا كثيراً. الحل: JWT_SECRET إجباري في .env، إذا غاب: السيرفر لا يُشغَّل.
7. لا refresh token — JWT مدته 7 أيام كاملة
إذا سُرق token المستخدم، المهاجم يملكه 7 أيام كاملة بدون طريقة للإلغاء. النظام لا يملك آلية logout حقيقية تُبطل الـ token.

المستوى الثالث — وظائف مُعلَنة لكنها لا تعمل
8. WhatsApp وSMS لا يُرسلان شيئاً

// integrationService.ts سطر 128-131:
case "sms":
  result = { success: false, error: "SMS integration not yet implemented" };
case "whatsapp":
  result = { success: false, error: "WhatsApp integration not yet implemented" };


كل سيناريوهات النظام تقول “يُرسل واتساب للعميل” و”SMS للمستأجر” — لكن هذه الأسطر تثبت أن لا شيء يُرسَل فعلاً. الإشعارات تُحفَظ في قاعدة البيانات فقط ولا تصل لأحد. هذا يُبطل قيمة كاملة من النظام.
9. الإيميل يعمل فقط إذا كان SMTP مُعدَّاً يدوياً
sendEmail يستخدم nodemailer لكن لا يوجد fallback. إذا لم تُعدَّ بيانات SMTP في جدول integrations — كل الإشعارات صامتة.

المستوى الرابع — ديون تقنية تتراكم
10. تعارض أسماء migrations — يسبب crash عند النشر

012_audit_enhanced.sql
012_dms_enhancement.sql    ← تعارض
012_job_titles.sql         ← تعارض
012_workflow_engine.sql    ← تعارض

013_business_rules.sql
013_integrations.sql       ← تعارض
013_template_center.sql    ← تعارض


نظام الـ migration يُشغّلها بالترتيب الأبجدي. إذا كان يتتبع بالرقم سيُنفِّذ واحدة ويتجاهل الباقي. هذا يعني جداول ووظائف كاملة قد لا تكون موجودة في بيئة الإنتاج.
11. finance.ts = 3882 سطر في ملف واحد
هذا الملف يحتوي: فواتير + مصروفات + ميزانية + P2P + قيود + تحصيل + ضريبة + سلف + عهد + تقارير. أي تعديل يخاطر بكسر كل المالية. يجب التقسيم: finance-invoices.ts + finance-journal.ts + finance-purchase.ts.
12. Cron jobs بدون mutex — يتداخلون
26+ وظيفة Cron، بعضها يشتغل “كل ساعة”. إذا استغرقت الوظيفة أكثر من ساعة ستبدأ نسخة ثانية منها والبيانات تُعالَج مرتين. لا يوجد cron_job_lock يمنع التداخل.

المستوى الخامس — فجوات وظيفية حقيقية
13. لا multi-tenancy حقيقي على مستوى قاعدة البيانات
كل الجداول في نفس الـ schema. فصل البيانات يعتمد فقط على WHERE companyId = $1 في كل استعلام. خطأ واحد في استعلام واحد = موظف يرى بيانات شركة أخرى. الحل الصحيح: PostgreSQL Row-Level Security (RLS) يفرض العزل على مستوى قاعدة البيانات نفسها.
14. لا نسخ احتياطي مُدار من النظام
لا يوجد endpoint أو Cron لعمل database dump. لا يوجد export لبيانات الشركة. إذا تعطل السيرفر — البيانات قد تضيع بالكامل. هذا يجب أن يكون ضمن admin.ts كأولوية.
15. بدلات الموظف لا تُراعي تاريخ التغيير
إذا تغيّر راتب موظف في منتصف الشهر، النظام يحسب الراتب كاملاً بالراتب الجديد. لا يوجد effectiveDate على تغييرات الراتب يُحسب منه الشهر الجزئي.
16. لا نظام archive للسجلات القديمة
بعد سنوات: جدول attendance سيحتوي ملايين السجلات، audit_logs أكثر. لا يوجد archiving policy أو partitioning. الأداء سيتدهور تدريجياً بدون أن يُلاحظه أحد حتى يصبح النظام بطيئاً.
