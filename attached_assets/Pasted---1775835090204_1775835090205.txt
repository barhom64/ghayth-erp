بناءً على الكود الفعلي وطبيعة عمل الدور — هذه الأفكار الأقوى:

أولاً: مسار العمرة — بناء من الصفر وفق مبادئ غيث
الفكرة الجوهرية: “المعتمر كـ Ticket”
بدلاً من جدول بيانات، كل معتمر هو ticket حي له lifecycle كامل داخل غيث:

Season → Group → Pilgrim → Status → Event → Impact


قاعدة البيانات المقترحة:

umrah_seasons       -- موسم رمضان 2026، العمرة الاعتيادية، حج
umrah_groups        -- مجموعة الوكيل A - 50 معتمر
pilgrims            -- المعتمر + جواز + تأشيرة + حالة حية
pilgrim_movements   -- كل حدث: وصول، خروج، غياب، مخالفة
pilgrim_violations  -- غرامة + سبب + ربط بالوكيل
agent_invoices      -- فاتورة الوكيل = تأشيرات + غرامات + خدمات
visa_purchases      -- شراء التأشيرات من نسك + تكلفة الوحدة


محرك الحالات — Job يومي الساعة 2 صباحاً:

للمعتمر:
  اليوم > expectedDeparture + 0 يوم → يحسب تجاوز
  اليوم > expectedDeparture + 3 أيام → تنبيه حرج + فاتورة غرامة
  تم تسجيل الخروج → Departed → إغلاق ملفه

للوكيل:
  مجموعه فيها 3+ متأخرين → تنبيه عاجل + تجميد استيراد جديد
  نسبة الإنجاز < 80% → تحذير في لوحة التشغيل


محرك الاستيراد الذكي — الفكرة الأقوى:
الملف من نسك يحتوي عادةً بيانات مكررة يومياً. المحرك يعمل هكذا:

ImportEngine:
  1. قراءة Excel → normalize (توحيد التواريخ والأسماء والجوازات)
  2. لكل صف: SELECT WHERE passportNo + visaNo
     - موجود → UPDATE إذا تغير الخروج المتوقع فقط
     - جديد → INSERT + إنشاء ملف المعتمر
     - تعارض (نفس جواز + تأشيرة مختلفة) → يدخل "قائمة المراجعة"
  3. بعد الاستيراد: تقرير فوري
     - X جديد | Y محدّث | Z تعارض | N مكرر
  4. إطلاق event: IMPORT_COMPLETED
     → تحديث لوحة التشغيل فوراً


فاتورة الوكيل — الربط المحاسبي الكامل:

فاتورة الوكيل تُنشأ تلقائياً عند:
  - نهاية الأسبوع (للمتجاوزين)
  - طلب يدوي
  - إغلاق الموسم

تشمل:
  تأشيرات: عدد × سعر البيع للوكيل
  غرامات:  تجاوز × سعر اليوم
  نقل:     عدد الرحلات × السعر
  خصم:     دفعات مسبقة

القيد المحاسبي التلقائي:
  مدين: ذمم الوكيل
  دائن: إيراد تأشيرات + إيراد غرامات + إيراد نقل


ثانياً: الفكرة الأقوى في المشروع كله — “الرابط الثلاثي”
الأقوى ليس في مسار واحد بل في الربط بين ثلاثة مسارات بحدث واحد:
مثال: تجاوز مدة المعتمر

Event: PILGRIM_OVERSTAYED (pilgrimId, agentId, days, companyId)

→ HR:       إنشاء مهمة لموظف التحصيل في تقويمه
→ Finance:  إنشاء غرامة + قيد مدين في حساب الوكيل
→ Legal:    إذا days > 7 → إنشاء ملف قانوني تلقائياً
→ CRM:      تحديث تصنيف الوكيل (reliable → at_risk)
→ Notify:   واتساب للوكيل + إشعار داخلي للمدير


حدث واحد يُطلق 5 سلاسل في 5 وحدات مختلفة — هذا هو الـ Event System الحقيقي.

ثالثاً: لوحة التشغيل اليومية — “غرفة العمليات”
بدلاً من لوحة إحصاء تقليدية، الفكرة هي لوحة قرار:

كل بطاقة فيها:
  - رقم + حالة (أخضر/أصفر/أحمر)
  - السبب إذا كانت غير خضراء
  - زر إجراء مباشر

مثال اليوم:

🔴 15 معتمر متأخر أكثر من 3 أيام
   ← [إنشاء فواتير جماعية] ← [إشعار الوكلاء]

🟡 8 عقود إيجار تنتهي خلال 30 يوم
   ← [مراجعة التجديدات]

🟢 التحصيل اليوم: 125,000 ريال (107% من المستهدف)

🔴 SLA الصيانة: 3 بلاغات تجاوزت 4 ساعات
   ← [إعادة توزيع تلقائي]


اللوحة لا تعرض أرقاماً فقط — تعرض ما يجب أن يحدث الآن.

رابعاً: محرك التوقع — “النظام يرى قبلك”
الفكرة: غيث يملك كل البيانات التاريخية، يجب أن يُنتج توقعات لا أرقاماً فقط:
توقع التدفق النقدي 30 يوماً:

// بناءً على:
// - فواتير مستحقة + نسبة التحصيل التاريخية لكل عميل
// - إيجارات مجدولة + نسبة التأخير التاريخية
// - رواتب مجدولة + تعويضات متوقعة

function forecastCashFlow(companyId, days = 30) {
  const receivables = getExpectedCollections(companyId, days);
  const payables = getScheduledPayments(companyId, days);
  const risk = calculateCollectionRisk(receivables);
  
  return {
    expectedInflow: sum(receivables),
    expectedOutflow: sum(payables),
    netCashFlow: expectedInflow - expectedOutflow,
    riskAdjusted: expectedInflow * (1 - risk),
    alert: riskAdjusted < 0 ? "تحذير سيولة" : null
  }
}


توقع ضغط العمرة:

النظام يقارن:
  - أيام الموسم الحالي مقابل نفس الأيام الموسم الماضي
  - نسبة الدخول اليومية
  - توقع ذروة المتأخرين

→ "الذروة متوقعة بعد 8 أيام — يُنصح بتخصيص موظف إضافي"


خامساً: تعدد العملات — الفجوة الحقيقية الوحيدة
لأن الوكلاء من دول مختلفة (إندونيسيا، باكستان، تركيا، ماليزيا):

-- جدول أسعار الصرف اليومية
exchange_rates (
  fromCurrency, toCurrency, rate, effectiveDate, source
)

-- كل فاتورة وكيل بعملته
agent_invoices (
  agentCurrency VARCHAR(3),
  totalAgentCurrency DECIMAL,
  exchangeRate DECIMAL,
  totalSAR DECIMAL,  -- للمحاسبة الداخلية دائماً بالريال
  rateDate DATE
)

-- محرك التحديث اليومي
CronJob: fetchExchangeRates()
  → استدعاء API مفتوح (exchangerate-api.com)
  → تحديث exchange_rates
  → إذا تغير الريال أكثر من 2% → تنبيه للمالي


القاعدة الجوهرية: كل شيء داخلياً بالريال، العرض الخارجي بعملة الوكيل.

سادساً: نظام إغلاق الموسم — “قفل كامل بـ Checklist”
قبل إغلاق أي موسم، النظام يرفض الإغلاق إلا بعد اجتياز 7 شروط:

[ ] كل المعتمرين في حالة Departed أو Violated
[ ] جميع الغرامات لها فواتير مرتبطة
[ ] نسبة تحصيل فواتير الوكلاء > 90%
[ ] لا يوجد مهام مفتوحة مرتبطة بالموسم
[ ] تقرير الموسم مُنتَج ومعتمد
[ ] القيود المحاسبية مرحّلة
[ ] الميزانية السنوية محدّثة بأرقام الموسم

→ اجتاز الكل → يُسمح بالإغلاق + إنشاء الموسم التالي تلقائياً
→ فيه بند لم يجتز → يعرض السبب + الإجراء المطلوب


سابعاً: الذكاء على مستوى الوكيل — “تصنيف الوكلاء”
مثل RFM للعملاء لكن مخصص للوكلاء:

Agent Score = 
  نسبة الالتزام بالخروج (40%) +
  سرعة السداد (30%) +
  نسبة المشاكل/العدد (20%) +
  حجم العمل (10%)

التصنيف:
  90-100: Strategic Partner → سعر مميز + أولوية تأشيرات
  70-89:  Reliable Agent   → معاملة عادية
  50-69:  Watch List       → ضمان مسبق + متابعة أشد
  <50:    Restricted       → حظر جزئي + موافقة يدوية

Cron أسبوعي يُحدّث التصنيف ويُنشئ تنبيهاً عند التغيير


ثامناً: الفكرة الأذكى — “Simulation قبل الإغلاق”
قبل أي إجراء كبير (إغلاق موسم، تشغيل رواتب، اعتماد ميزانية)، النظام يُشغّل simulation ويعرض النتيجة:

"لو أغلقنا الموسم الآن:
  ✅ سيتم إصدار 23 فاتورة نهائية بمجموع 1.2M ريال
  ✅ سيتم إغلاق 847 ملف معتمر
  ⚠️  12 معتمر لا يزالون داخل المملكة
  ❌ 3 فواتير وكلاء غير مسددة بمجموع 45,000 ريال

هل تريد المتابعة؟"


هذا موجود جزئياً في impactPreview.ts لكن يحتاج توسعة لتشمل الإجراءات الجماعية.

الخلاصة — ترتيب التنفيذ

المرحلة 1 (الأساس): 
  قاعدة بيانات العمرة + محرك الاستيراد + محرك الحالات اليومي

المرحلة 2 (الربط):
  Event: PILGRIM_OVERSTAYED → Finance + HR + Legal + CRM
  فاتورة الوكيل التلقائية + القيد المحاسبي

المرحلة 3 (الذكاء):
  تصنيف الوكلاء + توقع التدفق النقدي + لوحة القرار

المرحلة 4 (الإكمال):
  تعدد العملات + إغلاق الموسم بالـ Checklist
  Simulation للإجراءات الكبيرة


كل هذا يبنى فوق غيث وليس بجانبه — نفس Event Bus، نفس companyBootstrap، نفس Audit، نفس Notifications. المسار الجديد يتبع نفس مبادئ المسارات المستقلة المترابطة.​​​​​​​​​​​​​​​​
