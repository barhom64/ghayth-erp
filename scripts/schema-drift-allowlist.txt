# scripts/schema-drift-allowlist.txt
#
# Schema-drift scanner exceptions — see scripts/src/check-schema-drift.mjs.
#
# Each line is one of:
#   routes/<file>.ts                  → skip every finding in that file
#   routes/<file>.ts:<id>             → skip only that identifier in the file
#
# `#` starts a comment; blank lines are ignored. Paths are relative to
# artifacts/api-server/src/, so artifacts/api-server/src/routes/foo.ts is
# `routes/foo.ts`.
#
# Use sparingly. The scanner exists to catch a route referencing a column
# or table that does NOT exist in the database (the route → missing-column
# 500 class). The correct fix is almost always option 1 in the scanner's
# failure message: add the migration (or fix the typo).
#
# ── Why these entries exist ────────────────────────────────────────────
# Every identifier below was verified to EXIST in the live production
# database; it is absent only from the committed db/schema*.sql dump that
# CI loads into its harness. The dump lags behind several pre-cutoff
# migrations (e.g. 223/224/234/236/242/249) whose tables/columns were
# never folded back into the dump, and db/.baseline-cutoff (258) marks
# those migrations "applied" without running them — so CI's harness is
# missing objects that production has. Refreshing the dump in this PR is
# not viable: db/schema_pre.sql is a hot file (rewritten by nearly every
# feature merge) so a frozen refresh would perpetually conflict on
# update-branch, and pg_dump renormalises literals (NOT IN (...) ->
# <> ALL (ARRAY[...])) which breaks committed dump-pin tests. These are
# therefore baselined here until the committed dump is regenerated.
# ───────────────────────────────────────────────────────────────────────
