## الدفعة (أ) — F3: فصل التشغيل اليدوي للفاتورة المتكررة عن الدورة المجدوَلة

### العيب (مؤكَّد من الكود)
`runRecurringInvoice({ force })` في `recurringInvoiceProcessor.ts`:
- `runDate = tpl.nextRunDate` (السطر 79) ← مفتاح الـidempotency = `…:${runDate}` (80).
- بعد التوليد: `nextRunDate = computeNextRunDate(runDate, freq)` (114) — **يقدّم الجدول دائمًا**.
⇒ تشغيل يدوي (`force`) **قبل الاستحقاق** يُولّد فاتورة مؤرّخة باليوم، لكنه يحجز مفتاح `nextRunDate` المستقبلي **ويقدّم الجدول** → الفاتورة المجدوَلة لاحقًا لنفس التاريخ **تُحجَب**، وتُستهلَك الدورة.

### القرار الدلالي المطلوب من إبراهيم (نعم/لا — لا يُحسَم من الكود)
ماذا يعني «تشغيل الآن» يدويًّا قبل الاستحقاق؟
- **(تفسير 1) «شغّل هذه الدورة الآن»** → يقدّم الجدول (السلوك الحالي). مناسب لمن يريد تبكير الدورة.
- **(تفسير 2) «أصدر فاتورة إضافية الآن»** → لا يقدّم الجدول؛ الدورة المجدوَلة تبقى. مناسب لفاتورة عرضية.

### التصميم المقترح (يدعم التفسيرين بوضوح، الافتراض الآمن = 2)
وسيط `mode` صريح بدل `force` الغامض:
- `mode: "scheduled"` (افتراضي للـcron + الزر «شغّل المستحق»): يعمل فقط إن `nextRunDate <= today`؛ المفتاح على `nextRunDate`؛ **يقدّم** الجدول. (السلوك الحالي الصحيح للمستحق.)
- `mode: "adhoc"` (الزر اليدوي «أصدر فاتورة الآن»): يعمل في أي وقت؛ المفتاح `…:adhoc:${today}`؛ مؤرّخة باليوم؛ **لا يقدّم** `nextRunDate` ولا يحجب الدورة المجدوَلة.
- توافق خلفي: `force:true` الحالي → يُترجَم إلى `adhoc` (الافتراض الأسلم — لا يستهلك الدورة).

### assertion tests إلزامية (mocked، لا DB)
1. `adhoc` قبل الاستحقاق → فاتورة مؤرّخة باليوم، مفتاح `adhoc:today`، **`nextRunDate` لم يتغيّر**.
2. `scheduled` عند الاستحقاق → مفتاح `nextRunDate`، **`nextRunDate` تقدّم** بـ`computeNextRunDate`.
3. idempotency: إعادة `adhoc` نفس اليوم → لا تكرار (نفس sourceKey).
4. القيد عبر `postSalesInvoice` متوازن وسطوره صحيحة (لا منطق قيد جديد).

### الخطر
متوسط (يغيّر دلالة زر موجود). صفر منطق قيد جديد (يعيد استخدام `postSalesInvoice`). **يتطلب اعتمادك للتفسير (1 أم 2) + الاختبارات أعلاه خضراء.**

## الدفعة (ب) — تطبيع إشارة سطور القيد في `postJournalEntry` (دفاع مركزي)

### الواقع (مؤكَّد من الكود)
`postJournalEntry` (`financialEngine.ts`) **يخزّن debit/credit كما هي** (INSERT السطر 394، بلا تطبيع)، ويتحقق من التوازن عبر حساب وسيط 9999 (369–408). تدفقات داخلية **تُرحّل سالبًا عمدًا**:
- `finance-journal.ts:3565/3576/3583` (إقفال الفترة: `credit:-bal`, `debit:-bal`, `debit:-netIncome`).
- `propertiesEngine.ts:603` (`debit:-gainOrLoss`) · `hrEngine.ts:515` (`debit:-residual`).
سالب الـdebit وقت التطبيق **يعكس حركة الحساب** (= credit فعليًّا)، فتُشوَّه إجماليات المدين/الدائن والتقارير. ⇒ رفضٌ ساذج للسالب **يكسر هذه التدفقات** (لذا لم يُطبَّق في F9-B2 إلا عند حدود الإدخال).

### التصميم المقترح — تطبيع لا رفض (لا يكسر شيئًا)
عند مدخل `postJournalEntry`، قبل التوازن والكتابة، لكل سطر:
```
if (line.debit < 0)  { line.credit += -line.debit;  line.debit = 0; }
if (line.credit < 0) { line.debit  += -line.credit; line.credit = 0; }
```
- **الأثر المحاسبي مطابق تمامًا:** صافي السطر (debit − credit) ثابت؛ `debit:-100/credit:0` ⇒ `debit:0/credit:100` (نفس الحركة).
- **يختفي التخزين السالب** → إجماليات/تقارير صحيحة + سطور قيد قياسية.
- **لا يكسر التدفقات الداخلية** (إقفال/تخلص أصل/استقطاع تُرحّل نفس الأرصدة على العمود الصحيح).
- يجعل حُرّاس الإدخال (B2) دفاعًا في العمق؛ المحرّك يصبح صلبًا تجاه أي إشارة.
- حالة `debit<0 && credit>0` (نادرة): تُدمَج صحيحًا (credit_new = credit + |debit|).

### assertion tests إلزامية (mocked، على سطور القيد)
1. سطر `debit:-100, credit:0` → بعد التطبيع `debit:0, credit:100`؛ صافي الحساب مطابق؛ القيد متوازن.
2. قيد بنمط إقفال فترة برِبح (`debit:-netIncome`) → يُرحّل نفس الأرصدة بعد التطبيع.
3. قيد موجب عادي → بلا تغيير.
4. التوازن (Σdebit = Σcredit) صحيح بعد التطبيع.
5. تكامل: اختبارات الدفتر القائمة (glPosting/ledgerTruth/payroll/close) كلها خضراء.

### الخطر — عالٍ (تغيير محرّك مركزي يمسّ **كل** ترحيل)
- يتطلب اعتمادك الصريح **للنهج** + الاختبارات أعلاه + اجتياز كامل مجموعة اختبارات الدفتر.
- **السطور السالبة التاريخية في DB تبقى كما هي** (التطبيع للترحيل الجديد فقط). تنظيف السجل التاريخي = دفعة data-fix منفصلة اختيارية (تقرير قراءة-فقط أولًا) — **لا تُنفَّذ ضمن هذه الدفعة**.
