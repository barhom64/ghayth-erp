## البند 1 — حالة مراجعة المرفق (موصى به أولًا)

### لماذا
المراجع يحتاج أن يقول لكل مرفق: **مقبول / مرفوض / يحتاج استبدال / مكرر**، ويترك ملاحظة، وأن يُسجَّل القرار في Audit. اليوم `documents.status` حالة دورة حياة المستند (مسودة/معتمد…) **لا** حكم مراجعة، وقد يُربط نفس المستند بعدة كيانات بأحكام مختلفة.

### القرار المعماري: أين تُخزَّن الحالة؟
- **الموصى به: على `document_entity_links`** (حكم المراجعة يخص ربط المستند بكيان معيّن، لا المستند عالميًا) — يحترم أن نفس الملف قد يُقبل في كيان ويُرفض في آخر.
- البديل (مرفوض): على `documents` مباشرة — يكسر تعدد الروابط.

### الأعمدة المقترحة على `document_entity_links`
| العمود | النوع | ملاحظة |
|---|---|---|
| `reviewStatus` | VARCHAR(20) DEFAULT 'new' | enum: `new`, `accepted`, `rejected`, `needs_replacement`, `duplicate` |
| `reviewedBy` | INTEGER NULL (FK users) | من اتخذ القرار |
| `reviewedAt` | TIMESTAMPTZ NULL | متى |
| `reviewNote` | TEXT NULL | سبب — **إلزامي تطبيقيًا عند rejected/needs_replacement** |

> «منتهي» يبقى مُشتقًّا من `documents.retentionUntil` (لا عمود جديد) — مُنفَّذ عرضًا في الدفعة F1.
> «مكرر» (`duplicate`) حكم يدوي من المراجع الآن؛ الكشف الآلي للمكرر = البند 4 (مؤجَّل).

### الجداول المتأثرة
- `document_entity_links` (تعديل: 4 أعمدة).
- `audit_log` / `approval_actions` (إدراج أثر: actor + action=`attachment_reviewed` + note).
- لا تغيير على `documents` أو `document_versions`.

### RBAC / Audit / العقود (إلزامية)
- صلاحية جديدة محدودة: `documents:review` (أو إعادة استخدام صلاحية الاعتماد للكيان المالك). **لا توسيع** خارج ذلك.
- كل قرار يصدر Audit إلزامي + يبقى `check:audit-coverage` أخضر.
- endpoint جديد: `PATCH /documents/:id/links/:linkId/review` بـ`{ reviewStatus, reviewNote? }`، يفرض الصلاحية ويرفض note فارغ عند rejected/needs_replacement.

### واجهة (جاهزة للاستقبال)
- `attachment-preview` (الدفعة A) + `entity-documents` (الدفعة F1) جاهزان لعرض الحالة والملاحظة فور توفّر الحقول؛ يُضاف وضع `review` وأزرار قبول/رفض/طلب استبدال حينها.

### اختبار القبول
- assertion: قرار رفض بلا note → 400. قرار يصدر Audit واحدًا. تعدد الروابط: قبول في كيان لا يغيّر حالة الربط في كيان آخر.

## البند 2 — قائمة المتطلبات الإلزامية + بطاقة الاكتمال

### لماذا
المقدم/المراجع يحتاج أن يرى «المطلوب: هوية ✓ / عقد موقّع ✗ / شهادة ضريبية ✓» قبل الحفظ/الاعتماد، بدل اكتشاف النقص متأخرًا.

### المقترح
- جدول إعداد `document_requirements`: `(id, entityType, docCategory, label, required BOOLEAN, companyId, isActive)` — يُضبط من إعدادات المسار المالك (وراثة نظام←شركة←فرع حسب المبدأ 6).
- بطاقة الاكتمال **مشتقّة** (لا تخزين): تقاطع متطلبات الكيان مع روابط مستنداته الفعلية.
- لا منطق عابر: كل مسار قائد يملك متطلباته؛ العرض موحّد عبر مكوّن واجهة يقرأ الإعداد.

### الجداول المتأثرة
- جدول جديد واحد `document_requirements` + Audit عند تعديل الإعداد.

### قرار مطلوب
- إنشاء جدول جديد ⇒ يحتاج «نعم». البديل الأخف: تعريف المتطلبات في إعدادات المسار القائمة (JSON config) بلا جدول — **أوصي بالجدول** لقابلية الضبط والتدقيق.

## البند 3 — معاينة أثر القرار («عند الاعتماد سيتم…»)

### لماذا
قبل الاعتماد يرى المراجع الأثر: «قيد مالي + إشعار + قفل تعديل + ربط مستندات». هذا روح غيث (السياق قبل الفعل، المادة 4).

### المقترح (بلا جدول)
- **اشتقاق** الأثر من عقود الأحداث/الأفعال التلقائية القائمة لكل مسار (Event System، ملف 08) — لا بيانات جديدة، بل واجهة تقرأ «ما سيُطلقه الحفظ/الاعتماد» وتعرضه في `reviewPanel` (فتحة الدفعة G جاهزة).
- المخاطرة: إن لم تكن الآثار مُصرَّحة بنية قابلة للقراءة، فقد يلزم عقد وصف خفيف لكل إجراء — **حدّي معماريًا** ⇒ قرار إبراهيم.

### قرار مطلوب
- نعم/لا على بناء «عقد وصف الأثر» للمسارات، أو الاكتفاء بنص ثابت لكل إجراء يحرّره مالك المسار.

## البند 4 — مؤجّل (طباعة مجمّعة / تعليق على مرفق-حقل / كشف المكرر)
- طباعة السجل + المرفقات كملف واحد: تجميع PDF خادمي — جهد backend مستقل.
- تعليق على مرفق/حقل محدد: ربط تعليق بكيان فرعي — يحتاج بنية تعليقات موسّعة.
- كشف «مكرر» آليًا: تجزئة/مقارنة ملفات — مستقل.
- توصية: تأجيلها حتى اعتماد 1–3 وقياس الحاجة.
