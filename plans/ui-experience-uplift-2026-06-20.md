# خطة ترقية تجربة الواجهات فوق المكوّنات المشتركة القائمة

> **التوجيه:** رفع جودة تجربة المستخدم في غيث (مقدم/مراجع، استعراض المرفقات، النماذج، الجداول، صفحات التفاصيل، الأدراج) **عبر تعديل المكوّنات المشتركة القائمة فقط** — لا مسار جديد، لا خدمة جديدة، لا منطق منقول بين المسارات.
> **التاريخ:** 2026-06-20 · **المدير:** الجلسة الرئيسية (وضع تعميم التوجيهات) · **الحالة:** ⏸️ بانتظار اعتماد النطاق ونقطة البدء من إبراهيم.

---

## 0) المبدأ الحاكم لهذه الخطة

- ممنوع إنشاء مكوّن/خدمة/مسار جديد إن غطّى الموجود الغرض (قاعدة 5).
- البناء **فوق**: `entity-documents` · `attachment-preview` · `file-drop-zone` · `FormShell` · `DataTable` · `PageShell` · `DetailPageLayout` · `AllowCreateDrawer` · `navigation.registry` · `empty`/`page-state`.
- كل تعديل على مكوّن مشترك = **prop اختياري متوافق رجعيًا** بقيمة افتراضية تُبقي السلوك الحالي لكل المستهلِكين (300+ صفحة) دون كسر.
- ⛔ لا تطبيق شامل دفعة واحدة — **دفعة دفعة**، كل دفعة ببوّابتها وحكم مجلس قبل إعلان نجاحها.
- ⛔ أي بند يحتاج عمودًا/جدولًا/migration أو منطق أعمال جديدًا = **وقف وتقرير لإبراهيم (نعم/لا)**، لا يُنفَّذ ضمن هذه الخطة.

## 1) المسح (Phase 2 — قراءة فقط) — حقائق الكود الفعلي

المكوّنات تحت `artifacts/ghayth-erp/src/components`:

| المكوّن | الملف | الواجهة الحالية (مختصر) |
|---|---|---|
| EntityDocuments | `shared/entity-documents.tsx` | `{ entityType, entityId, title? }` — قائمة مسطّحة + رفع + معاينة. تستهلك `status` (draft/active/approved) و`currentVersion` و`uploaderName` (⚠️ لا يرجعه الـAPI). |
| AttachmentPreview | `shared/attachment-preview.tsx` | لوحة جانبية: صورة مباشرة + PDF عبر `<object>` + تحميل. لا تكبير/تدوير. |
| FileDropZone | `shared/file-drop-zone.tsx` | `{ files, onFilesChange, label?, maxSizeMB? }` — سحب/إفلات → base64. |
| FormShell | `form-shell.tsx` | `{ schema, defaultValues, onSubmit, submitLabel?, secondaryActions?, ... }` — لا أقسام قابلة للطي، لا ملخص جانبي. |
| DataTable | `ui/data-table.tsx` | غني: أعمدة/فرز/بحث/تحديد/ترقيم/تجميع/`statusOptions`/`emptyAction`. لا «عمود الخطوة التالية» ولا «إجراءات حسب الحالة» مدمجة. |
| PageShell | `page-shell.tsx` | `{ title, subtitle, breadcrumbs?, actions?, filters?, ... }` — لا فتحة ملخص. |
| DetailPageLayout | `shared/detail-page-layout.tsx` | تبويبات قياسية (overview/documents/timeline/comments/tasks/audit) + `extraTabs` + `actions`. **لا** وضع مقدم/مراجع، **لا** لوحة مراجعة، **لا** معاينة أثر. |
| AllowCreateDrawer | `shared/allow-create-drawer.tsx` | درج يحمّل النموذج الكامل (`ENTITY_CREATE_FORMS`). لا رأس سياق، لا تأكيد بعد الحفظ، لا زر «الصفحة الكاملة». |
| Empty / PageStateWrapper | `ui/empty.tsx` · `shared/page-state.tsx` | `emptyText/emptyHint/emptyAction` متاح أصلًا — يحتاج رسائل إرشادية أفضل فقط. |

### نموذج بيانات الوثائق (حاسم للتصنيف)
- `documents.status` ∈ {draft, active, approved, cancelled, archived} فقط. **لا** حالة مراجعة (مقبول/مرفوض/يحتاج استبدال/مكرر).
- **لا** عمود `reviewer_note` على المستند.
- `document_versions` موجود (إصدارات حقيقية) ✅ — عرضها ممكن الآن.
- `documents.retentionUntil` موجود ✅ — اشتقاق «منتهي» ممكن الآن.
- `uploaderName` تتوقّعه الواجهة ولا يرجعه الـAPI ⇒ JOIN ناقص (ثغرة قائمة).
- الاعتماد/الرفض/الإرجاع: نمط لكل مسار (`PATCH .../approve` بـ`{approved, reason}`) + `approval_actions.notes`. أي لوحة مراجعة تربط بهذا، لا تنشئه.

## 2) التصنيف: تشغيلي عرض-فقط مقابل يتطلّب قاعدة بيانات

### (أ) تشغيلي — إضافات متوافقة رجعيًا، بلا قاعدة بيانات (داخل نطاق هذه الخطة)
كلها props اختيارية على مكوّن مشترك، السلوك الافتراضي = الحالي.

### (ب) يتطلّب migration/منطق نطاق جديد — ⛔ وقف وتقرير لإبراهيم (خارج التنفيذ الآن)
- **حالة مراجعة المرفق** (مقبول/مرفوض/يحتاج استبدال/مكرر): عمود/enum جديد + Audit + تدفّق قرار per-attachment. (قاعدة 3 دستور + قاعدة 2 migration).
- **ملاحظات المراجع على المرفق**: عمود `reviewer_note` جديد.
- **قبول/رفض/طلب استبدال لكل مرفق**: تدفّق نطاق جديد على خدمة الوثائق.
- **قائمة المتطلبات الإلزامية لكل مسار** (الوثائق الناقصة): إعداد + منطق per-path.
- **بطاقة اكتمال الطلب** (محسوبة): منطق أعمال per-path.
- **معاينة أثر القرار** («عند الاعتماد سيتم… قيد + إشعار + قفل»): تحتاج بيانات أثر per-path — حدّي/معماري.
- **طباعة السجل + المرفقات كملف واحد**: backend/تجميع PDF.
- **تعليق على حقل/مرفق محدد**: ربط تعليق بكيان فرعي — backend.
- **كشف «مكرر»**: backend.

> ملاحظة: «منتهي» (من `retentionUntil`) و«الإصدارات» (من `document_versions`) **عرض-فقط** ⇒ ضمن (أ)، لا (ب).

## 3) الدفعات (Phase 3) — مرتّبة من الأقل خطرًا

> كل دفعة: prop اختياري + تبنٍّ في موضع/موضعين كنموذج إثبات فقط (لا تعميم قسري على 300 صفحة في نفس الدفعة).

| # | الدفعة | المكوّن المالك | العمل (إضافات متوافقة رجعيًا) | المخاطرة |
|---|---|---|---|---|
| **A** | معاينة المرفقات | `attachment-preview` | تكبير/تصغير + تدوير للصور · عارض PDF داخلي أمتن · أيقونة+تنزيل للأنواع غير المدعومة. لا تغيير بيانات. | منخفضة (مكتفية ذاتيًا) |
| **B** | درج الإنشاء | `AllowCreateDrawer` | رأس يوضح السياق (نص يُمرَّر) · toast «تم الإنشاء والتحديد» بعد الحفظ · زر «فتح الصفحة الكاملة». | منخفضة |
| **C** | قشرة الصفحة + الحالات الفارغة | `PageShell` · `empty`/`page-state` | فتحة `summary?` اختيارية في رأس الصفحة · رسائل فارغة إرشادية موحّدة (تعريب «جاري التحميل/لا نتائج…»). | منخفضة |
| **D** | الجداول | `DataTable` | عمود «الخطوة التالية» (مشتق من `statusField` عبر خريطة تُمرَّر) · إجراءات صف حسب الحالة (`rowActions?(row)`) · فلتر سريع «يحتاج إجراء مني» (عميل، على حقل قائم). | متوسطة (انتشار) |
| **E** | النماذج | `FormShell` | أقسام قابلة للطي (`sections?`) · ملخص جانبي (`aside?`) · أزرار حفظ متعددة (مسودة/حفظ وإضافة) · إظهار الحقول الناقصة داخل القسم · تلميحات حقول (`hint`). draft autosave يعتمد نمط `draftKey` القائم. | متوسطة (انتشار — كل النماذج) |
| **F1** | المرفقات (عرض) | `entity-documents` | `viewMode: list\|grid\|review` (افتراضي list) · تجميع حسب category/الرافع/التاريخ · شارة الحالة القائمة معرّبة · عرض الإصدارات (من `document_versions`) · شارة «منتهي» (من `retentionUntil`). | متوسطة |
| **G** | صفحات التفاصيل | `DetailPageLayout` | `mode: submitter\|reviewer\|readonly` (افتراضي القائم: تحكّم عرض/صلاحيات فقط) · **فتحة** «لوحة المراجعة» (`reviewPanel?: ReactNode`) يملؤها المسار القائد بإجراءاته القائمة — لا منطق اعتماد جديد · عرض مختصر لسجل التدقيق القائم. | متوسطة-عالية |

### الثغرة الآمنة المرافقة (إصلاح موضعي مستقل)
- **`uploaderName` الناقص:** خدمة الوثائق (api-server) لا ترجع اسم الرافع رغم استهلاك الواجهة له. إصلاح JOIN داخل **مسار الوثائق (الخادم)** فقط. يُفتح كإصلاح آمن منفصل (قابل للدمج مع F1 أو مستقلًا).

### ⛔ خارج التنفيذ — تقرير قرار لإبراهيم (نعم/لا) قبل أي عمل
البنود (ب) أعلاه: حالة مراجعة المرفق + ملاحظات المراجع + قبول/رفض per-attachment + قائمة المتطلبات + بطاقة الاكتمال + معاينة الأثر + طباعة مجمّعة + تعليق على حقل/مرفق + كشف المكرر. كلها migration/منطق نطاق ⇒ مقترح مستقل بعد إذن.

## 4) بوّابة كل دفعة (Phase 6 — إلزامية قبل إعلان النجاح)
1. `npm --prefix artifacts/ghayth-erp run typecheck`
2. `npm --prefix artifacts/ghayth-erp run test` (+ اختبار رندرة للمكوّن المعدّل يثبت توافق السلوك الافتراضي مع المستهلكين القائمين)
3. `node scripts/src/audit-domain-boundaries.mjs`
4. `node scripts/src/check-audit-coverage.mjs` + ما يلزم من `audit:wiring/routes/sidebar`
5. **حكم مجلس `ghayth-review` على diff الدفعة = «يُعتمد»** — لا إعلان نجاح برأي ذاتي.
- أي فشل → **تراجُع فوري** عن الدفعة، لا تُترك نصف مطبّقة.

## 5) الفرز والاعتماد (Phase 4)
- **نقطة الوقف الحالية:** تعديل المكوّنات المشتركة عابر للمسارات (cross-cutting) ⇒ أثر معماري ⇒ **يحتاج اعتماد إبراهيم على النطاق ونقطة البدء** قبل الدفعة الأولى.
- الدفعات A–G **تشغيلية لا تمس الدفتر** (إضافات عرض/تجربة) → تبدأ بعد الاعتماد، دفعة دفعة، مع تقرير وحكم مجلس بعد كلٍّ.
- لا توجد دفعة «تمس الدفتر» في هذا التوجيه.
- البنود (ب) موقوفة بانتظار قرار migration صريح.

## القيود الصارمة المطبّقة
- ⛔ لا تطبيق شامل دفعة واحدة — دفعة دفعة بحكم مجلس.
- ⛔ لا كتابة خارج المكوّن/المسار المالك.
- ⛔ لا migration ولا حقل جديد بلا اعتماد إبراهيم.
- ⛔ لا زر اعتماد بلا أثر، ولا رفض بلا سبب (تُحترم في فتحات العرض فقط، لا منطق جديد).
- ✅ كل تعديل prop اختياري متوافق رجعيًا — صفر كسر للمستهلكين القائمين.
- ✅ سلامة المعمارية مقدّمة على السرعة.
