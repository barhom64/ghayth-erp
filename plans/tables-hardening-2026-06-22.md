# خطة «توحيد الجداول وتقويتها + الترقيم» — 2026-06-22

مصدر: 3 تدقيقات قراءة-فقط (وكلاء فرعيون). هذه خطة دفعات تحت «وضع تعميم التوجيهات».

## نتائج المسح (مختصرة)

### 1) المكوّن الموحّد `DataTable` (components/ui/data-table.tsx)
موجود ويملك **أصلًا**: فرز عمودي (اختياري `sortable:true`)، تحديد (`selectable`+`onSelectionChange`)،
تجميع (`groupBy`+إجماليات بعرض كامل). **ينقص:** الفرز افتراضيًا لكل عمود، **محدّد حجم الصفحة في الواجهة**
(لا قائمة صفوف اليوم)، وحفظ حجم الصفحة. المكوّن **للقراءة فقط** (لا إدخال داخل الخلايا، لا colSpan في الصفوف).

### 2) مخزن تفضيلات المستخدم (Audit C)
توجد آلية لكل مستخدم في الحساب: عمودان على `users` (`preferredCalendar`/`preferredLocale`)،
تُقرأ بـ`GET /auth/me`، تُكتب بـ`PATCH /auth/me/preferences` (+Audit+Zod). **لا مخزن kv عام.**
حجم الصفحة بلا حفظ اليوم. → إضافته = migration صغير على `users` (بيان ضرورة + اعتماد).

### 3) الجداول الخام (Audit B): 80 ملف
- **~48 «قائمة تشغيلية»** → تحويل نظيف لـ`DataTable` (7 دفعات حسب الوحدة).
- **~6 شبكات إدخال محرّرة** (journal-manual-create، customer-receipt، commission-plan-editor،
  sales-wizard، warehouse-advanced، transport-rules-admin) → **التحويل يكسر الإدخال** (DataTable قراءة فقط).
- **كشوف طباعة** (customer/vendor-statement-print، monthly-close-pack) → تفقد محاذاة الطباعة.
- **5 مصفوفات/محاور** (budget-heatmap، income-statement-trend، risk-detail…) → إجماليات محاذية للأعمدة لا يدعمها groupBy.
- **6 ملفات اختبار** → تُتجاهل.

### 4) تغطية الترقيم (Audit A) — مفاجأة: النظام **ناضج وأخضر**
- خدمة قانونية واحدة `lib/numberingService.ts` (`issueNumber`+scheme resolution+audit)، 42 مخطط، 35 مُستدعى.
- **4 بوّابات CI خضراء** (audit-numbering-coverage، schemes-vs-callers، service-bypass، lint-patterns). 0 drift.
- كل جدول تنفيذي **مُعلَن** مُوصَّل بالكامل. الفجوات في **النقاط العمياء للمدقّق**:
  - **تشغيلي حقيقي:** `transport_bookings.bookingNumber` (نمط Date.now + مفتاح idempotency)، `cargo_manifests.manifestNumber` (إدخال — قد يكون بوليصة ناقل خارجية).
  - **يمسّ الدفتر (L1–L6):** vendor_advances.ref، vendor_credit_memos.ref، intercompany.ref، CIP-cost.ref، سند القبض JE ref، ومخططا RV/PV المُعطَّلان بلا جدول.
  - **حرج:** L1–L5 تستخدم `internalTechRef` — مخرج tech **مُقنَّن عمدًا** (يعيش في lib/ ليتجنّب lint الذي يفحص routes/). **قد تكون قرارات مقصودة من إبراهيم لا تسرّبًا** — لا يمكن الجزم بالقراءة.

> خلاصة الترقيم: ليس «أغلق فجوات كثيرة». النظام مُغطّى ومحكوم. المتبقّي: فجوتان تشغيليتان صغيرتان +
> بنود دفتر **قد تكون مقصودة** — تغييرها قرار معماري يمسّ الدفتر، يحتاج قرارك بندًا بندًا + assertion tests.

## الدفعات (الترتيب الحاكم)

### D1 — تطوير `DataTable` الموحّد (الأساس، نقطة واحدة → 230 جدولًا)
1. فرز كل عمود افتراضيًا (عدا actions/select/بلا-رأس).
2. محدّد حجم الصفحة في الواجهة (10/20/50/100/200) في شريط الترقيم.
3. توحيد التحديد: مفرد/متعدد/الكل + شريط إجراءات المحدّد + «تصدير المحدّد» مدمج.
4. حفظ حجم الصفحة لكل مستخدم (يعتمد على migration — قرار D-mig).
→ تطوير + اختبار انحدار شامل + مراجعة المجلس قبل الدمج.

### D2 — حارس توحيد الجداول
سكربت + اختبار يفشل عند `<table>` خام في صفحة تشغيلية جديدة (مع allowlist للشبكات المحرّرة/الطباعة/المصفوفات).

### D3 — تحويل الجداول الخام التشغيلية (48) — دفعات حسب الوحدة (قفل الحدود)
دفعة1 محافظ المالية(6) · دفعة2 طوابير/قوائم المالية · دفعة3 مالية بتجميع · دفعة4 الأسطول ·
دفعة5 العمرة+التفاصيل · دفعة6 معاينات القيود · دفعة7 إعدادات/أعلى. كل دفعة: typecheck + مجلس.

### D4 — الترقيم (قرارات قبل أي كود)
- تشغيلي: توصيل transport_bookings (يحتاج فصل المعروض عن مفتاح idempotency) + تصنيف cargo_manifests.
- دفتر L1–L6: **لكل بند** قرار إبراهيم (مقصود/تسرّب) + assertion tests على سطور القيد قبل أي تغيير.

## قرارات مطلوبة من إبراهيم (تَحكُم البدء)
- **D-mig:** شكل تخزين حجم الصفحة على `users` — عمود JSONB `tablePrefs` (مرن للمستقبل) أم عمود مفرد `tablePageSize`؟ + قائمة القيم المسموحة.
- **D-num:** نطاق الترقيم بالنظر لنضج النظام — تشغيلي فقط، أم مراجعة L1–L6 بندًا بندًا، أم إبقاؤه؟
- **D-raw:** الجداول الحساسة (إدخال/طباعة/مصفوفة) — إبقاؤها بمكوّنها الخاص أم تطويرها لاحقًا؟

البدء الفوري بعد القرارات: D1 (الأساس) ثم D3 دفعة-دفعة بالتوازي مع وكلاء.
