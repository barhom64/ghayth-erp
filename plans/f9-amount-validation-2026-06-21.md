# خطة إصلاح F9 — تحقّق المبالغ (سالب/صفر) — 2026-06-21

- **التوجيه:** «ابدأ بالمراجعة وسلّمني خطتك للإصلاح» (إبراهيم).
- **الوضع:** مراجعة دلالية مكتملة (قراءة فقط). **لم يُطبَّق أي تعديل.** ما يمسّ الدفتر ينتظر اعتمادك + assertion tests.
- **النطاق:** 61 حقل مبلغ في سكيمات zod بلا `.positive()/.min()/.nonnegative()` عبر المسارات المالية والمجاورة.

## الحقيقة المحورية (من مراجعة المحرّك)
`financialEngine.postJournalEntry` يتحقق من: وجود `sourceKey`، ثباته، idempotency، والتوازن (الفرق >0.05 يُسوّى عبر حساب وسيط 9999). **لكنه لا يرفض صراحةً `debit<0` أو `credit<0`.** فمدين سالب = دائن موجب حسابيًّا، يمرّ ويوازن لكنه يشوّه إجماليات المدين/الدائن والتقارير. ⇒ التحقّق المخططي دفاع مفقود فعليّ، لا تجميل.

## مبدأ حاكم للتصحيح (يمنع كسر الحالات المشروعة)
- **سطور القيد (debit/credit):** القاعدة الصحيحة محاسبيًّا = **كلاهما ≥ 0 دائمًا**. العكس (reversal) **يبدّل العمودين** فيبقيان ≥ 0 — لا استثناء سالب. ⇒ `.min(0)` صحيح 100% بلا كسر.
- **مبالغ المعاملات (`amount`):** قد تكون ≤ 0 مشروعةً (إشعار دائن، عكس قيد، تسوية). ⇒ **لا `.positive()` أعمى** — قرار لكل حقل.
- **التقريب (`roundingAmount`):** سالب مشروع (تقريب لأسفل) ⇒ **يُترك بلا حدّ**.
- **المعدّلات/سعر الوحدة:** ≥ 0 (المعدّل قد يكون 0، الصنف المجاني سعره 0) ⇒ `.nonnegative()` لا `.positive()`.

## التصنيف والدفعات

### الدفعة B1 — تشغيلية (غير دفترية) — تُنفَّذ فور إذنك، بلا اعتماد دفتر
حقول وصفية/تقديرية لا تُرحَّل للدفتر مباشرةً → `.nonnegative()` حارس إدخال آمن:
- `admin.ts:1910`، `employees.ts:105/222` (salary راتب تعريفي)
- `recruitment.ts:58/59/95/96` (salaryMin/Max نطاق إعلان وظيفة)
- `properties.ts` totalUnits/totalArea/totalContractValue (وصفي)
- `fleet.ts` coverageAmount/estimatedCost (تقدير/مرجع تأمين)
- **البوابة:** المجلس فقط (عمليّ). assertion tests غير مطلوبة (لا قيد).
- **الخطر:** منخفض. **قرار إبراهيم:** غير مطلوب — يكفي «ابدأ B1».

### الدفعة B2 — سطور القيد (دفترية، نظيفة) — تحتاج اعتمادك + assertion tests
`debit/credit ≥ 0` على سكيمات سطور القيد:
- `finance-journal.ts:404-405` (أرصدة افتتاحية)، `finance-recurring.ts:73-74` (قالب قيد متكرر)، `finance-hardening.ts:63-64`، `finance-algorithms.ts:29-30`، وأي سكيمة سطر قيد يدوي تظهر في المسح التنفيذي.
- **assertion tests (إلزامية قبل التطبيق):** (1) مدين سالب/دائن سالب يُرفَض؛ (2) قيد متوازن صحيح ما زال يُرحَّل؛ (3) **عكس قيد (reversal) يبدّل العمودين ويُرحَّل بنجاح** (إثبات عدم كسر العكوس).
- **الخطر:** متوسط (دفتري لكن القاعدة قاطعة بلا استثناء). **قرار إبراهيم:** نعم/لا.

### الدفعة B3 — مبالغ المعاملات والمعدّلات (دفترية + قرار دلالي لكل حقل)
- **المعدّلات/الوحدة** (`taxRate`/`vatRate`/`unitPrice`/`costPerLiter`): `.nonnegative()`.
- **مبالغ المعاملات** (`amount` في custodies/journal/hardening، invoices subtotal/total، purchase totalAmount/vatAmount، properties paidAmount/amount، umrah:445): **قرار لكل حقل** — هل يسمح بالسالب (عكس/إشعار دائن)؟ إن نعم يُترك أو يُقيَّد بـ `!= 0`؛ إن لا فـ `.positive()`.
- **يُستثنى صراحةً:** `finance-algorithms.ts:180 roundingAmount` (سالب مشروع).
- **assertion tests:** لكل حقل دفتري يُقيَّد، اختبار يثبت القبول/الرفض + عدم كسر العكوس.
- **الخطر:** متوسط-عالٍ (نُسخ متعددة + دلالة مختلفة). **قرار إبراهيم:** نعم/لا + حسم الحقول التي تسمح بالسالب.

## ترتيب التنفيذ المقترح
1. **B1** فور إذنك (عمليّ، دفعة واحدة، عبر guard + المجلس).
2. **B2** بعد «نعم»: أكتب assertion tests أولًا (حمراء)، ثم القيد `.min(0)`، ثم خضراء، ثم المجلس.
3. **B3** دفعة لكل مسار مالك، بعد حسمك للحقول التي تقبل السالب.

## ما لا يُفعَل
- ⛔ لا `.positive()` أعمى على أي حقل قد يكون سالبًا مشروعًا.
- ⛔ لا تطبيق لأي دفعة دفترية (B2/B3) بلا «نعم» صريحة + assertion tests خضراء.
- ⛔ لا توسيع المحرّك نفسه (إضافة رفض سالب داخل postJournalEntry) ضمن هذه الخطة — هذا تغيير محرّك مركزي يُقترح منفصلًا إن رغبت.
