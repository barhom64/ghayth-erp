# خطة: احتساب أجر السائق بالساعة (قيادة + توقف)

> حالة الوثيقة: **مقترح بانتظار الاعتماد** — لا كود قبل اعتماد إبراهيم (القرار ٤أ).
> التاريخ: 2026-06-30 · المالك التنفيذي: جلسة الوكيل · المرجع الأعلى: `docs/ghayth-constitution.md` + مواصفتا 02 (HR) و04 (الأسطول).

---

## 0. القرارات المعتمدة من إبراهيم (مدخلات ملزمة)

| # | القرار | الاختيار |
|---|--------|----------|
| نموذج الدفع | **بالساعة بالكامل** | لا راتب ثابت؛ المستحق = ساعات × معدّل |
| ساعات التوقف | **معدّل أقل** | معدّل توقف منفصل أقل من معدّل القيادة |
| مصدر الساعات | **عرض الاثنين + اعتماد يدوي** | التتبع واليدوي جنبًا لجنب، لا ترحيل إلا باعتماد بشري |
| الاعتماد | **خطة تفصيلية أولًا** | هذه الوثيقة؛ لا كود قبل الاعتماد |
| قيد إضافي | **كل شيء قابل للتعديل من الواجهة** | المعدّلات + الساعات + نوع الدفع لكل سائق |

---

## 1. الملخص

نبني آليّة تجعل أجر السائق يُحتسب من **ساعات القيادة** و**ساعات التوقف**، بمعدّلين منفصلين، حيث تأتي الساعات إمّا **مشتقّة من التتبع** (جلسات الملاحة/الرحلات/إشعال المحرك) أو **مُدخَلة يدويًا**، ويُعرض الرقمان جنبًا لجنب، ولا تُرحَّل أي ساعة إلى الراتب إلا بعد **اعتماد بشري**. ثم تحوّل **الموارد البشرية** الساعات المعتمدة إلى **بند راتب** وتُرحّل القيد عبر محرّك الرواتب القائم.

**المبدأ الحاكم (الدستور):** الراتب اختصاص الموارد البشرية (مسار قائد). الأسطول مسار خادم يوفّر **بيانات الساعات فقط** — لا يحسب أجرًا ولا يلمس الدفتر. المالية خادم في ترحيل القيد.

---

## 2. الحدود المعمارية — من يملك ماذا (قفل الحدود)

| المسار | يملك | لا يملك |
|--------|------|---------|
| **الأسطول** (خادم) | تسجيل/اشتقاق ساعات القيادة والتوقف، عرض التتبع مقابل اليدوي، **اعتماد الساعات** (بوابة بشرية) | لا معدّلات أجر، لا حساب مستحق، لا قيد محاسبي، لا كتابة في جداول الرواتب |
| **الموارد البشرية** (قائد) | معدّلات الأجر (قيادة/توقف)، نوع الدفع لكل سائق، تحويل الساعات المعتمدة إلى بند راتب، تشغيل المسيّر | لا يجمع بيانات GPS، لا يقرّر «ماذا حدث تشغيليًا» |
| **المالية** (خادم) | حساب الدفتر وترحيل قيد المسيّر (عبر محرّك HR القائم) | لا يملك سياسة أجر السائق |

**العقد بين المسارين:** خدمة قراءة فقط يكشفها الأسطول → `getApprovedDriverHours(assignmentId, periodStart, periodEnd)` تُعيد `{ drivingHours, stopHours }` من الصفوف **المعتمدة فقط**. الموارد البشرية تستهلكها وتضرب بمعدّلاتها. **صفر كتابة عابرة للنطاقات** — الأسطول لا يكتب أبدًا في `payroll_*`.

> هذا يطابق نمطًا قائمًا ومُثبتًا في النظام: مخالفة بمسؤولية السائق تشترط ربط السائق بموظف ثم تُسجَّل كخصم مصدره مُوسوم (`payroll_deductions.sourceEntity`). نعيد استخدام نفس الحوكمة، لكن للقراءة لا للكتابة العابرة.

## 3. نموذج البيانات (الـ migrations المطلوب اعتمادها)

### 3.1 جدول الأسطول الجديد — `fleet_driver_work_hours` (ساعات السائق اليومية)

صفّ واحد لكل سائق لكل يوم عمل. يحمل **الرقمين معًا** (تتبع + يدوي) وبوابة الاعتماد.

| العمود | النوع | الوصف |
|--------|------|-------|
| `id` | bigserial PK | |
| `companyId` | int NOT NULL | عزل المستأجر |
| `branchId` | int | |
| `driverId` | int NOT NULL → `fleet_drivers.id` | |
| `assignmentId` | int → `employee_assignments.id` | تعيين السائق في HR (مطلوب للترحيل) |
| `workDate` | date NOT NULL | يوم العمل |
| `derivedDrivingHours` | numeric(6,2) | **مشتقّة من التتبع** (تلقائي) |
| `derivedStopHours` | numeric(6,2) | **مشتقّة من التتبع** |
| `derivedSource` | text | `navigation` / `ignition` / `trip_events` — مصدر الاشتقاق |
| `manualDrivingHours` | numeric(6,2) NULL | **إدخال يدوي** |
| `manualStopHours` | numeric(6,2) NULL | **إدخال يدوي** |
| `approvedDrivingHours` | numeric(6,2) NULL | الساعة **المعتمدة** (يختارها المعتمِد: مشتقّة أو يدوية أو مُعدّلة) |
| `approvedStopHours` | numeric(6,2) NULL | |
| `status` | text NOT NULL DEFAULT `'pending'` | `pending` / `approved` / `void` — **بوابة الترحيل** |
| `approvedByAssignmentId` | int NULL | من اعتمد |
| `approvedAt` | timestamptz NULL | |
| `payrollLineId` | int NULL → `payroll_lines.id` | يُختم به بعد الترحيل (منع الترحيل المزدوج) |
| `notes` | text | |
| `createdByAssignmentId`, `createdAt`, `updatedAt`, `deletedAt` | | تدقيق |

قيود: فهرس فريد `(driverId, workDate)` حيث `deletedAt IS NULL` (يوم واحد لكل سائق). فهرس `(companyId, status, workDate)` لجلب المسيّر.

### 3.2 جدول HR الجديد — `hr_driver_pay_rates` (معدّلات الأجر، قابلة للتعديل من الواجهة)

| العمود | النوع | الوصف |
|--------|------|-------|
| `id` | bigserial PK | |
| `companyId` | int NOT NULL | |
| `assignmentId` | int NULL → `employee_assignments.id` | **NULL = معدّل افتراضي للشركة**؛ قيمة = تخصيص لسائق |
| `payType` | text NOT NULL DEFAULT `'hourly'` | `hourly` / `monthly` — **نوع الدفع لكل سائق، قابل للتبديل** |
| `drivingHourlyRate` | numeric(10,2) | معدّل ساعة القيادة |
| `stopHourlyRate` | numeric(10,2) | معدّل ساعة التوقف (أقل) |
| `effectiveDate` | date | بدء السريان (تاريخية المعدّل) |
| `isActive` | boolean DEFAULT true | |
| `createdBy`, `createdAt`, `updatedAt`, `deletedAt` | | تدقيق |

الوراثة: نظام (افتراضي عبر `resolveSettings`) ← شركة (`assignmentId = NULL`) ← سائق (`assignmentId = X`). الأقرب يفوز — مطابق لمبدأ الإعدادات في الدستور.

> بديل أبسط مرفوض عمدًا: وضع المعدّل على `fleet_drivers` — يخالف الحدّ لأنه يضع **سياسة أجر** (قرار قائد) داخل مسار خادم. لذلك المعدّل في HR.

### 3.3 توسعة `payroll_lines` (إضافية، غير كاسرة)

على نمط `overtimeHours` الموجود أصلًا (سابقة مُثبتة):

```sql
ALTER TABLE payroll_lines ADD COLUMN IF NOT EXISTS "drivingHours"       numeric(6,2)  DEFAULT 0;
ALTER TABLE payroll_lines ADD COLUMN IF NOT EXISTS "drivingHoursAmount" numeric(14,2) DEFAULT 0;
ALTER TABLE payroll_lines ADD COLUMN IF NOT EXISTS "stopHours"          numeric(6,2)  DEFAULT 0;
ALTER TABLE payroll_lines ADD COLUMN IF NOT EXISTS "stopHoursAmount"    numeric(14,2) DEFAULT 0;
```

`grossSalary` يجمع البندين الجديدين. للسائق بالساعة: الأساسي = 0 والإجمالي كلّه من هذين البندين.

### 3.4 المالية — حساب الدفتر (بلا migration للحساب)

حساب جديد يُحلّ ديناميكيًا عبر `financialEngine.resolveAccountCode(companyId, "fleet_driver_wages_expense", "debit", "5240")` — قيمة افتراضية `5240` «مصروف أجور سائقين»، تقابلها المالية في دليل الحسابات. الطرف الدائن: `salary_payable` (2120) القائم. **لا تغيير سكيمة على الدفتر** — فقط رمز حساب يُحلّ بافتراضي.

## 4. اشتقاق الساعات من التتبع (تلقائي)

**المصدر الأساسي — `driver_navigation_sessions`** (8 توقيتات لكل جلسة):

```
ساعات القيادة = (arrivedPickupAt − startedAt)          // التنقل للتحميل
              + (arrivedDropoffAt − loadedAt)           // التنقل للتفريغ
              + (endedAt − deliveredAt إن وُجد تنقل عودة)

ساعات التوقف  = (loadedAt − arrivedPickupAt)            // انتظار/تحميل
              + (deliveredAt − arrivedDropoffAt)        // انتظار/تفريغ
```

تُجمع كل جلسات السائق في اليوم → `derivedDrivingHours` / `derivedStopHours`، مع `derivedSource = 'navigation'`.

**مصدر بديل — `fleet_device_positions.ignitionOn`**: مدّة المحرك المُشغّل بين نبضتين = قيادة تقريبية؛ توقف = مُشغّل وسرعة ≈ 0. يُستخدم حين تنقص جلسات الملاحة (جدول ثقيل، احتفاظ 90 يومًا — يُحسب يوميًا عبر Cron لا عند الطلب).

**مصدر ثالث — `fleet_trip_events`** (وقائع الرحلة التي بنيناها): نفس منطق النوافذ الزمنية من `occurredAt` حسب `eventType` (depart/arrive…).

> الاشتقاق **اقتراح** يُعرض للمعتمِد، لا يُرحَّل تلقائيًا (القرار ٣ج).

## 5. التدفق الكامل (الواقعة ← الاعتماد ← الراتب ← القيد)

1. **يوميًا (Cron أو عند إغلاق الجلسة):** الأسطول يشتقّ ساعات كل سائق ويكتب/يحدّث صفّ `fleet_driver_work_hours` (`status = pending`).
2. **الإدخال اليدوي (اختياري):** المشرف/السائق يُدخل `manualDrivingHours/manualStopHours` من الواجهة عند نقص أو خطأ في التتبع.
3. **الاعتماد البشري (إلزامي):** المشرف يرى **التتبع | اليدوي** جنبًا لجنب، يختار/يعدّل `approvedDrivingHours/approvedStopHours`، ثم يضغط «اعتماد» → `status = approved` + أثر تدقيق.
4. **تشغيل المسيّر (HR):** لكل سائق بـ `payType = hourly`:
   - HR يستدعي خدمة الأسطول `getApprovedDriverHours(assignmentId, period)` → مجموع الساعات المعتمدة غير المرحّلة.
   - HR يجلب المعدّل عبر `hr_driver_pay_rates` (الأقرب يفوز).
   - يحسب: `drivingHoursAmount = drivingHours × drivingHourlyRate` و`stopHoursAmount = stopHours × stopHourlyRate`.
   - يكتب القيم في `payroll_lines`، ويختم صفوف الساعات بـ `payrollLineId` (منع الازدواج).
5. **القيد (المالية، عبر محرّك HR القائم):** `hrEngine.postPayrollRunGL()` يضيف سطر مدين «مصروف أجور سائقين» مقابل «رواتب مستحقة».

## 6. القيد المحاسبي + اختبارات assertion (إلزامية — القاعدة 3)

**القيد لكل مسيّر:**

| الطرف | الحساب | المبلغ |
|------|--------|--------|
| مدين | مصروف أجور سائقين (`fleet_driver_wages_expense` / 5240) | Σ(`drivingHoursAmount` + `stopHoursAmount`) |
| دائن | رواتب مستحقة (`salary_payable` / 2120) | نفس المبلغ |

(يندمج ضمن قيد المسيّر القائم؛ GOSI والخصومات تبقى كما هي.)

**اختبارات assertion على سطور القيد (لا ترحيل قيد بلا اختبار = رفض تلقائي):**
- مجموع المدين الجديد = Σ على كل السطور (قيادة×معدّل + توقف×معدّل) بدقّة هللة.
- القيد متوازن (Σ مدين = Σ دائن).
- بـ 0 ساعة معتمدة → **لا سطر قيد** (لا قيد صفري).
- ساعات معتمدة لكن السائق `payType = monthly` → تُتجاهل (لا بند ساعات).
- إعادة التشغيل لا تُرحّل صفوفًا مختومة بـ `payrollLineId` (لا ازدواج).

## 7. الشاشات (كلّها قابلة للتعديل من الواجهة)

1. **الأسطول — «ساعات السائق»** (`/fleet/drivers/:id/work-hours` + عرض الفترة):
   - جدول يومي بأعمدة: **التتبع (قيادة/توقف) | اليدوي (قيادة/توقف) | المعتمد (قيادة/توقف) | الحالة**.
   - حقول اليدوي والمعتمد **قابلة للتحرير**؛ زر «اعتماد» يقفل الصف.
   - نمط «الكيان يقود التجربة»: المعتمِد يرى الواقعة (التتبع) ويصحّح، لا يحسب أجرًا.
2. **HR — «معدّلات أجر السائق»** (`/hr/settings/driver-pay`):
   - افتراضي الشركة (قيادة/توقف) + جدول تخصيص لكل سائق + مبدّل `payType` (شهري/بالساعة) — **كلها قابلة للتعديل**.
3. **كشف الراتب**: سطران جديدان «أجر قيادة (س × معدّل)» و«أجر توقف (س × معدّل)» ضمن البنود.
4. **بوابة السائق (عرض فقط)**: السائق يرى ساعاته المعتمدة وأجرها المتوقّع.

## 8. RBAC والتدقيق

| الفعل | الصلاحية المقترحة |
|------|-------------------|
| إدخال/تعديل الساعات اليدوية | `fleet.driver_hours:update` (مشرف أسطول/منسّق) |
| اعتماد الساعات | `fleet.driver_hours:approve` (مشرف أعلى — منفصل عن الإدخال) |
| تعديل المعدّلات/نوع الدفع | `hr.driver_pay:update` (HR فقط) |
| رؤية الساعات | السائق يرى نفسه؛ المشرف فرعه؛ المالك الكل |

**Audit إلزامي** على: إنشاء/تعديل صف ساعات، الاعتماد، تعديل المعدّل، الترحيل للراتب. كل أثر يحمل: الفاعل + الوقت + السبب + القيمة قبل/بعد.

## 9. الإشعارات والأفعال التلقائية

- اشتقاق يومي مكتمل بنقص بيانات (جلسة ناقصة التوقيتات) → إشعار المشرف «راجع ساعات السائق X ليوم Y».
- اعتماد الساعات → إشعار السائق بساعاته المعتمدة.
- ترحيل المسيّر → بند الساعات يظهر في كشف السائق.
- تعارض كبير بين التتبع واليدوي (> عتبة) → تنبيه قبل الاعتماد.

## 10. التحققات (Validations)

- **ربط السائق بموظف إلزامي**: لا أجر بالساعة ما لم يكن `fleet_drivers.employeeId` و`assignmentId` موجودَين (نفس حارس المخالفة→خصم).
- لا ترحيل لصف `status != approved`.
- المعدّلات > 0؛ معدّل التوقف ≤ معدّل القيادة (تحذير لا منع).
- الساعات اليومية ضمن حدّ معقول (مثلًا ≤ 24، تحذير عند تجاوز ساعات القيادة النظامية).
- لا ترحيل مزدوج (`payrollLineId IS NULL` شرط الالتقاط).

---

## 11. حزمة الاختبارات الكاملة

- **اشتقاق:** من توقيتات جلسة → قيادة/توقف صحيحة (حالات: جلسة كاملة، ناقصة، متعددة الجلسات/يوم).
- **اعتماد:** صف غير معتمد لا يلتقطه المسيّر؛ معتمد يُلتقط مرّة واحدة.
- **حساب الأجر:** ساعات × معدّل بالمعدّلين، مع وراثة المعدّل (شركة/سائق).
- **القيد:** assertion على السطور (بند 6).
- **الحدود (ثابت):** الأسطول لا يستورد `postJournalEntry`/`payroll_*` (تأكيد `not.toContain`).
- **RBAC:** فصل الإدخال عن الاعتماد؛ HR وحده يعدّل المعدّل.
- **عزل المستأجر:** كل استعلام يفلتر `companyId`.

## 12. خطة التنفيذ دفعة دفعة (عبر المجلس)

| الدفعة | المحتوى | يمسّ الدفتر؟ | البوابة |
|--------|---------|:---:|---------|
| **1 — تشغيلية (الأسطول)** | جدول `fleet_driver_work_hours` + اشتقاق التتبع + شاشة الساعات (تتبع/يدوي/اعتماد) + RBAC + Audit | لا | تأكيد عام → مجلس |
| **2 — معدّلات HR** | جدول `hr_driver_pay_rates` + شاشة المعدّلات + `payType` | لا | مجلس |
| **3 — الراتب + القيد** | أعمدة `payroll_lines` + خدمة قراءة الأسطول + حساب البند في محرّك HR + **القيد + اختبارات assertion** | **نعم** | **اعتماد إبراهيم + assertion tests + مجلس** |

كل دفعة: بوابة الدستور + `guard` + مجلس «يُعتمد» قبل الانتقال. الدفعة 3 لا تُطبَّق بلا اعتماد بشري صريح واختبارات قيد ناجحة.

## 14. قرارات تبقّت لإبراهيم (قبل الدفعة 3)

1. **GOSI على أجر الساعات:** هل يخضع أجر الساعة لاشتراك التأمينات (يُحتسب على الإجمالي)، أم يُعامل كبدل خارج وعاء GOSI؟ (الافتراضي المقترح: يخضع — لأنه أجر أساسي بالكامل في النموذج «بالساعة».)
2. **رمز حساب الدفتر:** اعتماد `5240 مصروف أجور سائقين` أم دمجه في `salary_expense 5210`؟ (المقترح: حساب منفصل للوضوح التحليلي.)
3. **دورية الاشتقاق:** Cron يومي ليلي أم لحظة إنهاء كل جلسة ملاحة؟ (المقترح: عند الإنهاء + Cron تسوية ليلي.)

> ما عدا ذلك (التصميم التشغيلي للدفعتين 1 و2) جاهز للبدء فور اعتماد الخطة.
