# البند ٤ — تدفّقات الكيانات الغنية (ملحق أ): خطة التنفيذ المُدقَّقة

> **الحالة:** مُدقَّقة بالكود (٢٠٢٦-٠٦-٢٧). البند يُنفَّذ **شريحة-شريحة**، PR مستقل لكل
> واقعة، **assertion على سطور القيد** لكل ما يمسّ الدفتر، وبوابة المجلس قبل كل دمج.
> «نعم» إبراهيم الدائمة للبند ٤ معطاة («كلها على التوالي»). الإبلاغ بالنتيجة لا الاستئذان.

## المبدآن (توجيه إبراهيم) — كلاهما مبنيّ فعلًا

١) **التوجيه يقرّر الحساب (costBearer / مَن يتحمّل):** تكلفة الكيان لا تذهب لحساب ثابت
بل حسب الجهة المتحمّلة — على المركبة → مصروف محروقات المركبة؛ على الموظف → مصروف
الموظف + **ذمة عليه**؛ إلخ. **مبنيّ في م٥** (`resolveCostBearerAccounts` +
`registerCostBearerObligations` في `financeDocumentService`).

٢) **فتح حسابات الأصل تلقائيًّا عند إضافته:** أي أصل عند إنشائه تُفتح حساباته الفرعية
(محروقات/صيانة/إهلاك لكل لوحة/عقار…). **مبنيّ في `accounting-engine.createSubsidiary
AccountsForEntity`** — مُستدعى فعليًّا عند إنشاء المركبة (`fleet.ts:594`)، السائق،
العقار. والترحيل **يستبدل تلقائيًّا** الحساب الأب بحساب الأصل الفرعي عبر
`journalLineDimensionalEnricher.substituteSubsidiaryAccountCodes` حسب بُعد الكيان.

## خريطة إعادة الاستخدام (محرّكات قائمة مُختبَرة — لا بناء)

- **`postFinancialDocument(input)`** (م٥، مُختبَر بـassertion): يبني خطة قيد متوازنة من
  بنود + تخصيصات (entityType/entityId + costBearer)، يفرّع المتحمِّل، يُرحّل.
- **`deriveSpecializedAccount({itemType, targetType})`**: وقود→`vehicle_fuel_expense`،
  صيانة→`vehicle_maintenance_expense`، … (الحساب الأب المتخصّص).
- **`substituteSubsidiaryAccountCodes`**: الأب → الفرعي لكل لوحة عند الترحيل.
- **`createSubsidiaryAccountsForEntity`**: فتح الحسابات عند إضافة الأصل.
- **`obligationsEngine` / `ensureCostCenterForEntity`**: الالتزام الزمني + مركز التكلفة.

> فكل واقعة كيان = **بناء مدخل `FinancialDocumentInput` + استدعاء `postFinancialDocument`**.
> النوع/الحساب/الفرعي/التفريع/الأبعاد كلها يتولّاها المحرّك. الجديد: واجهة واعية بالكيان
> + نقطة الواقعة (تربط السجل التشغيلي + العداد + الترحيل) + اختبار assertion.

## الحوكمة (لكل شريحة)

- **assertion على سطور القيد** (الدستور م٣): توازن + الحساب الصحيح (المتخصّص الفرعي) +
  البُعد (vehicleId/employeeId…) + تفريع costBearer (موظف→ذمته).
- **حدّ المسار:** المسار القائد (الأسطول/العقار…) يستدعي **محرّك المالية الخادم**
  (`postFinancialDocument`) عبر عقد — الكتابة تحدث داخل المالية، صفر SQL عابر في
  ملف المسار القائد (audit-domain-boundaries أخضر).
- **دورة الحياة:** الترحيل يُنشئ القيد (لا حفظ المسودة). idempotency بـ`sourceKey`.
- عزل companyId · Audit · RBAC المسار القائد · بوابة المجلس قبل الدمج.

## شريحة ١ — مركبة/وقود (التصميم التنفيذي)

- **خلفي:** `POST /fleet/vehicles/:id/fuel-event` (مملوك للأسطول، RBAC `fleet.vehicles`):
  يسجّل `fleet_fuel_logs` (قائم) + يحدّث العداد + يستدعي `postFinancialDocument`
  (`documentKind:"expense"`, `direction:"payment"`, سطر وقود: كمية=لترات، سعر=سعر/لتر،
  `counterAccountCode`=الأب المتخصّص للوقود، تخصيص `entityType:"vehicle"`+`costBearer`).
  المحرّك يستبدل بالحساب الفرعي للوحة + يفرّع costBearer + يضع بُعد vehicleId.
- **assertion:** اختبار تكامل على سطور القيد (توازن · الحساب الفرعي الصحيح · vehicleId ·
  وعند costBearer=موظف: مدين ذمة الموظف بدل المصروف + التزام مطالبة).
- **واجهة:** «تسجيل وقود» واعية بالمركبة (اختر مركبة→عالمها→لترات/سعر/عداد/مَن يتحمّل).
- **لا هجرة** (`fleet_fuel_logs` + `journal_*` + `subsidiary_accounts` قائمة).
