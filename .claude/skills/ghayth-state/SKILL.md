# لقطة حالة نظام غيث

> **تاريخ اللقطة: 18 يونيو 2026.** هذه نقطة زمنية ثابتة من ذاكرة إدارة المشروع، **لا تتحدث تلقائيًا.** عند أي تعارض، الكود الفعلي وحالة الـ PRs الحية هي الحقيقة. حدّث هذا الملف يدويًا بعد كل تغيير جوهري.

## البنية التقنية الجارية (تطوّرت عن المواصفات الأصلية)

pnpm monorepo: `api-server` (Node.js/Express + PostgreSQL + Drizzle ORM) — `ghayth-erp` (React 19 + TypeScript + Tailwind) — `ghayth-erp-deck`. الهجرات (migrations) ملفات مرقّمة في `src/migrations`. النطاق: 3500+ commit، 3500+ ملف، 14 نطاقًا وظيفيًا، 300+ هجرة، 600+ صفحة، 100+ ملف مسار backend، 1200+ نقطة API.

## حالة المسارات

- **المالية (S7):** PR #1997 معتمد (معاينة فاتورة تعرض حسابات AR/إيراد/VAT — للقراءة فقط، لا تغيير ترحيل). محرك إطفاء التأمين معتمد للبناء: Engine A ينشئ جداول الاعتراف عند الرسملة؛ إطفاء شهري idempotent عبر `postJournalEntry`. تأمين العقار والتأمين الطبي بلا محركات ترحيل GL بعد.
- **الموارد البشرية:** Wave 0 مكتملة (كل النقاط مبوّبة بـ `authorize()`، اختبارات RBAC ratchet، ترتيب H0→H8). Wave 1B بدأت. 4 قوالب أدوار مكسورة: `hr_admin` بلا تعريف في الهجرات؛ `attendance_officer` و`discipline_officer` و`performance_reviewer` بلا `rbac_role_grants`.
- **النقل:** الفرع النظيف معتمد — commit السقالة أُسقط، الهجرة 282 مؤكدة (لا تمس القيود).
- **الأملاك:** PR #1999 معتمد (contractType موحّد، هجرة idempotent 286). الـ PRs الماسة للقيود 4/5/6 معلّمة لتحقق خاص.
- **المستودعات:** قلق ProductSelect العابر للنطاقات حُلّ (وكيل المستودعات عدّل `invoices-create.tsx` المملوك للمالية)؛ الإصلاح مؤكد (`sellPrice` هو الحقل الصحيح).
- **المشاريع:** فشلان صامتان أُصلحا — أخطاء 500 في إكمال المرحلة (كانت تعطّل فوترة المعالم)، وترحيل WIP ميت بسبب مفاتيح GL غير مزروعة.
- **العمرة:** PRs #1920 و#1931 و#1995 جاهزة للدمج. PRs #1996 و#1998 فيها bug واجهة حقيقي (`formatDateAr` بدل `formatUmrahDate` — يخالف تفضيل التقويم الهجري) — مأذون بإصلاحه، لا بتجاوز الحارس.

## ديون البنية التحتية (تتطلب حلًا)

- أرقام هجرات مكررة (282×2، 284×3) من العمل المتوازي — تحتاج مصالحة تصادم.
- تفريغ مخطط فاسد (HR-022) ينقصه 13+ هجرة وكل بذور البيانات — السبب الجذري لصعوبة التحقق من قاعدة البيانات.
- خلل إعداد حماية الفرع: عدم تطابق اسم الفحص المطلوب يحجب 10+ PRs خضراء.
- 3 سكربتات تدقيق DB (`check-schema-drift`، `check-ghost-rows`، `check-sql-ambiguity`) لم تُشغّل على الهجرات المتراكمة — تحتاج `DATABASE_URL` حيًا.

## فجوات السلامة المالية (نشطة)

- أقساط التأمين تُرسمل على حساب 1172 صحيحًا لكنها **لا تُطفأ أبدًا** — لا محرك اعتراف موجود.
- `propertiesEngine.ts:198` يسيء استخدام حساب 2160 (إيراد غير مكتسب) كـ fallback لمستحقات الصيانة — فساد صامت في الميزانية العمومية.
- تعبئة الأبعاد (dimensions) غير متسقة عبر المستدعين (`clientId/vendorId/umrahAgentId` تبقى null على الفواتير).
- نموذج القيد اليدوي (`journal-manual-create.tsx`) باب خلفي غير مبوّب يتجاوز انضباط السيناريوهات — إصلاح معماري معتمد (تصنيف Type A/B مع ربط إلزامي وموافقة GM للقيود الهيكلية).

## سكربتات التدقيق

تعمل بدون `node_modules`: `node scripts/src/[name].mjs`. منها: `audit-domain-boundaries.mjs`، `audit-stop-ship.mjs`، `audit-event-bus.mjs`، `check-frontend-backend-wiring.mjs`، `check-duplicate-migrations.mjs`. ثلاثة تحتاج `DATABASE_URL` حيًا وتُشغّل على الخادم.

## على الأفق

- المسارات 9–14 (القانونية، الدعم، CRM، الاتصالات، الوثائق، الحوكمة/BI) لم تمر بدورة التنظيم/التدقيق بعد.
- لا يوجد harness لاختبار سلوك الواجهة — مؤجل بالكامل.
- لوحة الصدق (Truthfulness dashboard) معتمدة للبناء — تتبّع % من P&L المُبعّد بالكامل، القيود اليتيمة، ترحيلات حسابات الـ fallback؛ بوابة إقفال صارمة.
- تعميم نمط AllowCreateDrawer على بقية الوحدات.
- مسار `db:provision-agent` — مفتاح فك العمل المتوقف server-side عبر المالية/العمرة/الأملاك.
- معمارية الاشتراك/الاستحقاق لكل وحدة: `subscriptionGate.ts` على مستوى الشركة فقط؛ الاستقلال لكل وحدة ~25–30% متحقق.
- الجوال: إكمال PWA أولًا (manifest، أيقونات، caching) ثم Capacitor (سائق الأسطول، الموافقات، الحضور الميداني).
- ZATCA Phase 2 وتعدد العملات: مؤجلة.

## مبادئ متراكمة (إضافة للدستور)

- **فهرسة الحقول المشفّرة:** لا يمكن فهرسة الحقول المشفّرة مباشرة (AES-GCM IVs عشوائية). استخدم نمط Encrypted+Hash: AES-256-GCM للتخزين، HMAC-SHA256 للبحث.
- **Mock-First للتكاملات الخارجية:** 3 شروط مسبقة قبل بناء المحوّلات الحقيقية.
- **ProductSelect مشترك واحد** يُبنى مرة (المستودعات) ويُعاد استخدامه — لا تكرار.
