# 04 — جرد نقل الحمولات (tripFamily = cargo)

> مساران متكاملان على main: (أ) حجز حمولة عبر السقالة الكنسية T0→T9، (ب) بوليصة الشحن `cargo_manifests` بدورة 15 حالة — مرتبطان عبر `transport_dispatch_orders.linkedManifestId`.

---

## 1. البوليصة — دورة الحياة الكاملة (15 حالة)

```text
draft → requested → approved → assigned_to_driver → driver_accepted
→ trip_started → arrived_pickup → loaded → in_transit → arrived_delivery
→ delivered → completed → ready_for_invoice → financially_closed (+cancelled)
```

| الشريحة | المالك | الموضع |
|---|---|---|
| draft…assigned_to_driver | المشغّل | `PATCH /cargo/manifests/:id` بجدول انتقالات صارم |
| driver_accepted…delivered (7 خطوات) | **السائق** | `POST /fleet/me/cargo/:id/advance` — خطوة واحدة لكل نقرة، forward-only، لا إلغاء بيد السائق |
| completed | المشغّل (الإغلاق التشغيلي) | cargo.ts |
| ready_for_invoice | بوابة «جاهز للمحاسبة» — هنا فقط يُنشأ المرشّح | cargo.ts → `fleetEngine.createCargoBillingCandidate` |
| financially_closed | المحاسب (materialize) | transport-billing-candidates.ts |

**حدود مثبتة:** السائق يقف عند delivered؛ لا JE من أي شاشة نقل؛ العصمة على (companyId, sourceType, sourceId).

## 2. الإنشاء — إجبار العميل المربوط
- `createManifestSchema.refine`: customerId إلزامي (CRM)؛ الاسم النصّي وحده مرفوض. الواجهة: Select يملأ الاسم/الجوال للعرض فقط، زر الحفظ معطَّل بلا عميل.
- **بقيّة:** PATCH جزئي يسمح بتعديل customerName دون customerId (سجلات قديمة) — مقبول، يُغطّى بسجل التدقيق (TA-SEC-02).

## 3. نقاط التشغيل التفصيلية (TR-016 — checkpoints)
- جدول `cargo_manifest_checkpoints` (305): 10 أنواع مقيَّدة CHECK — loading_start/complete، **weighing** (كغ)، **rest_break** (دقائق)، inspection، customs، **fueling** (لتر)، unloading_start/complete (وحدات)، other + GPS + recordedBy مفروض من الـscope.
- Endpoints: `POST/GET /fleet/me/cargo/:id/checkpoint(s)` (السائق) + `GET /cargo/manifests/:id/checkpoints` (المشغّل). حدث `fleet.cargo.checkpoint_recorded` بعربيّته في الكتالوج.
- **فجوة واجهة:** الـbackend مكتمل (#2056) لكن **حوار السائق في الـSPA غير مدموج** — جاهز ومركون في stash بانتظار اكتمال هذا الجرد (قاعدة التجميد). TA-GAP-02.

## 4. الحمولة المتكررة (TR-017 — Route Patterns)
- `transport_route_patterns` (295): daysOfWeekMask 7-bit (بت 0=أحد بتوقيت الرياض) + departureTime + activeFrom/Until + defaults (عميل/عقد/وزن/فئة مركبة) + operationalWaypoints jsonb.
- Materialise مفرد + **materialise-range** (#2052): نافذة ≤90 يومًا، idempotent على (companyId, bookingNumber=`RP-{code}-{date}`)، يفصل created عن skipped، حجوزات draft بمصدر recurring_schedule وtripFamily=cargo.
- **تعارض توثيقي مؤكَّد:** تعليق الراوتر يعد بـ«cron يومي يجسّد القوالب» — **لا وظيفة cron لذلك في `cronScheduler.ts`** (فُحص JOB_DEFINITIONS). التجسيد يدوي فقط. TA-CONF-01 (حرج توثيقيًا/تشغيليًا).
- **فجوة واجهة:** لا صفحة SPA لإدارة Route Patterns إطلاقًا — الإدارة API فقط. TA-GAP-03.

## 5. الميزان والوزن
- بوليصة: totalWeight مجمَّع آليًا من الأصناف (recomputeManifestTotals).
- إسناد: `assertVehicleCapacity` ضد operationalPayloadKg/payloadKg مع تجاوز موثَّق.
- checkpoint نوع weighing يلتقط وزن الميزان الفعلي أثناء الرحلة.

## 6. التسعير والفوترة (أثر فقط)
- نموذج المالك: «فاتورة ببند واحد (وزن×سعر طن) للعميل، وداخليًا إيراد لكل مركبة + مركز تكلفة + عمولات + مستحق سائق».
- الموجود: `transport_price_rules` (unitOfMeasure يشمل kg/tonne) + `pricingEngine` + `transport_service_lines` + invoice-batches + مرشّح الحمولة يحمل quantity=وزن/unit=kg وsuggestedRevenue/Cost.
- **الفجوات:** التفصيص الداخلي (إيراد لكل مركبة داخل فاتورة موحَّدة + عمولات + مستحق السائق الآلي) **غير مبنيّ** — مؤجَّل لمرحلة الترحيل المالي (TA-GAP-04، الموجة المالية).

## 7. A-03 / A-04 — معايير القبول
- A-03 (قالب أسبوعي + N ردود = N instances): مثبَت بـ22 اختبارًا in-memory؛ **الإثبات الحيّ على Postgres مطلوب** (الملف 17).
- A-04 (خطوات السائق التفصيلية تعمل من عرض السائق): backend مكتمل + 20 اختبارًا؛ ينقصه دمج الحوار + E2E حيّ.
