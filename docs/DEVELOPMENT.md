# Development guide — Ghayth ERP

This is the short, opinionated day-to-day playbook. It assumes you have already read `README.md` and `docs/ARCHITECTURE.md`.

---

## 1. First-run checklist

1. Install Node 24, pnpm, and PostgreSQL 16.
2. Clone the repo and `pnpm install`.
3. **Bootstrap the local DB from the committed schema dump:**
   ```bash
   pnpm db:bootstrap
   ```
   This drops + recreates the `ghayth_erp` Postgres database, loads
   `db/schema.sql` (the canonical DDL — full schema dumped from the
   live Replit instance), `db/seed.sql` (reference rows: companies,
   branches, permissions, chart of accounts, …), and
   `db/seed-admin-user.sql` (deterministic test admin
   `owner@local.test` / `Test1234!`). It also pre-marks every
   migration in `artifacts/api-server/src/migrations/` as applied,
   so the runtime migration runner is a no-op on first boot.

   See `db/README.md` for what the bootstrap loads, the relationship
   to incremental migrations, and how to regenerate the dumps from a
   live Replit instance via `pnpm db:dump-schema` / `pnpm db:dump-seed`.

4. Drop this `.env` into `artifacts/api-server/.env`:
   ```
   DATABASE_URL=postgres://ghayth_erp:ghayth_erp@localhost:5432/ghayth_erp
   JWT_SECRET=local-dev-secret-must-be-at-least-32-characters-long-test-only
   NODE_ENV=development
   PORT=5000
   ```

5. `pnpm run typecheck` — confirm a cold workspace compiles.
6. `pnpm --filter @workspace/api-server run dev` — start the API.
   On first boot you should see `Migration already applied: NNN_*.sql`
   for every file in `src/migrations/`. **No `column does not exist`
   or `relation does not exist` errors should appear.** If they do,
   the committed `db/schema.sql` is out of date — run
   `pnpm db:dump-schema` on Replit to regenerate it (see §2.4 below).
7. `pnpm --filter @workspace/ghayth-erp run dev` in a second terminal.

8. Login with the deterministic test admin:
   ```bash
   curl -X POST http://localhost:5000/api/auth/login \
     -H "Content-Type: application/json" \
     -d '{"email":"owner@local.test","password":"Test1234!"}'
   ```

## 2. Migrations

### Where they live

- **Source files:** `artifacts/api-server/src/migrations/NNN_name.sql`
- **Built copies:** `artifacts/api-server/dist/migrations/NNN_name.sql` (generated by `build.mjs`; do not edit by hand).

### Rules

1. **Numbering is sequential and append-only.** Never renumber a published migration; add a new one.
2. **Every migration must be idempotent.** Use `CREATE TABLE IF NOT EXISTS`, `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, `INSERT ... ON CONFLICT DO NOTHING`, etc. Migrations run on every boot.
3. **Mixed DDL/DML is fine.** Seeding default rows in the same migration that creates the table is the norm (see `066_hr_discipline_regulation.sql`).
4. **Prefer partial unique indexes over `UNIQUE` constraints** when seeding so `ON CONFLICT DO NOTHING` still works.
5. **Check the row shape of each `INSERT`** before pushing. A common bug is dropping a column in the DDL but forgetting to update the literal rows below.

### Writing a new migration

```bash
# highest-numbered file tells you the next number
ls artifacts/api-server/src/migrations | tail -5
# then create
touch artifacts/api-server/src/migrations/067_my_change.sql
```

Restart the API server to apply it; `lib/migrate.ts` will pick it up. `build.mjs` copies the SQL files into `dist/migrations` during `pnpm --filter @workspace/api-server run build`.

### Rolling back

There is no automated rollback. Write a new, compensating migration.

### 2.4 Regenerating `db/schema.sql` (Phase 2)

Whenever you land a migration that adds tables, columns, or constraints,
re-export the canonical schema so `pnpm db:bootstrap` keeps working
for fresh local instances:

```bash
# On Replit (or any environment where DATABASE_URL points at a fully
# migrated DB):
pnpm db:dump-schema

# Review the diff in db/schema.sql:
git diff db/schema.sql

# If it looks right, also re-export the reference seed if you touched
# any of the reference tables (companies, permissions, chart_of_accounts,
# job_titles, …):
pnpm db:dump-seed

# Commit + push as part of the same change-set as the migration.
git add db/schema.sql db/seed.sql artifacts/api-server/src/migrations/NNN_*.sql
git commit -m "feat(thing): NNN_my_change + regenerate schema dump"
```

Existing instances (Replit, staging, production) will apply the
incremental migration normally on next boot. The runtime migration
runner detects that `schema_migrations` already has rows and skips
loading the dump as a baseline — see `src/lib/migrate.ts`
`detectAndApplyBaselineIfNeeded` for the exact logic.

**Don't** hand-edit `db/schema.sql` — always regenerate it from a
real Postgres via `pg_dump`. Hand edits silently drift from the
actual schema and break the bootstrap on fresh clones.

## 3. Adding a new route

1. **Create the router** in `artifacts/api-server/src/routes/my-thing.ts`:

   ```ts
   import { Router } from "express";
   import { requirePermission } from "../lib/permissions.js";
   import { rawQuery, rawExecute, withTransaction } from "../lib/db.js";

   const router = Router();

   router.get("/", requirePermission("my:read"), async (req, res, next) => {
     try {
       const rows = await rawQuery(
         `SELECT * FROM my_table WHERE "companyId" = $1 ORDER BY id DESC LIMIT $2`,
         [req.scope.companyId, req.query.limit ?? 20],
       );
       res.json({ data: rows, total: rows.length });
     } catch (err) {
       next(err);
     }
   });

   export default router;
   ```

2. **Mount it in `routes/index.ts`** under the right prefix, *before* any fallback monolithic router. Sub-routers (`/hr/discipline`) must precede their parent (`/hr`).

3. **Seed the required permissions** in a new migration: insert rows into the role-permission table for `hr_manager`, `general_manager`, `owner` as appropriate. Use AND-semantics: if you write `requirePermission("my:read", "my:admin")` the user must hold **both**.

4. **Typecheck:** `pnpm --filter @workspace/api-server run typecheck`.

## 4. Adding a new frontend page

1. **Create the page** at `artifacts/ghayth-erp/src/pages/<module>/<name>.tsx`. Prefer existing patterns from the same module.

   Minimal skeleton:

   ```tsx
   import { Link } from "wouter";
   import { useApiQuery, apiFetch, getErrorMessage } from "@/lib/api";
   import { Card, CardContent, CardHeader, CardTitle } from "@/components/ui/card";
   import { DataTable, type DataTableColumn } from "@/components/ui/data-table";
   import { useToast } from "@/hooks/use-toast";

   export default function MyPage() {
     const { data, isLoading } = useApiQuery<{ data: any[] }>(["my-things"], "/my");
     return <DataTable columns={[]} data={data?.data ?? []} isLoading={isLoading} />;
   }
   ```

2. **Register the route** in `src/routes/<module>Routes.tsx`:

   ```tsx
   const MyPage = lazy(() => import("@/pages/<module>/<name>"));
   // …
   { path: "/<module>/<name>", component: MyPage, subKey: "<menu-group>" },
   ```

3. **Expose in the sidebar** in `src/components/layout/sidebar-layout.tsx`. Add a top-level entry and/or a child entry under the appropriate parent path.

4. **Typecheck:** `pnpm --filter @workspace/ghayth-erp run typecheck` (or just `npx tsc --noEmit` inside the package).

### Conventions (non-negotiable)

- **Routing:** `wouter`. `Link` uses `href=`, not `to=`. For params use `useRoute("/x/:id")`.
- **No modals for create/edit.** Use full pages navigated via `<Link>`. Allowed dialogs: `QuickPreviewDialog` (read-only) and `AlertDialog` (delete confirm).
- **Lists:** always support `?page=X&limit=20`.
- **DataTable prop:** `isLoading`, **not** `loading`.
- **Formatters:** use `lib/formatters.ts` for Arabic numerals, dates, and currency (`formatCurrency`, `formatDateAr`, `formatNumber`).
- **RTL:** the app renders `dir="rtl"`. Use logical properties (`me-2`, `ms-auto`, `pe-3`) rather than `mr-`/`ml-` when you need direction-aware spacing.
- **Error and empty states:** include an icon, an Arabic message, and a retry action for errors.

## 5. Seeding permissions for a new feature

Permissions are the single most common source of "it works for the owner but 403s for everyone else" bugs.

1. Pick a permission name in the form `<module>:<resource>:<action>` (e.g. `hr:discipline:approve`).
2. Add a seed `INSERT` to your migration for each role that should hold it:

   ```sql
   INSERT INTO role_permissions ("roleKey", "permission")
   VALUES
     ('hr_manager',       'hr:discipline:read'),
     ('hr_manager',       'hr:discipline:create'),
     ('hr_manager',       'hr:discipline:update'),
     ('general_manager',  'hr:discipline:read'),
     ('general_manager',  'hr:discipline:approve'),
     ('owner',            'hr:discipline:read'),
     ('owner',            'hr:discipline:approve')
   ON CONFLICT DO NOTHING;
   ```
3. **Use one permission per `requirePermission` call where possible.** `requirePermission("a","b")` requires **both**; if the business rule is "either", split into two routes or combine into a single permission.

## 7. Commit / branch hygiene

- Work on a feature branch; never push directly to `main`.
- Commit messages use a short imperative (Arabic or English both fine), with a body that explains the "why" when the diff is not self-evident.
- Do **not** commit `node_modules/.bin` symlinks that changed only because of your local machine path (a common pnpm artefact). Restore them before committing.
- Do **not** commit `.env` files.

## 8. Troubleshooting

| Symptom | Probable cause | Fix |
| ------- | -------------- | --- |
| `rawdb.ts` throws `DATABASE_URL is required` | Missing env var | Set `DATABASE_URL` in `.env`. |
| `JWT_SECRET is required for careers portal` | Missing env var | Set `JWT_SECRET` in `.env`. |
| 403 on a new route even as owner | Forgot to seed `role_permissions` for that perm | Add a migration that seeds the rows. |
| `requirePermission` fails for a role that "should" have the perm | AND semantics — the route requires two perms but the role has only one | Split or combine the permissions. |
| `ON CONFLICT` fails in a seed `INSERT` | No unique index / constraint covers the conflict target | Add a partial unique index in the same migration. |
| Frontend `Cannot find module 'react-router-dom'` | You used the wrong router | The app uses `wouter`. Import `Link`, `useRoute` from `wouter`. |
| `Link` click does nothing | Used `to=` instead of `href=` | Wouter's `Link` uses `href=`. |
| Migration isn't running | You edited `dist/migrations/*.sql` by hand | Edit the source under `src/migrations/` and rebuild. |
