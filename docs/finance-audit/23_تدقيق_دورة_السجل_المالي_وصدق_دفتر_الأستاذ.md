# 23 — تدقيق دورة السجل المالي وصدق دفتر الأستاذ (FIN-FINANCIAL-INPUTS-REALIGNMENT — #2231 الموسّع)

> **الموجة:** #2230 / #2231 الموسّع — **P1 Deep Audit، docs-only.** لا كود · لا UI · لا API · لا backend · لا SQL · لا migration · لا engine · لا COA · لا mapping · لا قيد · لا ترحيل · لا fallback · لا workaround.
> **النطاق:** خريطة صفحات المالية (إدخال/اعتماد/عرض) + **صدق دفتر الأستاذ** (تغطية الأبعاد، الباب الخلفي للقيد اليدوي، `source=fallback`، سوء استخدام 2160، المحركات الدورية، الإقفال) + جدوى `requiredDimension` وموضع عقد النزاهة.
> **يكمّل** المستند 22 (تدقيق صفحة المصروف + جذر `5510-0001`) — لا يكرّره؛ يُحيل إليه ويوسّعه إلى كامل الدورة.

## 0. فحص عدم التكرار والحدود

- **ما الموجود أصلًا؟** أساس محاسبي صلب: بابان للترحيل يتحققان من التوازن ووجود الحساب وإقفال الفترة وidempotency؛ أعمدة أبعاد كاملة على `journal_lines`؛ تقارير ربحية لكل كيان تقرأ من الأستاذ؛ مُصنِّف حسابات (`accountUsage`) بنمط ratchet؛ سجل أغراض مُحكَم (`account-purposes.ts`) لـFX/المخزون/الرواتب.
- **هل يخصّ المالية؟** نعم، مع حدّ صريح مع الأسطول/الموارد/المخزون/الموردين (§7).
- **هل يكرّر جردًا سابقًا؟** لا. الجرود 01–21 رصدت عرض الحالة والتكرارات عمومًا، والمستند 22 رصد صفحة المصروف. **لا يوجد تدقيق مخصّص لدورة السجل المالي كاملة + صدق الأستاذ.**
- **ما الذي لن يُلمس في P1؟** كل شيء — تقرير فقط.
- **الحدّ المعماري المُثبَت:** المالية تملك القيد/التوجيه/الترحيل؛ الأسطول يملك المركبة/العداد/سجل الوقود؛ الموارد تملك بيانات الموظف/الخصم؛ المخزون يملك المورد ككيان. المالية **تنسّق** عبر عقود/أحداث ولا تخترق.

## 3. صدق دفتر الأستاذ (الطبقة الموسّعة — المصدر: جرد الكود)

### 3.أ — بابان للأستاذ لا باب واحد ⚠️
| الباب | الملف | يُستخدم لـ | يتحقق |
|---|---|---|---|
| **أ (تشغيلي)** | `businessHelpers.createJournalEntry` (`:542`) | المصروفات/الفواتير/السندات/اليومية | توازن · وجود الحساب (`:599`) · إقفال الفترة (`:560`) · يكتب 27 عمودًا |
| **ب (typed)** | `gl/posting.postJournalEntry` | FX/مخزون/رواتب/جرد | توازن (`:85`) · وجود الحساب (`:142`) · إقفال (`:127`) · idempotency (`:104`) |

**أثر على العقد:** `assertDimensionContract` و«منع fallback» يجب تركيبهما عند **البابين** (أو chokepoint مشترك يستدعيانه). فحص وجود الحساب **موجود في كليهما** — **لا يُعاد كتابته** (يحسم تعديل #2233).

### 3.ب — تغطية الأبعاد: تناقض لا غياب
- `enrichJournalLines` (`journalLineDimensionalEnricher.ts:469`) يملأ `costCenterId` فقط (سلسلة مشروع>عقد>مركبة>قسم>فرع)؛ **لا يخترع** vendorId/clientId.
- أبعاد الكيان تُملأ بخطوة منفصلة `inferHeaderDimensionsFromSource` (`businessHelpers.ts:652`) محكومة بـ`switch` على `sourceType` (`:211–284`). المغطّى: `invoice→clientId`، `expense→employeeId`، `fleet_maintenance→vehicleId`، `umrah_sales_invoices→umrahAgent/Season`…
- **التسريب الصامت (مُعترَف به في التعليقات):**
  - `journalLineDimensionalEnricher.ts:160-164`: «callers … DON'T necessarily put the entity FK on every line → clientId/vendorId/umrahAgentId stay null».
  - **`vendorId` مُعلَن ويُطبَّق (`:178,:302`) لكن لا حالة `switch` تملؤه إطلاقًا** — فواتير الموردين/الذمم لا تُوسَم بالمورد.
  - `default: return empty` (`:282`) لأي sourceType خارج القائمة؛ والأخطاء **تُبتلَع** (`catch{return empty}` `:285`).
- **النتيجة:** التقرير «يطلع» رقمًا واثقًا لكنه ناقص (ربحية مركبة/مورد/موسم منقوصة عند غياب البُعد). ← هذا جوهر «مو حقيقي».

### 3.ج — الباب الخلفي للقيد اليدوي (أعلى خطورة)
- `journal-manual-create.tsx` يعرض `LineAllocationPanel` بكل الأبعاد، لكن الـbackend (`finance-hardening.ts`) **يُسقط الأبعاد**: مخطط Zod يقبل 4 فقط (`:61-70`) ثم `insertDraftManualJournal` يحفظ `{accountId,debit,credit,description}` فقط (`:403-413`). **فحرّيًا: قيد يدوي بحساب حر ومدين/دائن، وكل الأبعاد null — بالتصميم.** إقفال الفترة لا يُفحص عند الإنشاء (يؤجَّل للترحيل `:794`).

### 3.د — `source=fallback` علم ميّت
- مُعرَّف في `account-purposes.ts:71`، يُضبَط configured (`:146`)/fallback (`:164`). **يُحسَب ولا يُقرأ في الإنتاج** — كل المستهلكين الـ12 (cogsPosting/fx/inventory/mudad) يتجاهلون `.source`. (المُحلِّل الأقدم `getAccountCodeFromMapping` يكتب سطر `audit_log` بعد الحدث، لا منعًا.) ← موضع المنع: عند `getAccountForPurpose` للأغراض الواضحة، أو حارس عند مواضع النداء.

### 3.هـ — سوء استخدام 2160
- 2160 = «إيرادات مقبوضة مقدمًا» (`companyBootstrap.ts:297`).
- **استخدام مشروع:** دفعات العملاء المقدمة (`finance-invoices.ts:4078,4258`).
- **سوء استخدام مُثبَت:** `propertiesEngine.ts:198` — `property_maintenance_payable` يرجع fallback إلى `2160`، فيُدخِل التزام صيانة طرف ثالث في حساب الإيراد المؤجل (تضخيم الإيراد المؤجل + تصنيف خاطئ). 
- **نمط تاريخي أُصلح (سابقة):** `umrahInvoicingEngine.ts:714-720` نقل ضريبة العمرة من 2160 إلى 2300 بعد أن «نقّص ضريبة المخرجات المبلّغة».

## 4. عقد النزاهة — الجدوى وموضع التركيب (أسئلة #2231 الخمسة)

| السؤال | الإجابة (مسنودة بالكود) |
|---|---|
| **`account-purposes` يحمل `requiredDimension`؟** | هيكليًا نعم (Record مُحكَم مثل `FALLBACK_CODE`)، **لكنه يغطّي FX/مخزون/رواتب فقط**؛ أغراض المصاريف التشغيلية خارج هذا الـenum (في عالم `accounting_mappings` الحر + `MAPPING_INTENT`). فـ`requiredDimension` للتشغيلي يحتاج: إمّا **توسيع الـenum ليشمل أغراض opex** (توحيد النظامين)، أو **محور تصنيف موازٍ** (توسيع `accountUsage`/`financeAccountClassifier` بمحور «كائن التكلفة»، أو سجل `account_dimension_requirement`). التوصية: محور تصنيف على مستوى **صنف الحساب/الغرض** لا قائمة يدوية. |
| **`financePostingPolicy` هو المكان الصحيح؟** | **نعم.** الوحدة صريحة: «مصدر الحقيقة — الـbackend يرفض حتى لو تجاوز المستخدم الواجهة»، وتطبّق **نفس نمط ratchet** المطلوب: مُصنَّف→يُفرَض، غير مُصنَّف→soft + تقرير فجوة (`assertPaymentSourceAllowed:54`). `assertDimensionContract` يسكن هنا شقيقًا. |
| **ماذا في `posting.ts` حتى لا نكرّر فحص الحساب؟** | فحص وجود الحساب/قابلية الترحيل **موجود** (`posting.ts:142-158`)، ومثله في `createJournalEntry` (`businessHelpers:599`). #2233 **لا يعيده**؛ يضيف منع fallback + عقد البُعد. |
| **أين منع `source=fallback`؟** | عند `getAccountForPurpose` (`account-purposes.ts:160-166`) للأغراض الواجب ضبطها، و/أو حارس عند مواضع النداء في المسار التشغيلي حيث السيناريو «واضح». اليوم العلم **يُحسَب ولا يُقرأ**. |
| **أين `assertDimensionContract`؟** | عند **البابين** (`gl/posting.postJournalEntry` قبل حلقة الإدراج ~`:137`، و`createJournalEntry` قبل إدراجها ~`:596`) — مثاليًا دالة مشتركة `assertPostingContract(line,ctx)` يستدعيانها. والمسار التشغيلي يضيف فحصًا مبكرًا في الواجهة لرسالة صحيحة. |

**رسالة الخطأ الصحيحة (تحسم #2233):** بدل «الحساب 5510-0001 غير موجود» بعد الحفظ → «**إعداد التوجيه المحاسبي غير مكتمل: لا يوجد حساب فعّال قابل للترحيل للغرض `vehicle_fuel_expense`**» مبكرًا.

## 5. Blueprint (المخرج الإلزامي)

- **صفحة الإدخال (Financial Document Workspace):** أعلى=بيانات السجل · يمين=المورد+البنود · يسار=عارض المرفق · أسفل=معاينة القيد+الأثر+preflight. الحساب/مركز التكلفة **read-only مشتقّان**؛ التجاوز خلف صلاحية+سبب.
- **صفحة الاعتماد:** مراجعة لا إدخال — توحيد `ApprovalActions`/الصفحتين المخصّصتين؛ لا إنشاء موازٍ (#2239).
- **صفحة العرض:** قراءة وتتبّع — فكّ الخلط في `invoice-detail` وإيقاف التحرير غير المنضبط (#2240).
- **المرفق:** عارض جانبي (صورة/PDF) في الإدخال/الاعتماد/العرض + رقم تسلسل داخلي + ربط بالسجل (#2237) — ترقية لا بناء موازٍ.
- **البنود:** تقود الحساب/المبلغ/الضريبة من ذاكرة المورد (#2235).
- **القيد:** ترقية `LiveImpactPreview` الموجود (مستهلَك في عدة مسارات) إلى واجهة عقد النزاهة (#2238) — جرد المستهلكين أولًا + اختبارات انحدار.
- **المورد:** `suppliers` الكنسي؛ `vendorId` بُعد يشير إلى `suppliers.id`؛ لا جدول vendor منفصل؛ المخزون يُنشئ، المالية تضيف الملف المالي على نفس الصف (#2234/#2235/#2241).
- **preflight:** عقد النزاهة في `financePostingPolicy` عند البابين، ratchet صنفًا-بصنف.
