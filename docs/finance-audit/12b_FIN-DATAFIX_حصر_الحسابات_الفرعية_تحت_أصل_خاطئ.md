# 12b — FIN-DATAFIX: حصر الحسابات الفرعية القديمة تحت أصل خاطئ (read-only)

> المهمة #2090 (= FIN-SUB-02 في جرد #2083) · متابعة #2070. **هذه الشريحة تقرير حصر فقط — لا تعديل بيانات، لا نقل حسابات، لا تعديل أرصدة، لا ترحيل.** يُعرض على المالك قبل أي تصحيح فعلي.

## الخلفية
قبل #2070 كان `createSubsidiaryAccountsForEntity` يفتح حساب كل كيان تحت **أصل مغروس خاطئ** لا يطابق شجرة البذرة ولا SOCPA:
- ذمم العميل تحت **1111 (الصندوق/نقد)** بدل 1130.
- سلفة الموظف تحت **1121 (بنك)** بدل 1140.
- عهدة الموظف/السائق تحت **1131 (عملاء)** بدل 1142/1113.
- المورّد تحت كود غير موجود (فلم يُفتح له حساب).

#2070 أصلح **الإنشاء الجديد** فقط. هذا الحصر يجد **الأوراق القديمة** في البيئات القائمة لتُخطَّط معالجتها (PR منفصل، بموافقتك).

## الأداة (read-only)
| الملف | الدور |
|---|---|
| `scripts/finance-audit/fin_datafix_subsidiary_parent_audit.sql` | استعلام **SELECT فقط** يكشف كل صف `subsidiary_accounts` حسابه تحت أصل يخالف الأصل الضابط المتوقَّع لنوع الكيان/الحساب. |
| `scripts/finance-audit/fin-datafix-report.mjs` | مُشغِّل **read-only** (يفتح `BEGIN TRANSACTION READ ONLY`) يطبع جدولًا + ملخّص خطورة ويكتب تقرير markdown في `docs/finance-audit/findings/` (غير مُتتبَّع في git). |

### آلية الكشف
لكل صف فرعي: يُقارَن **الأصل الحالي** لحسابه بـ **الأصل الصحيح المتوقَّع** المُشتق بالنيّة (نفس خريطة #2070 `resolveSubsidiaryParent` بالحرف: النوع + كلمات الاسم، أقصر كود). إن اختلفا → **مشتبه**.

### الأعمدة (لكل مشتبه — كما طلب المالك)
الحساب · الأصل الحالي · **الأصل الصحيح المقترح** · الكيان المرتبط (نوع/معرّف/اسم) · الرصيد الحالي · القيود المرتبطة · القيود المُرحَّلة · سبب الاشتباه · **درجة الخطورة** · **التصرف (آلي/مراجعة مالية)**.

### الخطورة والتصرف
- **high** — عليه رصيد ≠ 0 أو قيود مُرحَّلة → `needs_finance_review` (يحتاج قيد تحويل رصيد + مراجعة، لا إعادة ربط مباشرة).
- **medium** — له قيود غير مُرحَّلة لكن بلا رصيد.
- **low** — رصيد صفر وبلا قيود → `auto_fixable` (إعادة تصنيف بحتة، آمنة للتصحيح الآلي **بعد موافقتك**).

## التشغيل على الإنتاج
```bash
DATABASE_URL=postgres://…(الإنتاج، read-only credentials يُفضَّل)… \
  node scripts/finance-audit/fin-datafix-report.mjs
```
الأداة تفتح معاملة `READ ONLY` فترفض القاعدة أي كتابة بنيويًا. النتيجة: جدول + تقرير markdown تعرضه عليّ/عليك.

## النتيجة على قاعدة رأس main (الوكيل)
**0 مشتبه** — متوقَّع: القاعدة المُزوَّدة حديثًا (بعد #2070) لا تحوي أوراقًا قديمة. الأداة مُثبتة بالكشف على بيانات مُصطنعة داخل معاملة مُتراجَعة (انظر `finDatafixSubsidiaryAudit.dynamic.test.ts` — 3/3): تكشف ذمم-تحت-نقد وعهدة-تحت-عملاء بالأعمدة الصحيحة، ولا تُبلّغ عن ورقة سليمة، **ولا تُبقي أثرًا** (rollback مؤكَّد).

## ما بعد هذا الحصر (لا يبدأ إلا بموافقتك)
PR تصحيح منفصل (#2090 المرحلة الثانية): لكل مجموعة معتمدة منك —
- `auto_fixable` → إعادة ربط `parentId`/`code` للأصل الصحيح (إعادة تصنيف، تقرير قبل/بعد).
- `needs_finance_review` → قيد تحويل رصيد للورقة الصحيحة ثم أرشفة القديمة، **بمعيار: إجمالي ميزان المراجعة قبل = بعد**.
لا تُهاجَر بيانات حيّة قبل عرض الحصر واعتماده لكل مجموعة.
