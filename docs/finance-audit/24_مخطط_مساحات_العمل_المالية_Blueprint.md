## 0. المبدأ الحاكم
> **«السهولة لغير المالي» نتيجة لـ«نزاهة الأستاذ».** المستخدم يُدخل **الواقعة + المستند**، والنظام **يشتقّ المحاسبة** ويعرضها للمراجعة قبل الحفظ. كل مستند مالي = ثلاث حقائق: **تشغيلية** (يُدخلها) · **تجارية** (مورد/فاتورة/بنود — جزء يُدخل وجزء يُستحضر) · **محاسبية** (قيد/حساب/بُعد — تُشتقّ ١٠٠٪).

---

## 1. لوحة السياق الموحّدة (المكوّن الأوحد المعاد استخدامه)

مكوّن واحد `FinanceContextPanel` يُستخدم في **الإدخال + الاعتماد + العرض + القيد اليدوي المربوط** — يضمن أن «السبب + الحساب المشتقّ + الأبعاد + الأثر» تُعرض بنفس الشكل في كل مساحة، ولا تتباعد.

**الأساس الموجود (نُرقّيه لا نبنيه):**
- `FinanceOperationContextPanel` + `AllocationTargetSelect` (`scenario-model.ts`) — موجود؛ يُعمّم.
- `LiveImpactPreview` (`/finance/expenses/impact-preview`) — موجود ومستهلَك في عدة مسارات؛ **يُرقّى لا يُستبدل** (#2238، بعد جرد المستهلكين).

**ما تعرضه اللوحة الموحّدة (read-only للمستخدم العادي):**
الحساب المشتقّ · سبب التوجيه (السيناريو/البند/المورد الذي أنتجه) · مركز التكلفة المشتقّ · الأبعاد (vehicle/property/project/vendor/client/branch) · المتسبّب (`costBearer`) · الأثر التشغيلي المتوقّع · حالة preflight (موجود/قابل للترحيل/الفترة مفتوحة/مصدر الصرف يطابق طريقة الدفع).

## 2. مساحة الإدخال — Financial Document Workspace

تخطيط واحد لكل سجلّات الإدخال (مصروف/فاتورة مورد/سند صرف/قبض/عهدة/قيد يدوي مربوط)، يتكيّف بالسيناريو والدور.

```
┌────────────────────────── أعلى: بيانات السجل ──────────────────────────┐
│ نوع السجل · الحالة · التاريخ · الفترة · الفرع · الطرف(مورد/عميل/موظف)   │
│ رقم فاتورة المورد · رقم تسلسل داخلي* · طريقة الدفع · مصدر الصرف          │
│ حالة الدفع · حالة الترحيل · رقم القيد(إن وُجد) · رقم سند الصرف(إن وُجد) │
├──────────────────── يمين (منطقة العمل، RTL) ──┬─── يسار: عارض المرفق ───┤
│ المورد + فرع/موقع المورد + رقم/تاريخ الفاتورة │ صورة/PDF · تكبير/تدوير  │
│ البنود (من ذاكرة المورد): بند·كمية·وحدة·سعر· │ صفحات · نوع/حالة المرفق │
│   ضريبة·إجمالي · السيناريو والطرف لكل بند      │ رقم تسلسل داخلي*        │
│ الحقول التشغيلية حسب السيناريو (مركبة/عداد..) │ ربط واضح بالسجل         │
├──────────────────── أسفل: معاينة القيد والأثر (read-only) ──────────────┤
│ مدين/دائن · الحساب المشتقّ · الضريبة · مصدر الصرف · مركز التكلفة · أبعاد │
│ متوازن؟ · الحسابات موجودة/قابلة للترحيل؟ · الفترة مفتوحة؟ · الأثر · أخطاء│
└── قسم override المتقدم (مطوي، بصلاحية+سبب+Audit) — تعديل الحساب/الأبعاد ──┘
```
\* رقم التسلسل الداخلي = متطلب numbering center (لا يُنفَّذ الآن — قرار المالك 3).

**قاعدة الحقول (قرار المالك 5):** الحساب ومركز التكلفة **read-only مشتقّان** + سبب التوجيه + حالة التحقق؛ لا يُحرَّران إلا عبر override بصلاحية وسبب وAudit. ما يُدخله المستخدم في «وقود مركبة»: المورد · المركبة · (السائق) · البند · اللترات · السعر · العداد · المرفق · مصدر الصرف. الباقي يُشتقّ (تفصيل المستند 22 §3).

**الإكمال التدريجي:** السجل يُحفظ ناقصًا في **المسودة** (نص حر للمحطة مسموح مسودةً فقط — قرار 1)، ويُكمَّل/يُعتمد لاحقًا. **بوّابة الترحيل:** لا يترحّل حتى يكتمل: مورد محفوظ + حساب مشتقّ قابل للترحيل + بُعد مطلوب موجود + مرفق إن لزم (قرار 1، ويُنفَّذ إنفاذه في #2233 ratchet).

## 3. مساحة الاعتماد — مراجعة لا إدخال (قرار 6)

**لا تكون نسخة من الإدخال.** تعرض: ملخص السجل · المورد · البنود · المرفق (بجانب البنود للمقارنة) · القيد المتوقّع · سبب التوجيه · الأثر · سجل التعديلات/override. الأفعال: اعتماد/رفض/إرجاع/طلب مرفق/طلب تصحيح. **لا تحرير عشوائي للحسابات/الأبعاد** — أي تغيير جوهري = إرجاع أو override مسجّل.

**القيد اليدوي نوعان (قرار 6):**
- **نوع أ (محاسبي بحت):** موافقة مالية + سبب.
- **نوع ب (مرتبط بواقعة):** كائن تشغيلي إلزامي + سبب عدم استخدام المسار النظامي + موافقة أعلى.

**إعادة الاستخدام:** `ApprovalActions` (`@workspace/workflow-kit`) + صفحتا الاعتماد المخصّصتان القائمتان (`budget-approvals`، `expense-bulk-approvals`) — **توحيد لا إنشاء موازٍ** (#2239). المُجمِّع `approvals-inbox` يبقى بوابة دخول.

**إصلاح إلزامي:** فكّ الخلط في `invoice-detail.tsx` (تحرير+دفع+اعتماد+ترحيل في عرض واحد — المستند 23 §1.ج)، وإيقاف التحرير غير المنضبط في `account/vendor/budget/recurring/fixed-asset-detail`.

## 8. نموذج المورد — بُعدي (الحسم المعماري)

> **«حساب فرعي للمورد» = بُعد على القيد، لا ورقة في الشجرة.** (المستند 23 — يقتل فئة 5510.)

- `suppliers` الكيان الكنسي (المخزون يُنشئه؛ المالية تضيف الملف المالي على نفس الصف — لا جدول vendor منفصل، لا ازدواج `warehouse/suppliers-create` ↔ `finance/vendors-create`).
- `vendorId` + `branchId` بُعدان على `journal_lines` → رصيد المورد ومفصّلًا لكل فرع بترشيح الأستاذ (تقارير `vendor-detail` القائمة).
- **فرع/موقع المورد (قرار 2):** الهدف `supplierLocationId`؛ مؤقتًا `supplierLocationName` في metadata **مرتبطًا بـ`supplierId`** — لا جدول جديد الآن.
- **ZATCA:** هوية المورد (اسم/رقم ضريبي/عنوان) مستر-داتا على المستند، يحقّقها `vendorId` القوي.
