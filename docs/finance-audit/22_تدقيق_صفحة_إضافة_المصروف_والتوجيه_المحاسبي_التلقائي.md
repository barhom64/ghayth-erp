# 22 — تدقيق صفحة «إضافة مصروف جديد» والتوجيه المحاسبي التلقائي

> **الموجة:** `FIN-EXPENSE-UX-ROUTING-WAVE` — **المرحلة P1 — Audit فقط (تدقيق وتشخيص، بلا تنفيذ).**
> **المحفّز:** فشل حفظ مصروف وقود لمركبة برسالة `الحساب "5510-0001" غير موجود في شجرة الحسابات`، رغم وضوح السيناريو التشغيلي.
> **النطاق:** صفحة إضافة المصروف + سلسلة التوجيه المحاسبي لسيناريو «وقود + مركبة».
> **هذا المستند لا يغيّر أي كود.** لا UI · لا API · لا backend · لا SQL · لا محرك محاسبي · لا شجرة حسابات · لا إنشاء حسابات أو mappings · لا قيود/ترحيل · لا workaround/fallback جديد.

## 3. تحليل سيناريو «وقود + مركبة» — السلسلة الصحيحة المتوقَّعة

```
المستخدم يختار:  نوع العملية = وقود  →  (يقود) سيناريو = مركبة  →  المركبة المحددة
                 + اللترات + سعر اللتر + قراءة العداد + المحطة + طريقة الدفع + مصدر الصرف
─────────────────────────────────────────────────────────────────────────────
النظام يجب أن يشتق تلقائيًا (بلا تدخّل مستخدم):
  • بند المصروف (الحساب المحاسبي)  ← حساب وقود المركبة الصحيح الموجود فعلًا
  • الوحدة                          ← «لتر»
  • المبلغ                          ← اللترات × سعر اللتر
  • مركز التكلفة                    ← مركز تكلفة المركبة (costCenterSource:"vehicle")
  • البيان                          ← «مصروف وقود / <المركبة> / <المبلغ>»
  • الأثر التشغيلي                  ← سجل وقود (fleet_fuel_logs) + تحديث العداد
  • حالة الدفع                      ← «مدفوع» عند تحديد مصدر الصرف
  • حالة الترحيل                    ← حسب سياسة الاعتماد
```

**الواقع الحالي (أين تنكسر السلسلة):**
1. **«نوع العملية = وقود» لا يقود شيئًا محاسبيًا** — هو تجميلي: يؤثّر فقط على البيان التلقائي (`generateAutoDescription`, `:105`) وعلى الربط الحكومي/إلزام المرفق. لا يضبط الحساب ولا السيناريو.
2. **السيناريو الفعلي يأتي من «ربط المصروف بـ» = مركبة**، وهدفه يعطي `accountPurpose = "vehicle_expense"` (عام)، **وليس** `vehicle_fuel_expense`، لأن الصفحة تقرأ `TARGET_HINTS` (`scenario-model.ts:273`) لا السجل الأغنى `FINANCE_SCENARIOS.vehicle_fuel` (`:107`).
3. **ازدواج نموذج السيناريو:** يوجد سجلّان — `FINANCE_SCENARIOS` (فيه `vehicle_fuel` بالحقول والمحاسبة الصحيحة) موصوف صراحةً بأنه «المصدر الوحيد»، لكنه **Phase 1 غير مُستخدَم في العرض** (التعليق `:19-28` يقرّ أن Phase 2 لم تُنفّذ)، بينما العرض يعتمد `TARGET_HINTS` الأبسط. فالنموذج الصحيح موجود لكنه غير موصول.
4. **بند المصروف يبقى يدويًا مطلوبًا** (`:496` + تحقق `:305`)، فالمستخدم يضطر لاختيار حساب بنفسه رغم وضوح السيناريو.
5. **اللترات/السعر يُدخلان مرتين** (حقول الوقود + حقول «تفاصيل البند» الظاهرة لأن `expenseType==="fleet"`, `:511`)، وحقول «تفاصيل البند» فقط هي التي تملأ المبلغ تلقائيًا.

## 4. تحليل الحساب `5510-0001`

| السؤال | النتيجة |
|---|---|
| **من أين جاء؟** | من **اختيار الواجهة لبند المصروف** (`accountCode`)، لا من محلّل خادمي للوقود/المركبة. الورقة `5510-0001` تظهر في قائمة «بند المصروفات» من `/finance/accounts` (`expenses-create.tsx:148,156`). |
| **هل هو في `chart_of_accounts`؟** | ليس بالضرورة وقت الحفظ. هو **ورقة فرعية لكل مركبة**؛ قد لا تكون أُنشئت (أو أُنشئت لاحقًا/بسياق مختلف). |
| **هل مولّد من اسم/معرّف المركبة؟** | نعم — من **معرّف** المركبة: `5510-` + `padStart(4,"0")` (`accounting-engine.ts:738`). المركبة #1 ⇒ `5510-0001`. |
| **هل هو حساب فرعي كان يجب إنشاؤه؟** | نعم. يُنشأ في `createSubsidiaryAccountsForEntity` لنوع `vehicle`، spec الوقود `parentCode:"5510"` (`:713`). |
| **هل من `accounting_mappings`؟** | لا مباشرةً. الـmapping يعطي الأب `5510` (`fleet_fuel_expense`, migration 254 / `fleetEngine.ts:51`)، لا الورقة `5510-0001`. |
| **هل هو fallback؟** | لا. على مسار المصروف اليدوي **لا يوجد fallback** إلى الأب `5510`. (الـfallback موجود فقط في `fleetEngine.ts:272` غير المستدعى هنا.) |
| **هل selector يعرضه دون تحقق؟** | نعم — القائمة تعرض كل ما يبدأ بـ`5` من `/finance/accounts` دون التأكد من بقائه قابلًا للترحيل/غير محذوف وقت الحفظ. |
| **هل API يقبل حسابًا غير موجود حتى لحظة الحفظ؟** | نعم. `POST /finance/expenses` يمرّر `accountCode` كما هو إلى سطر القيد (`finance-journal.ts:833`) ثم `postJournalEntry`؛ ولا preflight قبل ذلك. |
| **أين يجب أن يتوقف الخطأ؟** | يقع حاليًا **متأخرًا** عند الحفظ داخل `createJournalEntry` (`businessHelpers.ts:599`، ضمن معاملة الترحيل `finance-journal.ts:860/885`). يجب أن يتوقف **مبكرًا**: عند الاختيار/المعاينة (preflight) — أو ألّا يُطلب الاختيار اليدوي أصلًا للوقود+مركبة (اشتقاق read-only). |

### آلية الإنشاء «أطلِق وانسَ» (best-effort) — جذر الغياب
- الورقة الفرعية تُنشأ عند إنشاء المركبة عبر **استدعاء غير منتظَر** مع ابتلاع الخطأ: `createSubsidiaryAccountsForEntity(...).catch(...)` (`fleet.ts:569`، ومكرّر `:589`).
- إذا تعذّر إيجاد الأب `5510` (شجرة مختصرة) فإن الحلقة **تتخطّى** (`accounting-engine.ts:736`) وتُسجّل `parentFailures` **والمركبة تُحفَظ بنجاح** — فتبقى المركبة بلا ورقة وقود.
- النتيجة: قد توجد المركبة بلا `5510-0001`، ومسار المصروف اليدوي لا يشتقّ الحساب الآمن (لا يستدعي `resolveVehicleAccountCode` ولا fallback الأب)، فيصل كود غير موجود إلى الحفظ ويفشل صلبًا.

### تصنيف الخلل (الإجابة الصريحة المطلوبة)
**الخلل مركّب، وأبرزه بالترتيب:**
1. **غياب preflight validation** على وجود الحساب قبل الحفظ (السبب المباشر لظهور الرسالة وقت الحفظ بدل منعها مبكرًا).
2. **حساب فرعي غير منشأ** (إنشاء best-effort يُتخطّى بصمت عند غياب الأب) — السبب الجذري لعدم وجود `5510-0001`.
3. **selector يعرض حسابًا قد لا يكون صالحًا** وقت الحفظ، ويطلب قرارًا محاسبيًا من المستخدم بلا داعٍ.
4. **غياب اشتقاق/fallback خادمي** على مسار المصروف اليدوي (بخلاف fleetEngine) — لا توجيه تلقائي ولا رجوع للأب `5510`.

*(ليس: mapping ناقص للأب — الأب 5510 مبذور؛ وليس fallback خاطئ — لا يوجد fallback أصلًا على هذا المسار.)*

| كمية الوقود | عدد اللترات (#10) + الكمية (#18) | حقلان منفصلان لنفس القيمة؛ أحدهما فقط يملأ المبلغ. |
| سعر الوقود | سعر اللتر (#11) + سعر الوحدة (#20) | تكرار؛ تباعد محتمل بين القيمتين. |
| الحساب المحاسبي | بند المصروف اليدوي (#15) ⟷ ما يجب أن يشتقّه السيناريو | المستخدم يقرّر يدويًا ما يملك الخادم اشتقاقه. |
| مركز التكلفة | مركز التكلفة (#26) + القسم (#25) + المشروع (#27) | ثلاثة مداخل متداخلة؛ والمركبة يجب أن تشتقّ مركزها. |

**ما يجب أن يبقى للمستخدم (إدخال تشغيلي):** المركبة، اللترات، سعر اللتر، قراءة العداد، المحطة، طريقة الدفع، مصدر الصرف، المرفق، (واختياريًا السائق/المرجع/الضريبة).
**ما يجب أن يصبح آليًا (اشتقاق):** بند المصروف (الحساب)، الوحدة، المبلغ، مركز التكلفة، البيان، سجل الوقود، تحديث العداد، حالتا الدفع/الترحيل.

## 6. حدود المالية والأسطول (الحدّ المعماري المُثبَت)

| المالك | يملك | الدليل |
|---|---|---|

## 7. خطة التنفيذ اللاحقة (مقترحة — لا تُنفَّذ في P1)

PRs صغيرة، كلٌّ بسطح/مسؤولية واحدة، بفحص عدم تكرار وبوابة guard+clean:

| PR | العنوان | السطح | المخاطرة |
|---|---|---|---|
| **P2** | preflight validation يمنع الحساب غير الموجود قبل الحفظ (وصل `preflightAccountCodes` أو معاينة تتحقق من الوجود) — رسالة مبكرة عند الاختيار/المعاينة | backend + ربط FE خفيف | منخفضة (حارس، لا منطق قيد جديد) |
| **P3** | جعل بند المصروف **مشتقًّا read-only** لسيناريو «وقود + مركبة» (مع إبقاء التجاوز اليدوي المتقدم للمعتمد) — تمرير `vehicleId` إلى `impact-preview` واستهلاك الحساب المشتق | FE (`expenses-create.tsx` + `impact-preview` payload) | منخفضة-متوسطة |
| **P4** | ضمان وجود الورقة الفرعية للمركبة عند السيناريو (اشتقاق آمن عبر `resolveVehicleAccountCode` + fallback الأب `5510` على مسار المصروف، أو إنشاء مضبوط بسياسة) — معالجة جذر «best-effort المُتخطّى» | backend (`finance-journal` ↔ `fleetEngine`/`accounting-engine`) | متوسطة (يلمس الإنشاء/الترحيل — يحتاج تفويضًا دقيقًا) |
| **P5** | تنظيف تكرار «تفاصيل البند» (اللترات/الوحدة/الكمية/السعر) مقابل حقول الوقود — حقل واحد لكل قيمة، واشتقاق المبلغ | FE | منخفضة |
| **P6** | توليد البيان تلقائيًا للسيناريو (تفعيل افتراضي + صياغة أغنى تشمل المركبة/اللترات/العداد) | FE | منخفضة |
| **P7** | توحيد نموذج السيناريو: جعل العرض يقرأ `FINANCE_SCENARIOS` (Phase 2 المؤجّلة) بدل `TARGET_HINTS` فيصبح `vehicle_fuel` هو المحرّك | FE + parity test | متوسطة |
| **P8** | اختبار E2E: حفظ مصروف وقود ينجح، يُنشئ سجل وقود + قيدًا متوازنًا على حساب المركبة الصحيح، ويُحدّث العداد | اختبار فقط | منخفضة |

**أول PR تنفيذي موصى به:** **P2 (preflight)** — يوقف النزيف فورًا (يمنع وصول حساب غير موجود إلى الحفظ ويعطي رسالة مبكرة واضحة)، بأقل مخاطرة، ويمهّد لـP3/P4.
