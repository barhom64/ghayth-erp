# غيث — خطة التسليم لكل وكيل (Agent Handoff Protocol)

> **مرجع تابع لـ`غيث_العقيدة_الحاكمة.md`.**
> هذا الملف يعرّف بروتوكول استلام وتسليم المهام لكل وكيل تلقائي (Claude Code, Cursor, GPT, …) يعمل على هذا المستودع.
> أي مهمة تُعطى لوكيل بدون اتباع هذا البروتوكول — قابلة للرفض من المُلّاك.

---

## §0 — الفلسفة

**الوكيل ليس مهندساً مبتدئاً ينتظر إذن لكل سطر.** لكنه أيضاً **ليس مالك القرار التشغيلي**.

الحدّ الفاصل:
- ✅ يجوز للوكيل: استكشاف الكود، كتابة slice محدد بحدوده، ترقية اختبار, دمج PR بعد أن guard أخضر.
- ❌ ممنوع للوكيل: تفعيل سلوك تشغيلي جديد (engine activation, default flip, schema breaking), فتح اتفاقيات قانونية, لمس migrations حساسة, التظاهر بالاكتمال.

---

## §1 — قبل ما يبدأ: عقد الاستلام (Intake Contract)

كل مهمة يستلمها الوكيل يجب أن تأتي بالشكل التالي. لو نقص بند — يطلب الوكيل توضيحاً ولا يبدأ.

### 1.1 ملف الـIntake (يكتبه المالك أو وكيل سابق):

```yaml
# مهمة: <عنوان قصير>
issue: #<rqq>             # رقم Issue على Github
slice_id: <eg: U-04-P5>   # ID المختصر من roadmap
classification: 🟢 autonomous | 🟠 borderline | 🔴 hard-pause
authority:
  owner_decision_required: true | false
  if_true_link: <لينك للقرار>
scope:
  touches:
    - routes/<file>.ts
    - lib/engines/<file>.ts
    - …
  must_not_touch:
    - any migration > 290
    - financial engine (until SLICE-B)
    - …
permanent_hard_rails_applicable:
  - no JE outside engine
  - no client creation
  - …
deliverable:
  - code change description (1-line)
  - smoke pin description
  - PR link expected
acceptance_criteria:
  - vitest run X — green
  - tsc --noEmit — clean
  - guard — green
  - merged via treadmill (no override)
```

### 1.2 إذا الـIntake ناقص:

```
الوكيل يكتب comment على الـIssue:
"الـIntake ناقص الـ[<bullets>]. لن أبدأ قبل توضيحها."

ثم ينتظر.
```

## §2 — مرحلة الاستكشاف (Discovery)

قبل ما يكتب سطراً، الوكيل يلتزم:

1. **يقرأ `غيث_العقيدة_الحاكمة.md`** — لا يُفترض إن المبادئ معروفة.
2. **يقرأ `غيث_دليل_المكتبات_المشتركة.md`** — يحدد المكتبة الصحيحة.
3. **يقرأ Issue + audit doc + roadmap للـslice**.
4. **يفحص المسار الكودي:**
   - يستخدم `Grep` / `Glob` لاستكشاف الكود.
   - **لا يستخدم Bash لقراءة الملفات** — استخدم `Read`.
5. **يحدد slice سطح صغير قدر الإمكان:**
   - لو السطح > 5 ملفات أو > 300 سطر — يقسّمه أو يسأل المالك.

**يتوقف عن الاستكشاف إذا:**
- وجد slice مكسور (CI أحمر) لم يكن في الـIntake → يكتب comment.
- وجد intent غامض في الـIssue → يطلب توضيحاً.
- اكتشف conflict مع PR مفتوح آخر → ينتظر دمجه أو يسأل أولوية.

## §3 — مرحلة التنفيذ (Execution)

### 3.1 الـbranch

```
git checkout main
git fetch origin main && git reset --hard origin/main
git checkout -b claude/<slice-id>-<short-description>
```

**اسم الـbranch:** يبدأ بـ`claude/`، يحوي slice id، short description (lowercase + dashes).

### 3.2 ترتيب الكتابة

1. **migration** أولاً (إذا فيه).
2. **schema dump** ثانياً (`db/schema_pre.sql`).
3. **engine / lib** ثالثاً.
4. **route** رابعاً.
5. **smoke** خامساً.

### 3.3 الـsmoke

كل slice له smoke واحد على الأقل. الـsmoke يحوي:
- **§A** — وجود الـunit (route, function, migration).
- **§B** — السلوك المباشر (returns 200, dispatches correct branch).
- **§C** — tenant scope (companyId + deletedAt).
- **§D** — read-only / no-write على الـhandler.

### 3.4 الـvalidation المحلية

```bash
cd artifacts/api-server
npx vitest run tests/unit/<sliceSmoke>.test.ts
npx tsc --noEmit
```

**كلاهما يجب أن يكون أخضر قبل الـcommit.**

### 3.5 الـcommit

```
git add -A
git commit -m "<type>(<scope>): <slice-id> — <one-line summary>

<paragraph 1: what + why>

<paragraph 2: permanent hard rails respected (3 bullets)>

<paragraph 3: smoke pins (4 bullets)>

https://claude.ai/code/session_<id>"
```

### 3.6 الـpush + PR

```bash
git push -u origin claude/<branch>
gh pr create / mcp__github__create_pull_request
```

PR description يحوي:
- ✅ ملخص (1 paragraph).
- ✅ Permanent Hard Rails (3-5 bullets).
- ✅ Smoke pins (table).
- ✅ Test plan checklist.
- ✅ session link.

## §4 — مرحلة الـmerge (Treadmill)

### 4.1 الحلقة الرسمية

```
1. أنتظر guard أخضر (5-7 دقيقة عادة)
2. أحاول merge:
   - لو ✅ نجح → انتهى
   - لو ❌ "branch behind" → update_pull_request_branch + رجوع للخطوة 1
3. لو ❌ guard أحمر:
   - أقرأ log الـguard
   - أحدد السبب
   - أكتب fix commit (لا تجمع مع الـoriginal)
   - أنتظر guard أخضر
   - رجوع للخطوة 2
```

### 4.2 الحالات الخاصة

| الحالة | الإجراء |
|---|---|
| Guard أحمر بسبب يخص الـslice | إصلاح + commit + push |
| Guard أحمر لسبب غير معروف بعد 2 محاولات | **توقف** + اكتب comment للمالك |
| الـmain تحرك > 5 مرات تواصلاً | **توقف** + اكتب comment "treadmill loop, intervention needed" |
| Conflict مع PR آخر | لا تحل تلقائياً — **اسأل** |

### 4.3 ممنوع override

❌ ممنوع `--force` على main.
❌ ممنوع `--no-verify` على commits.
❌ ممنوع `gh pr merge --admin`.
❌ ممنوع disable branch protection.
❌ ممنوع نسخ guard config وتعديله ضمن نفس الـPR.

## §5 — التواصل (Communication)

### 5.1 متى الوكيل يكتب نص للمالك

| الحالة | شكل الرد |
|---|---|
| البداية | "بدأت [slice]، أتوقع 30-45 دقيقة + treadmill" |
| توقفت لسبب | "توقفت لأن [reason]. ينتظر قرارك." |
| الـPR نزل | "PR <رقم> مفتوح، guard شغّال" |
| Guard أخضر، أحاول merge | (لا تكتب — افعل) |
| Merged | "✅ #<رقم> اندمج. التالي: …" |
| نهاية الـsession | تقرير شامل في نهاية الـturn |

### 5.2 ممنوع

❌ التعبير عن "أنا متأكد" بدون فعل التحقق.
❌ ادعاء الاكتمال قبل ما merge فعلاً.
❌ كتابة فقرة طويلة لشرح ما هو واضح من الـcommit message.
❌ commentary على ما يفعله الـtool — اقفز للنتائج.

### 5.3 متى يستخدم العربي ومتى الإنجليزي

| السياق | اللغة |
|---|---|
| رد المالك في الـchat | عربي (المالك يكتب عربي) |
| commit message | إنجليزي (technical, searchable) |
| PR title | إنجليزي |
| PR body | يجوز عربي للمالك / إنجليزي للتقني |
| code comments | إنجليزي |
| user-facing labels (Arabic UI) | عربي |
| error messages للمستخدم النهائي | عربي |
| smoke test descriptions | إنجليزي |

## §6 — توقف الوكيل (Stop Conditions)

الوكيل **يتوقف فوراً ولا يكمل** إذا:

1. **CI أحمر بسبب غير معروف** بعد محاولتين.
2. **Conflict semantic** على ملف لا يفهم intent تغييره.
3. **Owner Decision** مطلوب وليس متوفراً في الـIntake.
4. **Slice متوقف على slice آخر لم ينزل بعد** (e.g. SLICE-A لا ينفع بدون SLICE-B).
5. **رأى نمطاً يقترح refactor عريض** — يقترح ولا ينفذ.
6. **اكتشف bug قائم خارج scope الـIntake** — يفتح Issue منفصل ويستمر.
7. **الـsession context قارب الـlimit** — يكتب handoff للوكيل التالي.

في كل حالة: **رسالة قصيرة، واضحة، تنتظر إشارة المالك.**

## §7 — تسليم بين الجلسات (Inter-Session Handoff)

عندما تنتهي جلسة وتبدأ أخرى، الوكيل السابق يكتب في PR description أو comment:

```yaml
# Handoff for next session
state:
  active_branch: claude/<branch>
  open_prs:
    - #<num> — <slice> — status: <guard | merged | conflict>
  in_progress:
    - <slice-id> — current step: <step>
  blocked_on:
    - <slice-id> — reason: <reason>
  permanent_rules_active_now:
    - <any rule that applies specifically>

next_action:
  - <one specific bullet>
```

الوكيل التالي يقرأ هذا قبل ما يبدأ.

## §8 — أنماط الـsession التي عمل بها هذا المستودع

### 8.1 Pattern A: "Autonomous Sweep"

المالك يعطي directive عامة (مثلاً "كلها بالتسلسل"). الوكيل ينفذ كل slice ضمن العقيدة، يتوقف عند:
- borderline slice يحتاج إذن explicit
- hard-pause item يحتاج Owner Decision

### 8.2 Pattern B: "Targeted Slice"

المالك يحدد slice واحد بـIntake كامل. الوكيل ينفذها فقط ويتوقف.

### 8.3 Pattern C: "Treadmill Babysit"

PR مفتوح، الوكيل يتابعه عبر cycles من guard → update → guard → merge.

### 8.4 Pattern D: "Doctrine / Governance"

يكتب أو يحدّث ملفات `docs/governance/*` — مثل هذا الملف نفسه.

## §9 — الـmetrics (تقاس بها جودة الوكيل)

| المقياس | الحد المقبول | حالياً |
|---|---|---|
| نسبة الـPRs المدموجة بدون treadmill > 3 cycles | > 60% | ~70% |
| نسبة الـsmokes الفاشلة في الـCI من أول مرة | < 10% | ~7% |
| نسبة الـoverrides لـbranch protection | 0% | 0% |
| نسبة الـsessions التي توقفت بسبب لا تخص الـslice | < 5% | ~3% |
| متوسط حجم الـPR (سطور) | 100-400 | ~250 |

كل وكيل جديد يُقيَّم بناءً على هذه المقاييس بعد أول 5 PRs.

## §10 — قائمة المراجعة قبل تسليم slice مكتمل

- [ ] الـcode change موافق للـIntake.
- [ ] الـmigration (إذا فيه) additive + nullable + `@rollback`.
- [ ] الـschema dump محدّث.
- [ ] الـsmoke يغطي §A/§B/§C/§D.
- [ ] vitest + tsc — أخضر.
- [ ] PR description يحوي 5 أقسام معيارية.
- [ ] Guard أخضر.
- [ ] Merged عبر treadmill (لا override).
- [ ] رسالة المالك تذكر الـPR + الـnext step.

---

## §11 — السلوكيات المرفوضة (Red Flags)

أي من هذه السلوكيات تستحق توقف فوري ومراجعة:

1. **Slice ينمو خلال التنفيذ** (من 100 سطر متوقع إلى 500).
2. **الوكيل يقرر بنفسه owner decision** (يفعل default flip بحجة "واضح أنه المقصود").
3. **الوكيل يصلح bug خارج scope** (نقل لـIssue منفصل).
4. **الوكيل يكتب "تم" قبل ما merge** فعلاً.
5. **الوكيل يدّعي اختبار يدوي في المتصفح** بدون دليل.
6. **الوكيل يحذف smoke موجود** بدون مبرر صريح في الـcommit message.
7. **الوكيل يضيف Comment يبرر سلوك** لازم يكون صريحاً في الكود.
