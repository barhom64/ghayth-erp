# غيث — دليل الأدوار والصلاحيات

> **مرجع تابع لـ`غيث_العقيدة_الحاكمة.md` §6.5 + §6.6.**
> هذا الملف يفصّل نظام RBAC (Role-Based Access Control) في النظام: featureCatalog، actions القياسية، أنماط التحقق، أنماط الـtemplates التي تُلصق الصلاحيات على kies.
> الهدف: **كل endpoint يصير محمياً بنفس الـpattern؛ كل دور قابل للتعقّب من قائمة واحدة؛ صفر `if (user.role === 'admin')` في الكود.**

---

## §0 — المبادئ الـ6 للـRBAC

1. **كل route محمي بـmiddleware `authorize()`** — لا route كتابة بدون.
2. **كل صلاحية تُعرَّف بـ`{feature, action}`** — لا role strings ثابتة.
3. **الـactions قياسية ومحدودة** — `view | list | create | update | delete | approve | read | export`.
4. **الـfeatures مسجّلة في catalog مركزي** — `lib/rbac/featureCatalog.ts`.
5. **الأدوار = تجميعات صلاحيات** — لا تُرسَّخ في الكود؛ تُدار من DB.
6. **الفرع جزء من الـscope** — كل تحقق يأخذ `branchId` بالحسبان.

## §1 — featureCatalog: الفهرس المركزي

### 1.1 الـshape

كل feature مسجّلة في `lib/rbac/featureCatalog.ts` بهذا الـshape:

```ts
interface FeatureSpec {
  key: string;                    // "umrah" | "finance" | "hr" | "fleet" | "properties" | ...
  arabicLabel: string;             // "العمرة"
  actions: FeatureAction[];        // قائمة الـactions المتاحة
  sensitive?: boolean;             // لو true: يحتاج logSensitiveAccess
  scope?: "global" | "branch";     // لو "branch": يحتاج branchId في scope
  category?: "operational" | "financial" | "hr" | "admin" | "report";
}

interface FeatureAction {
  key: ActionKey;                  // "view" | "list" | "create" | ...
  arabicLabel: string;             // "إنشاء"
  requiresApproval?: boolean;
  minLevel?: "manager" | "gm" | "owner";  // أدنى مستوى مسموح
}
```

### 1.2 الـactions القياسية

| Action | المعنى | متى يُستخدم |
|---|---|---|
| `view` | عرض سجل واحد | `GET /:id` |
| `list` | عرض قائمة | `GET /` بأي filters |
| `read` | قراءة شاملة (تقرير/درل-داون) | `GET /reports/...` |
| `create` | إنشاء سجل جديد | `POST /` |
| `update` | تحديث سجل | `PUT /:id` / `PATCH /:id` |
| `delete` | حذف (soft delete) | `DELETE /:id` |
| `approve` | اعتماد سجل (workflow) | `POST /:id/approve` |
| `export` | تصدير CSV/PDF | `GET /export.csv` |

**ممنوع actions خارج هذه القائمة.** لو احتجت "publish" أو "archive" — أضفها في الـcatalog بـPR منفصل + ratification.

### 1.3 مثال على feature

```ts
{
  key: "umrah",
  arabicLabel: "العمرة",
  category: "operational",
  scope: "branch",
  actions: [
    { key: "view", arabicLabel: "عرض" },
    { key: "list", arabicLabel: "قائمة" },
    { key: "create", arabicLabel: "إنشاء", minLevel: "manager" },
    { key: "update", arabicLabel: "تحديث", minLevel: "manager" },
    { key: "delete", arabicLabel: "حذف", minLevel: "gm" },
    { key: "approve", arabicLabel: "اعتماد", requiresApproval: true, minLevel: "gm" },
    { key: "export", arabicLabel: "تصدير" },
  ],
}
```

## §2 — الـmiddleware: `authorize()`

### 2.1 الـshape

```ts
authorize(spec: { feature: string; action: ActionKey }): Express.Handler
```

### 2.2 ما يفعله بالتفصيل

1. يستخرج `req.scope` (موجود من session middleware).
2. لو `scope.userId` غير موجود → 401.
3. يستدعي `checkAccess(userId, feature, action, companyId)` async.
4. لو الـfeature.scope === "branch" + scope.branchId غير محدد → 422 "اختر فرعاً".
5. لو الـfeature.sensitive === true → يحدث `logSensitiveAccess(...)`.
6. لو passed → `next()`.
7. لو failed → 403 ورسالة عربية.

### 2.3 الاستخدام الإلزامي

```ts
// ✅ صحيح
router.get("/umrah/pilgrims",
  authorize({ feature: "umrah", action: "list" }),
  async (req, res) => { ... }
);

router.post("/finance/manual-journals",
  authorize({ feature: "finance", action: "create" }),
  async (req, res) => { ... }
);

router.get("/reports/umrah/commissions-summary/export",
  authorize({ feature: "umrah", action: "export" }),
  async (req, res) => { ... }
);
```

### 2.4 الممنوع

```ts
// ❌ ممنوع: role check يدوي
if (req.user.role === "admin") { ... }

// ❌ ممنوع: route بدون authorize
router.get("/secret-data", async (req, res) => { ... });

// ❌ ممنوع: feature غير مسجّلة في catalog
authorize({ feature: "random_feature", action: "list" })

// ❌ ممنوع: action خارج القائمة القياسية
authorize({ feature: "umrah", action: "publish" })
```

## §3 — `scopeCan()` + `checkAccess()`

### 3.1 الفرق

| الـfunction | Synchronous? | الاستخدام |
|---|---|---|
| `scopeCan(scope, feature, action)` | Sync | داخل route handler — التحقق السريع من الـscope الحالي |
| `checkAccess(userId, feature, action, companyId)` | Async | داخل background jobs، crons، event listeners |
| `authorize({...})` | Async middleware | على route definition |

### 3.2 متى يُستخدم `scopeCan`

```ts
router.post("/umrah/pilgrims", authorize({ feature: "umrah", action: "create" }),
  async (req, res) => {
    const scope = req.scope!;

    // تحقق إضافي conditional
    if (body.status === "violated" && !scopeCan(scope, "umrah", "approve")) {
      throw new ForbiddenError("لا تملك صلاحية اعتماد حالة المخالفة.");
    }

    // ...
  }
);
```

### 3.3 متى يُستخدم `checkAccess`

```ts
// في cron handler
async function umrahVisaExpiryAlerts() {
  // ...
  for (const company of companies) {
    const adminAssignment = await getManagerAssignmentId(company.id);
    if (!adminAssignment) continue;

    // التحقق إن المدير له صلاحية استلام الإشعارات
    const canReceive = await checkAccess(
      adminAssignment.userId,
      "umrah",
      "view",
      company.id,
    );
    if (!canReceive) continue;

    // ...
  }
}
```

## §4 — جدول الأدوار في DB

### 4.1 الجداول الأساسية

```
users
  └─ employee_assignments       ← الـjoin user → role + branch + company
       └─ role: string             "manager" | "gm" | "owner" | "accountant" | ...
       └─ status: "active" | "inactive"
       └─ branchId, companyId

role_permissions               ← أي رول له أي صلاحية
  - role: string
  - feature: string
  - action: ActionKey
  - companyId (لو tenant-specific)

user_permissions_override      ← override خاص بمستخدم محدد
  - userId, feature, action, granted: boolean
```

### 4.2 منطق `checkAccess` المختصر

```
1. اقرأ user.assignments WHERE status='active' AND companyId=$companyId
2. لكل assignment:
   a. اقرأ role_permissions WHERE role=assignment.role AND feature=$f AND action=$a
   b. لو موجود → granted = true
3. اقرأ user_permissions_override WHERE userId=$u AND feature=$f AND action=$a
   - لو granted=true → الـuser له صلاحية explicit
   - لو granted=false → الـuser محروم explicitly
4. الـresult النهائي:
   - explicit override يَغلب role grant
   - role grant يَغلب الـdefault deny
   - الافتراضي: deny
```

## §5 — الأدوار المعرَّفة في النظام

### 5.1 الأدوار القياسية (system roles)

| Role | المستوى | الصلاحيات الافتراضية |
|---|---|---|
| `owner` | الأعلى | كل الـactions على كل الـfeatures (الـtenant-wide) |
| `general_manager` (gm) | عالٍ | كل الـactions ما عدا الـdelete على الـfinancial features |
| `branch_manager` (manager) | متوسط | view/list/create/update على الـfeatures التشغيلية لفرعه |
| `accountant` | متوسط | full access على finance/hr-payroll؛ view على غيرها |
| `hr_manager` | متوسط | full access على hr؛ list على غيرها |
| `umrah_operator` | تشغيلي | view/list/create/update على umrah فقط |
| `fleet_operator` | تشغيلي | view/list/create/update على fleet فقط |
| `viewer` | قراءة | view/list فقط على كل شيء |

### 5.2 الأدوار المتخصصة

| Role | المسؤولية |
|---|---|
| `audit_officer` | read-only على كل شيء + access على audit_log |
| `procurement_officer` | full access على purchase_orders + goods_receipts |
| `treasury_officer` | full access على payments + treasury_accounts |
| `legal_officer` | full access على legal cases + contracts |
| `agent_relations` | full access على umrah agents + sub-agents |

**كل دور جديد** يحتاج:
1. PR منفصل يضيفه في `lib/rbac/roles.ts`.
2. seed migration يضع الـrole_permissions الافتراضية.
3. ratification من المالك (Owner Decision).
4. سطر في هذا الدليل.

## §6 — SoD (Separation of Duties)

### 6.1 المبدأ

لا يُسمح لشخص واحد بأن يحمل دوران متعارضان في نفس الـcompany.

### 6.2 القائمة الحالية (`lib/rbac/sodRules.ts`)

| Conflict | السبب |
|---|---|
| `procurement_officer` + `treasury_officer` | منع شراء + دفع لنفس الـPO |
| `accountant` + `audit_officer` | منع تدقيق سجلات الـaccountant نفسه |
| `hr_manager` + `treasury_officer` | منع صرف الرواتب على نفس الـHR manager |
| `agent_relations` + `accountant` | منع إنشاء فاتورة + اعتمادها لنفس الوكيل |
| `branch_manager` (فرع X) + `branch_manager` (فرع Y) | معتمد فقط إذا approved by gm |

### 6.3 الـenforcement

```ts
// في assignment creation
async function assignRole(userId, role, companyId, branchId) {
  const existingRoles = await rawQuery(`
    SELECT role FROM employee_assignments
    WHERE "userId" = $1 AND "companyId" = $2 AND status = 'active'
  `, [userId, companyId]);

  for (const existing of existingRoles) {
    const conflict = checkSoDConflict(role, existing.role);
    if (conflict) {
      throw new ValidationError(
        `لا يمكن منح هذا الدور — يتعارض مع ${conflict.existingRoleLabel}`,
        { fix: `إلغاء الدور المتعارض أولاً، أو طلب استثناء من المدير العام.` },
      );
    }
  }

  // ...
}
```

## §7 — الفرع + الصلاحيات

### 7.1 الـbranch scope

كل assignment يربط user → role → company → **branchId**.

- لو الـrole "owner" → branchId يكون NULL (وصول على كل الفروع).
- لو الـrole "gm" → branchId NULL غالباً (وصول كامل).
- لو الـrole "manager" / "operator" → branchId محدد.

### 7.2 منطق التحقق

```ts
function scopeCan(scope, feature, action) {
  // owner / gm = unlimited branch access
  if (scope.role === "owner" || scope.role === "general_manager") return true;

  // غير ذلك: branch محدد
  if (!scope.branchId) return false; // لا فرع → لا صلاحية

  // التحقق العادي
  return scope.permissions.has(`${feature}:${action}`);
}
```

### 7.3 الـUI

- topbar يعرض الـbranch selector إذا للـuser فروع متعددة.
- لو فرع واحد فقط → يصير محدد ضمنياً.
- لو owner/gm → "كل الفروع" + الـoption للتركيز على فرع واحد.

## §8 — Sensitive Access + Audit

### 8.1 الـsensitive features

بعض الـfeatures مسجّلة كـ`sensitive: true` في الـcatalog:

- `finance.gl_export`
- `hr.salary_review`
- `umrah.pilgrim_pii_export`
- `legal.case_documents`
- `treasury.payment_export`

### 8.2 ما يحدث

عند الـauthorize يُسجّل تلقائياً:

```ts
await logSensitiveAccess({
  userId: scope.userId,
  companyId: scope.companyId,
  feature, action,
  endpoint: req.path,
  ipAddress: req.ip,
  userAgent: req.headers["user-agent"],
});
```

### 8.3 التقارير

`GET /admin/sensitive-access-log` يعرض كل access الحساس بفلاتر:
- الـuser
- الـfeature
- الـtime window
- الـip

## §9 — الـtemplate لـroute كتابة جديد

```ts
import { authorize } from "@/lib/rbac/authorize";
import { handleRouteError, ValidationError, ForbiddenError } from "@/lib/errorHandler";
import { zodParse } from "@/lib/zodParse";
import { rawExecute, withTransaction } from "@/lib/rawdb";
import { issueNumber } from "@/lib/numberingService";
import { createAuditLog, emitEvent, createGuardedJournalEntry } from "@/lib/businessHelpers";

const createSchema = z.object({
  documentDate: z.string().date(),
  clientId: z.number().int().positive(),
  lines: z.array(lineSchema).min(1),
  notes: z.string().optional(),
});

router.post(
  "/umrah/sales-invoices",
  authorize({ feature: "umrah", action: "create" }),  // ← step 1: RBAC gate
  async (req, res) => {
    try {
      const scope = req.scope!;
      if (!scope.branchId) {
        throw new ValidationError("اختر فرعاً من رأس الصفحة قبل المتابعة.");
      }

      const body = zodParse(createSchema.safeParse(req.body));

      // ... domain logic ...

      const { number } = await issueNumber({
        moduleKey: "umrah", entityKey: "sales_invoice",
        companyId: scope.companyId, branchId: scope.branchId,
      });

      const result = await withTransaction(async (client) => {
        const { insertId } = await rawExecute(/* INSERT */, [...]);
        // optional: createGuardedJournalEntry
        return { id: insertId };
      });

      await createAuditLog({
        companyId: scope.companyId, userId: scope.userId,
        action: "create", entity: "umrah_sales_invoices", entityId: result.id,
        after: { number, ...body },
      });

      await emitEvent({
        companyId: scope.companyId, userId: scope.userId,
        action: "umrah.sales_invoice.created",
        entity: "umrah_sales_invoices", entityId: result.id,
      });

      res.json({ id: result.id, invoiceNumber: number });
    } catch (err) {
      handleRouteError(err, res, "POST /umrah/sales-invoices");
    }
  },
);
```

## §10 — الـUI ↔ الـRBAC binding

### 10.1 إخفاء/إظهار الـbuttons

```tsx
import { GuardedButton } from "@/components/shared/permission-gate";

<GuardedButton
  feature="umrah"
  action="create"
  onClick={() => openCreateDialog()}
>
  معتمر جديد
</GuardedButton>
```

### 10.2 إخفاء/إظهار الـmenu items

```tsx
import { GuardedMenuItem } from "@/components/shared/permission-gate";

<GuardedMenuItem feature="umrah" action="approve">
  اعتماد الفاتورة
</GuardedMenuItem>
```

### 10.3 المبدأ

- **server هو المرجع الوحيد للـRBAC.** الـUI يخفي/يُظهر للراحة فقط.
- لو الـUI أخفى لكن الـuser استدعى endpoint مباشرة → الـserver يرفض بـ403.
- لا تثق في الـ`req.headers["x-user-role"]` — استخدم session فقط.
