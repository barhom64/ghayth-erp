# ٠٧ — الاعتماد الآلي وسلاسل الموافقات (مكثّف)

> القاعدة: لا اعتماد يدوي مبعثر. كل قرار يمرّ عبر **سلسلة موافقات معرّفة** بفصل مهام مُنفَّذ.

## ١. آلية السلسلة
- التعريف في `approval_chains` / `approval_chain_steps` (نوع السلسلة + مراحل + الدور المطلوب لكل مرحلة).
- البدء: `initiateApprovalChain` / `submitWorkflow`؛ التقدّم: `processApprovalStep` (راجع `businessHelpers`/`workflowEngine`).
- لكل مرحلة **دور مطلوب** يُطابَق بـ`roleMatchesStage` (مطابقة موقع — تبقى فحص دور، راجع `06_` §٣).

## ٢. فعل الاعتماد (إلزامي)
- مسار الموافقة دائمًا `authorize({feature, action:"approve"})`، والرفض `action:"reject"` — **لا `"update"` عام** (وإلا صار أي مالك تعديل معتمِدًا، وفقد `approvableActions` معناه). هذا تثبيت قرار HR-REV-1 #2.
- فحص داخل المعالِج (عند اللزوم) عبر `scopeCan(scope, feature, "approve")`.

## ٣. فصل المهام (SoD) — maker ≠ checker
- `sodEnforcement.ts` + `selfApprovalCreators.ts` يمنعان أن يعتمد منشئ العملية عمليته.
- العمليات الحرجة (رواتب/نهاية خدمة/قيود) تتطلّب سلسلة + قاعدة SoD مزروعة (مثل `hr_payroll_calculate_approve`).

## ٤. الأتمتة (الاعتماد الآلي حيث يصحّ)
- المراحل التي لا تتطلّب حكمًا بشريًّا (تحقّقات اكتمال، حدود مبالغ، فترات مفتوحة) تُؤتمَت: النظام يتحقّق ويُقدّم/يحجب آليًّا.
- حدود الاعتماد (`approval_limits`) تُقيّم آليًّا للأفعال ذات المبالغ؛ تجاوز الحدّ يصعّد للمرحلة الأعلى تلقائيًّا.
- التصعيد الزمني آليّ (مثل: HR بعد ٨ ساعات، GM بعد ٢٤) عبر `registerObligation`/المجدوِل.

## ٥. الأثر والتدقيق
كل قرار اعتماد/رفض: يُطلق حدثًا (`emitEvent`) باسم من الكتالوج، ويكتب `createAuditLog` بـ`resolved_scope`، ويُحدِّث الحالة عبر آلة الحالة (`lifecycleEngine`)، ويُولّد الأثر المالي (إن وُجد) عبر `accounting-engine`.

## ٦. ممنوعات
- اعتماد بـ`action:"update"`.
- اعتماد منشئ لعمليته (كسر SoD).
- قرار اعتماد بلا حدث + تدقيق + تحديث حالة.
- سلسلة موافقات مكتوبة نصًّا في المسار بدل التعريف المركزي.
