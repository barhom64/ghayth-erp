# ٠١ — نمذجة الطلبات والتوحيد

> الهدف: نموذج إدخال **واحد موحّد** لكل عائلة عمليات، ذكيّ وتدريجي، لا نماذج مكرّرة ولا حقول ميتة.

## ١. القاعدة
لكل عائلة (صرف/قبض/طلب/مستند) **مكوّن نموذج موحّد** يقود المستخدم بالخطوات (راجع `08_...` للفرز الذكي). يُمنع إنشاء نموذج جديد إن وُجد نموذج موحّد يغطّي العائلة — يُوسَّع الموحّد بدل التكرار.

## ٢. مبادئ النموذج الموحّد
1. **تدريجي (Progressive Disclosure)**: لا تُعرض الحقول دفعة واحدة؛ كل حقل يظهر بعد ما يُحدِّد ما قبله.
2. **مُلكية الحقل**: لكل حقل **مالك واحد** (مَن يعبّئه) و**سبب** و**أثر**. لا حقل بلا مالك (راجع `HR_INPUT_FORMS_FIELD_OWNERSHIP` و`05_...`).
3. **السياق آليّ**: الشركة/الفرع/المستخدم/تاريخ الإنشاء لا تُعرض كحقول إدخال — تُحقَن (راجع `10_...`).
4. **التحقّق موحّد**: استخدام `zod` + رسائل عربية موحّدة عبر `errorHandler` (`ValidationError`).
5. **الترقيم موحّد**: رقم المستند من `numberingService` بعد الحفظ، لا حقل يدوي (راجع `03_...`).
6. **التقويم موحّد**: تاريخ المستند عبر `date-utils`/`currentDateInTz`، مفصول عن تاريخ الإنشاء (راجع `04_...`).

## ٣. بنية النموذج الموحّد (الطبقات)
1. **طبيعة الحركة** → 2. **الفئة الذكية** → 3. **نوع العملية** → 4. **استحضار بيانات الكيان** (إن مُسجَّل) → 5. **الحقول الخاصة بالعملية** → 6. **التوجيه المحاسبي الآلي** (للعمليات المالية) → 7. **المرفقات** (راجع `05_...`) → 8. **الاعتماد** (سلسلة موافقات، راجع `07_...`).

## ٤. التوحيد عبر الوحدات
نفس البنية تنطبق على كل المسارات (مالية/مستودعات/عقارات/أسطول/عمرة/موارد بشرية). الاختلاف فقط في **الفئات وأنواع العمليات وقوالب التوجيه**، لا في آلية النموذج. أي وحدة تبني نموذجًا خاصًّا خارج هذه البنية = مخالفة.

## ٥. ممنوعات
- نموذج يطلب من المستخدم اختيار الحساب المدين/الدائن يدويًّا (يأتي من `accounting_mappings`).
- نموذج يطلب الفرع/الشركة/المستخدم/تاريخ الإنشاء.
- حقل يظهر دائمًا بلا شرط، أو حقل بلا مالك/سبب/أثر.
- نسخة ثانية من نموذج موجود.
