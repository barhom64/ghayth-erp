## ٢. الحماية أولًا — حاجز الدخول الإلزامي

**قبل أي إدخال، النظام يتحقّق:**

| الشرط | المصدر | عند الفشل |
|------|--------|----------|
| فرع محدّد (`branchId`) | `req.scope` | خطأ 403 — لا كتابة |
| شركة محدّدة (`companyId`) | `req.scope` | خطأ 403 — لا كتابة |
| دور مخوَّل له الفعل | `authorize({feature, action})` | خطأ 403 — لا كتابة |
| مستخدم موثّق (`userId`) | `req.scope` | خطأ 401 |
| فرع نشط (غير موقوف) | `allowedBranches` في `buildScope` | خطأ 403 |

**القاعدة الحديدية:**
- `companyId` / `branchId` / `userId` / `createdAt` ← تأتي من `req.scope` و`NOW()` **فقط**.
- أي قيمة لهذه الحقول في `req.body` **تُتجاهَل صراحةً** (لا تُقرأ، لا تُكتب).
- `buildScopedWhere(scope, ...)` على كل قراءة — بلا استثناء.
- `authorize({feature, action})` على كل مسار كتابة — بلا استثناء.

## ٣. تاريخ الإنشاء ≠ تاريخ المستند (تفريق إلزامي، لا يقبل خلطًا)

| النوع | الاسم في الكود | المصدر | قابل للتعديل؟ | يُعرض في UI/طباعة؟ |
|------|--------------|--------|------------|-------------------|
| تاريخ الإنشاء | `createdAt` | `DEFAULT NOW()` — آليًّا | **لا** أبدًا | نعم، بتسمية «تاريخ الإدخال» |
| تاريخ المستند / العملية | `documentDate` / `invoiceDate` / `entryDate` … | إدخال المستخدم | نعم (ضمن فترة مفتوحة) | نعم، بتسمية «تاريخ المستند» |
| تاريخ التحديث | `updatedAt` | `NOW()` عند كل تعديل — آليًّا | آليًّا | لا (داخلي) |

**تطبيق عملي:**
- أي نموذج إدخال يعرض حقل «التاريخ» → هو تاريخ المستند فقط.
- كل طباعة تُظهر **الاثنين معًا** بتسميتين مختلفتين.
- `createdAt` لا يُقبَل من المستخدم ولا يُعدَّل بعد الإنشاء.
- تاريخ المستند يُفحَص ضد `checkFinancialPeriodOpen` قبل الترحيل.

## ٤. الفرع والشركة والمستخدم — الحقن الآلي (لا إدخال يدوي)

```ts
// ✅ صحيح — كل حقن من req.scope
const { companyId, branchId, userId } = req.scope;
await db.insert(table).values({
  companyId,
  branchId,
  createdBy: userId,
  // createdAt → DEFAULT NOW() من قاعدة البيانات
});

// ❌ خطأ — قراءة من req.body
const { companyId } = req.body; // محظور تمامًا
```

**في كل INSERT:** `companyId` ← scope، `branchId` ← scope، `createdBy` ← scope.userId، `createdAt` ← NOW().

## ٥. نمذجة الطلبات — الفرز الذكي والكشف التدريجي

**تسلسل الإدخال الإلزامي (لكل الوحدات):**

```
١. طبيعة الحركة
   ┌──────────────────────────────┐
   │ صرف / قبض / تحويل / طلب ... │
   └──────────────────────────────┘
           ↓ اختيار
٢. الفئة الذكية (من كتالوج موحّد)
   ┌────────────────────────────────────────────────────────┐
   │ عميل · صيانة · عقار · مركبة · عمرة · موظف · إداري … │
   └────────────────────────────────────────────────────────┘
           ↓ اختيار الفئة يكشف
٣. نوع العملية (بناءً على الفئة)
   مثال «مركبة» ← شراء / صيانة / وقود / تأمين / مخالفة مرورية
   مثال «عميل» ← فاتورة / مقبوضة / رصيد / خصم
   مثال «موظف» ← سلفة / راتب / نهاية خدمة / خصم
   مثال «عقار» ← إيجار / بيع / صيانة / وديعة / عمولة إدارة
           ↓ اختيار النوع يكشف
٤. استحضار بيانات الكيان (إن كان مُسجَّلًا)
   مثال: مركبة مضافة → يستحضر النظام: نوع الوقود، اللوحة،
          مركز التكلفة، الفرع، السعة — المستخدم لا يُعيد الإدخال
           ↓
٥. حقول العملية الخاصة (فقط ما يلزم هذا النوع)
           ↓
٦. التوجيه المحاسبي آليًّا
   من: accounting_mappings (نوع عملية + فئة + فرع → مدين/دائن)
   لا اختيار يدوي للحساب — ممنوع
           ↓
٧. المرفقات (مكوّن موحّد)
           ↓
٨. سلسلة الاعتماد
```

**قواعد الفرز الذكي:**
- لا حقل يظهر قبل تحديد شرطه.
- الكيانات المُسجَّلة = اختيار + استحضار، لا إعادة إدخال.
- كتالوج الفئات مركزيّ (لا نصوص متناثرة في كل نموذج).
- أي نموذج يخرج عن هذا التسلسل = مخالفة.

## ٦. نظام الترقيم الموحّد

```ts
// ✅ الطريقة الوحيدة المسموحة
import { issueNumber } from "../lib/numberingService.js";

// داخل withTransaction، بعد INSERT:
const docNumber = await issueNumber("INVOICE" /* أو نوع المستند */, scope.companyId);
```

**القواعد:**
- الترقيم دائمًا **بعد** الحفظ الناجح، داخل نفس المعاملة.
- لكل شركة تسلسل مستقل (عزل مستأجرين).
- البوادئ معرَّفة مركزيًّا — لا نص مكتوب يدويًّا في `routes/*`.
- الرقم لا يُقبَل من المستخدم ولا يُعدَّل بعد الإصدار.

**محظور تمامًا:**
- `MAX(id)+1` أو `Date.now()` كرقم مستند.
- ترقيم في الواجهة الأمامية.
- قبول رقم من `req.body`.

## ٧. نظام التقويم الموحّد والتواريخ

```ts
// ✅ خادم — «اليوم» في منطق الأعمال
import { currentDateInTz, todayISO } from "../lib/businessHelpers.js";
const today = currentDateInTz("Asia/Riyadh"); // دائمًا — لا new Date() خام

// ✅ أمام — عرض + هجري/ميلادي
import { formatDate, toHijri, fromHijri } from "@/lib/date-utils";
```

**القواعد:**
- «اليوم» في الخادم = `currentDateInTz("Asia/Riyadh")` دائمًا.
- `new Date()` خام في منطق الأعمال = مخالفة (يكسر الحضور/الإقفال عبر المناطق الزمنية).
- التخزين: دائمًا ISO ميلادي في قاعدة البيانات.
- الهجري: طبقة عرض فقط عبر `date-utils.ts` — لا تحويل يدوي مبعثر.
- تاريخ المستند يُفحَص ضد الفترة المحاسبية قبل الترحيل.

- طباعة بلا رقم مستند أو بخلط التاريخين.

## ٩. مكتبة الأدوار والصلاحيات — RBAC v2

### بوّابة قدرة (مَن يستطيع X):
```ts
// ✅ على المسار
router.post("/...", authorize({ feature: "hr.payroll", action: "approve" }), handler);

// ✅ داخل المعالِج (فحص إضافي)
if (!scopeCan(scope, "hr.payroll", "approve")) throw new ForbiddenError();
```

### بوّابة موقع/نطاق/سلطة (تبقى فحص دور — لا تُحوَّل):
```ts
// ✅ مثال: مدير مقابل ذاتي، مدير الفرع المستقبِل
if (scope.role !== "branch_manager") throw new ForbiddenError();
```

**القواعد:**
- بوّابة القدرة ← `scopeCan` / `authorize` (المنح مصدر وحيد).
- بوّابة الموقع/النطاق/السلطة ← فحص `scope.role` (ليس تكرار منح).
- لا مفتاح ميزة خارج `featureCatalog.ts`.
- لا `authorize({action:"update"})` لمسار اعتماد (استخدم `approve`/`reject`).
- ظهور الصفحات/الأزرار في الواجهة ← من `flatProjection` / `permission-match.ts`.
- أي دور جديد = منح كاملة في `rbac_role_grants` (دور بلا منح = خطأ).

## ١٠. الاعتماد الآلي وسلاسل الموافقات

**آلية إلزامية لكل عملية تتطلّب اعتمادًا:**

```
المرحلة ١: تقديم الطلب
  ← initiateApprovalChain / submitWorkflow
  ← يُطلق حدثًا من eventCatalog
  ← يكتب createAuditLog

المرحلة ٢: فحص آلي (بلا بشر)
  ← اكتمال الحقول الإلزامية
  ← حدود المبالغ (approval_limits)
  ← فترة محاسبية مفتوحة (للعمليات المالية)
  ← فصل المهام (sodEnforcement — maker ≠ checker)
  إذا تجاوز الحدّ → تصعيد آلي للمرحلة الأعلى

المرحلة ٣: موافقة بشرية (للعمليات الحرجة)
  ← authorize({feature, action:"approve"})
  ← لا يعتمد المنشئ عمليته (selfApprovalCreators)

المرحلة ٤: الأثر
  ← تحديث الحالة عبر lifecycleEngine
  ← emitEvent(catalog_event_name, ...)
  ← createAuditLog مع resolved_scope
  ← ترحيل قيد عبر gl/posting (إن كانت مالية)
  ← توليد رقم المستند (إن لم يُولَّد بعد)
```

**محظور:**
- اعتماد بـ`action:"update"`.
- منشئ العملية يعتمدها (كسر SoD).
- قرار اعتماد بلا حدث + تدقيق + تحديث حالة.
- سلسلة موافقات مكتوبة نصًّا في المسار.

## ١٢. التفعيل السريع الموزّع (إضافة موظف/كيان)

**لا نموذج طويل — قالب الوظيفة يقرّر:**

```
١. المُدخِل المختصّ يُدخِل فقط:
   الاسم + رقم الهوية + الفرع + المسمّى الوظيفي

٢. النظام يقرأ قالب الوظيفة (job activation profile) ويُولّد:
   - هل تلزم عهدة؟ وأي بنود؟
   - هل تلزم مركبة/رخصة؟
   - الأدوار/المنح الافتراضية
   - الوثائق المطلوبة (إقامة/عقد/تأمين)
   - متطلّبات الرواتب/البنك

٣. توزيع المهام آليًّا على المالكين:
   الموظف نفسه → بياناته الشخصية/البنكية
   مدير القسم   → التبعية والمهام
   الرواتب      → الراتب والبدلات
   المستودع     → العهدة والأصناف
   الأسطول      → المركبة والرخصة
   مسؤول الصلاحيات → الأدوار والمنح

٤. كل مالك يرى حصّته فقط
   الموظف = «قيد التفعيل» حتى تُغلَق الحصص الإلزامية
```

**محظور:**
- نموذج يطلب كل شيء دفعةً من مُدخِل واحد.
- خيارات يدوية «عهدة/مركبة إن وجدت» (القالب يقرّر، لا سؤال).
- إظهار خيارات الأدوار/المسمّيات لموظف غير مُفعَّل.

## ١٤. انضباط كل PR (بلا استثناء)

1. **صغير ومركّز**: PR واحد = مهمة واحدة.
2. **typecheck + guard أخضر** قبل الدمج.
3. **وصف PR مطابق للكود حرفيًّا** (لا وصف تقريبي).
4. **لا خلط وحدات**: HR ≠ Finance ≠ Properties في PR واحد.
5. **لا حذف صلاحيات/سلوك تشغيلي** بلا توثيق صريح.
6. **بوّابة القبول التشغيلي** تُثبَت برحلة حيّة على DB حيّ — لا ادّعاء نظري.
7. **ممنوع دمج PR قبل إغلاق السابق**.
