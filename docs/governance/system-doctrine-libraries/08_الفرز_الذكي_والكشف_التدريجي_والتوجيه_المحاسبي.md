# ٠٨ — الفرز الذكي والكشف التدريجي والتوجيه المحاسبي

> عقيدة الإدخال: لا نموذج طويل غبي. النظام يقود المستخدم خطوة-خطوة، ويستحضر البيانات، ويوجّه القيد آليًّا.

## ١. تسلسل الإدخال (إلزامي)
1. **طبيعة الحركة**: صرف / قبض / تحويل / طلب / مستند.
2. **الفئة الذكية** (قائمة موحّدة): عميل · صيانة · عقار · مركبة · عمرة · موظف · إداري · تشغيلي … (تُغذّى من كتالوج الفئات، لا نصوص متناثرة).
3. **نوع العملية** يظهر **بناءً على الفئة**. مثال «مركبة» → شراء مركبة / صيانة مركبة / وقود مركبة / تأمين / مخالفة مرورية.
4. **استحضار الكيان**: إن كانت المركبة **مُسجَّلة**، يجلب النظام بياناتها تدريجيًّا (نوع الوقود، السعة، اللوحة، مركز التكلفة، الفرع) — المستخدم لا يعيد إدخالها.
5. **حقول العملية فقط**: ما يلزم نوع العملية المختار، لا أكثر.
6. **التوجيه المحاسبي آليًّا**.

## ٢. التوجيه المحاسبي الآلي
- المصدر: جدول `accounting_mappings` (نوع العملية → حساب مدين/دائن)، يتحقّق منه `validateAccountingMapping` في `accounting-engine` **قبل** الاعتماد.
- المستخدم **لا يختار** الحساب المدين/الدائن — يأتي من القالب حسب (الفئة + نوع العملية + الفرع/النشاط).
- إن كان التوجيه ناقصًا → يُمنع الاعتماد برسالة واضحة (لا قيد بحساب مجهول).
- الترحيل عبر `gl/posting` بعد الاعتماد، ضمن فترة مفتوحة (`checkFinancialPeriodOpen`).

## ٣. الكشف التدريجي (Progressive Disclosure)
كل خطوة تكشف ما بعدها فقط؛ لا يُعرض حقل قبل تحديد شرطه. الكيانات المُسجَّلة تُختصر إلى **اختيار + استحضار**، لا إعادة إدخال.

## ٤. التوحيد عبر الوحدات
نفس التسلسل لكل الوحدات؛ يتغيّر فقط: كتالوج الفئات، أنواع العمليات، وقوالب `accounting_mappings`. لا وحدة تبتكر تسلسلًا خاصًّا.

## ٥. ممنوعات
- اختيار يدوي للحساب المدين/الدائن.
- إعادة إدخال بيانات كيان مُسجَّل.
- عرض كل الحقول دفعة واحدة بلا فرز.
- اعتماد عملية مالية بلا توجيه محاسبي مكتمل.
