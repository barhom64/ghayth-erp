# قالب المهمة الفرعية للعمرة

> مرتبط بـ #2080 §«شكل كل مهمة فرعية للعمرة». **إلزامي** لكل مهمة في الملف 14 ولكل Issue يُفتح منها. مهمة ناقصة الحقول تُعاد لصاحبها.

```markdown
## [U-xx] العنوان

**الوصف:**
(فقرة واحدة: ما الذي سيتغير ولماذا الآن)

**الموقع داخل العمرة:**
(مسار الملف/الـroute/الجدول/المفتاح — محدداً لا عاماً)

**الوضع الحالي:**
(الواقع كما هو، بدليل: سطر كود، رقم migration، لقطة سلوك)

**المشكلة:**
(الفجوة بين الواقع والميثاق — مصنّفة على تصنيفات الملف 4)

**الأثر:**
(مَن يتضرر من بقاء الوضع: المشغّل/المحاسب/الوكيل/التقارير/الدفتر — وكيف)

**المطلوب:**
- (خطوات قابلة للتحقق، مرقّمة)

**الممنوع:**
- (ما لا يجوز لمسه في هذه المهمة — حدودها الصارمة)
- لا كتابة مالية خارج محرك المالية.
- لا نقل خارج عقد الخدمة #1902.
- لا نجاح بلا counters + audit + event + أثر مرئي.

**معيار القبول:**
- (شروط موضوعية قابلة للفحص — اختبار/استعلام/لقطة شاشة)
- guard أخضر بلا override.

**الأولوية:** P0 / P1 / P2 / P3
**الخطورة:** حرجة / عالية / متوسطة / منخفضة

**الارتباط:** عمرة / مالية / نقل / HR / وثائق / إشعارات / قوالب / صلاحيات / تقويم / تقارير
**هل ترتبط بـ #1870؟** نعم — §… / لا
**هل ترتبط بـ #2071؟** نعم — قاعدة… / لا
```

## قواعد استخدام القالب

1. **مهمة واحدة = PR واحد.** إن احتاجت المهمة أكثر من PR فهي مهمتان.
2. **«الوضع الحالي» بدليل** — لا «يبدو أن». المرجع: ملف وسطر، أو نتيجة استعلام على بيئة `db:provision-agent`.
3. **«الممنوع» ليس زينة** — هو حدود الـPR التي يراجَع عليها. الأسطر الثلاثة الثابتة (المالية/النقل/النجاح الوهمي) تُنسخ في كل مهمة.
4. **«معيار القبول» يُفحص آلياً ما أمكن** — اختبار تكامل يؤكد سطور القيد للمهام المالية (النمط المعمول به في سلسلة #2016/#2025/#2027)، واختبار smoke يثبّت العقد للمهام البنيوية.
5. **الترقيم:** U-01، U-02… تسلسلياً من الملف 14، ويبقى الرقم ثابتاً عند التحويل لـIssue (يُذكر في عنوانه).
