# خطة ضبط صلاحيات العمرة

> مرتبطة بـ #2080 (نطاق 20 + المثال 3) و#2071. القاعدة: **لا زر ولا إجراء يظهر لمستخدم لا يملك صلاحيته، وفصل الإنشاء عن التعديل عن الاعتماد عن الإلغاء عن الطباعة والتصدير.**

## 1. الوضع القائم (خط الأساس)

- الخادم: كل routes العمرة خلف `authorize({ feature: "umrah", action: ... })` — الأفعال المستخدمة حالياً تدور حول `view / list / create / update`.
- الواجهة: نمط `GuardedButton perm="umrah:update"` قائم (مثال حي: زر حفظ الإعدادات في `settings.tsx`).
- RBAC v2 يدعم سلّم أفعال أوسع (موجود في النظام: view, list, export, print, create, submit, approve, reject, reopen, update, delete, cancel, close, share).

**الفجوة المبدئية:** أفعال العمرة على الخادم أضيق من سلّم RBAC — إجراءات حسّاسة (اعتماد فاتورة، إلغاء، طباعة، تصدير، إقفال موسم) تمرّ غالباً تحت `update`/`create` العامة. هذا يطابق التصنيف 7 (صلاحية أوسع من لازمها).

## 2. مصفوفة الأفعال المستهدفة

| الإجراء في العمرة | الفعل المستهدف | لماذا فصله |
|--------------------|----------------|-------------|
| عرض/قوائم | `umrah:view` / `umrah:list` | قائم |
| إنشاء كيان (معتمر/مجموعة/وكيل/خطة عمولة) | `umrah:create` | قائم |
| تعديل بيانات | `umrah:update` | قائم |
| **اعتماد** (فاتورة بيع، احتساب عمولة، خطة عمولة) | `umrah:approve` | الاعتماد التزام مالي — لا يجوز لمن يدخل البيانات أن يعتمدها (فصل واجبات) |
| **رفض/إرجاع** | `umrah:reject` | قرار رقابي مستقل |
| **إلغاء** (فاتورة/مخالفة/حجز) | `umrah:cancel` | الإلغاء يعكس قيوداً — أخطر من التعديل |
| **إقفال** (موسم) | `umrah:close` | يقفل الكتابة على موسم كامل |
| **طباعة** | `umrah:print` | تسريب بيانات محتمل (كشوف معتمرين = بيانات شخصية مشفّرة الحقول) |
| **تصدير** | `umrah:export` | كذلك — التصدير CSV قائم على كشوف المعتمرين |
| **استيراد (تأكيد الدفعة)** | `umrah:create` + شرط إضافي للدفعات الكبيرة (سياسة `maxRowsPerFile`) | الاستيراد إنشاء جماعي |
| حذف | `umrah:delete` | soft-delete فقط، ولأضيق دائرة |

## 3. قواعد الضبط

1. **الواجهة تتبع الخادم لا العكس:** الحارس الحقيقي هو `authorize` على الـroute؛ `GuardedButton` يخفي الزر تحسيناً للتجربة، لا بديلاً عن حارس الخادم. م1 يفحص كل endpoint كتابة في `umrah.ts` + `umrah-entities.ts` ويطابق فعله مع المصفوفة أعلاه.
2. **لا ترقية صامتة:** تغيير فعل endpoint من `update` إلى `approve` تغيير كاسر لمن يملك update فقط — كل تغيير صلاحية يمر بمهمة مستقلة بقائمة المتأثرين، **ولا يدخل في PR الخطة هذا إطلاقاً** (قيد #2080: لا تغيير صلاحيات في PR الأول).
3. **الحقول الحساسة:** حقول المعتمر المشفّرة (`SENSITIVE_PILGRIM_FIELDS` + `logSensitiveAccess` قائمة في الخادم) — أي شاشة تعرضها تُسجَّل وصولاً؛ م1 يتحقق أن `maskFields` مطبَّق على كل قوائم المعتمرين.
4. **صلاحيات الإعدادات:** تعديل سياسات العمرة الـ11 (`/umrah/settings` + PoliciesTab) يبقى خلف `umrah:update` كحد أدنى، ويُدرس رفعه لـ`umrah:approve` لأن السياسات تغيّر اتجاه الترحيل المالي (VAT mode / commission via HR) — قرار مالك.

## 4. مخرجات م1 لهذا المحور

- جدول: كل endpoint كتابة → الفعل الحالي → الفعل المستهدف → المتأثرون.
- جدول: كل زر إجراء في الصفحات الـ44 → هل خلف حارس؟ → الصلاحية الحالية/المستهدفة.
- قائمة الإجراءات التي تظهر اليوم لمن لا يملكها (تصنيف 7) — أولوية P1.

**معيار قبول هذا المحور:** كل إجراء حسّاس (اعتماد/إلغاء/إقفال/طباعة/تصدير) له فعل مستقل على الخادم، وكل زر في الواجهة خلف حارس مطابق، وقائمة المتأثرين بكل تغيير معتمدة من المالك قبل التنفيذ.
