# خطة تنفيذ جرد وتنظيم وإصلاح مسار العمرة

> مرتبطة بـ #2080 (المهمة) · #1870 (ميثاق العمرة) · #2071 (الميثاق التشغيلي الموحد)

## 1. الهدف

تحويل مسار العمرة من «مجموعة شاشات ومحركات تراكمت عبر PRات» إلى **مسار قائد محكوم**: منظّم، غير متكرر، غير متعارض، مرتبط بالمحركات المركزية الصحيحة، واضح الصلاحيات والتقارير ودورة العمل، ومانع للنجاح الوهمي.

الهدف **ليس** إصلاح أخطاء ظاهرة فقط؛ بل تحكيم كامل: كل عنصر في العمرة له مبرر وجود، ومالك، وحدود خدمة، وأثر قابل للقياس.

## 2. النطاق (عمرة فقط)

يشمل الجرد العناصر الـ31 المذكورة في #2080 حصراً: صفحات/قوائم/تبويبات/أزرار العمرة، Routes تحت `/umrah/*`، محركات `lib/umrah*`, الاستيراد Excel/نسك، الكيانات (معتمر/وكيل/وكيل فرعي/مجموعة/موسم)، فواتير نسك وبيع، العمولات، المخالفات والتخلف، التقويم، التقارير الـ17، السياسات الـ11، الصلاحيات، القوالب والمطبوعات، الربط بالخوادم (مالية/نقل/HR/وثائق/إشعارات)، الصفوف اليتيمة، التكرارات والتعارضات، المنطق العامل خارج المحركات، وما يحتاج إعادة نمذجة.

**خارج النطاق:** أي وحدة أخرى في النظام. أي ملاحظة تمسّ وحدة خادمة تُسجَّل من زاوية «عقد الخدمة مع العمرة» فقط، ولا تتوسع لإصلاح الخادم نفسه.

### 3.2 قواعد عمل ملزمة أثناء كل المراحل

1. **Service Boundary Lock:** أي route/UI/helper يكتب قيداً مالياً دون المرور بمحرك المالية (`createGuardedJournalEntry` عبر engine رسمي) = خلل معماري جسيم يُسجَّل فوراً بتصنيف «خلل حرج».
2. **منع النجاح الوهمي:** أي عملية تدّعي النجاح بلا counters + audit + event + أثر مرئي تُسجَّل بتصنيف 15.
3. **لا اجتهاد شخصي:** كل توصية تذكر السبب، الأثر، ومعيار القبول. التوصية الذوقية تُرفض في المراجعة.
4. **PR صغير:** مهمة واحدة لكل PR. الـrefactor (مثل تقسيم `umrah-entities.ts`) لا يُخلط بميزة.
5. **الإبلاغ:** رقم كل PR يُبلَّغ للمالك للتدقيق.

### 3.3 أدوات الجرد

- **الواجهة:** `artifacts/ghayth-erp/src/pages/umrah/` + `navigation.registry.ts` + `umrah-tabs-nav.tsx` + `umrahRoutes.tsx` — مطابقة ثلاثية (صفحة موجودة ↔ route معرّف ↔ ظاهرة في الملاحة).
- **الخادم:** `artifacts/api-server/src/routes/umrah.ts` + `umrah-entities.ts` — حصر endpoints ومطابقتها مع استدعاءات الواجهة (orphan endpoints / orphan calls).
- **المحركات:** `lib/umrah*` — لكل محرك: مَن يستدعيه؟ هل يُستدعى مباشرة من route أم عبر طبقة؟ هل فيه منطق مالي/نقل يخص خادماً؟
- **الكتالوجات:** `umrahReportsCatalog.ts` (17) و`umrahSettingsPoliciesCatalog.ts` (11) — مطابقة الكتالوج مع الواقع (catalog vs routes vs pages vs أثر فعلي على المحرك).
- **الأحداث والتدقيق:** مطابقة كل عملية كتابة مع `emitEvent` + `createAuditLog`، والتحقق من اكتمال حقول الحدث الإلزامية (راجع الفجوات المرصودة في الملف 11).
- **البيئة الحية:** `pnpm db:provision-agent` ثم تشغيل دورات فعلية (استيراد → فوترة → عمولة) والتحقق من القيود المتوازنة والـcounters.
