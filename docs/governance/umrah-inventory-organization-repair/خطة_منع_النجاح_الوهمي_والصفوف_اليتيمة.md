# خطة منع النجاح الوهمي والصفوف اليتيمة

> مرتبطة بـ #2080 (نطاق 27 + المثال 5 + قاعدة «أي استيراد يقول تم بنجاح دون ظهور أثر تشغيلي يعد نجاحاً وهمياً ممنوعاً») و#1870 §3.

## 1. تعريف النجاح الوهمي (العقد الرباعي)

عملية تدّعي النجاح دون أن تستوفي **أربعة أركان معاً**:

| الركن | المعنى | مثال على الاستيراد |
|-------|--------|---------------------|
| **Counters** | أعداد دقيقة لما حدث فعلاً | newCount / updatedCount / skippedCount / errorCount / unlinkedAgentCount / unlinkedGroupCount / unlinkedSubAgentCount |
| **Audit** | سجل تدقيق بمن فعل ماذا | `createAuditLog` بحمولة العملية |
| **Event** | حدث صالح (يجتاز الـvalidator) يصل للمستهلكين | `umrah.import.confirmed` بكل حقوله الإلزامية |
| **أثر مرئي** | الكيانات تظهر فعلاً للمشغّل | معتمرون/وكلاء/مجموعات في قوائمهم + الدفعة في سجل الدفعات + غير المربوط في شاشته |

غياب أي ركن = نجاح وهمي = تصنيف 15، لا يؤجَّل.

## 2. الوضع القائم (إنصافاً قبل الفحص)

البنية التحتية للعقد **موجودة** في مسار الاستيراد: العدّادات السبعة قائمة في `umrahImportEngine` (بما فيها unlinked الثلاثة من migration 279)، الحدث `umrah.import.confirmed` يُبث، شاشة `import-unlinked.tsx` تعرض غير المربوط بثلاث تبويبات وتربطه جماعياً بحارس «ما زال غير مربوط + ينتمي للدفعة»، والعدّاد يتناقص مع كل ربط. **هذا هو النمط المرجعي** الذي تُقاس عليه بقية عمليات العمرة.

## 3. الفجوات المعلومة (تدخل التنفيذ مباشرة)

### 3.1 الصفوف اليتيمة التاريخية (#1983)

**المشكلة:** الدفعات المستوردة **قبل** migration 279 لها صفوف فعلية بلا agentId/groupId/subAgentId، لكن عدّاداتها صفر — فلا تظهر في شاشة الاسترداد، وتغيب عن كل التجميعات بصمت. نجاح وهمي تاريخي مستمر.

**المعالجة (مهمة U-07):** backfill يحسب العدّادات الفعلية لكل دفعة تاريخية (`COUNT WHERE agentId IS NULL AND batch linkage…`) — بعدها الشاشة القائمة تغطيها بلا كود واجهة جديد. الصفوف التي لا يمكن إسنادها لدفعة تُجمع في «دفعة استرداد» اصطناعية موثّقة.

**معيار القبول:** استعلام تحقق يُظهر صفر صفوف يتيمة غير ظاهرة في شاشة الاسترداد؛ عيّنة يدوية تُربط من الشاشة ويتناقص العدّاد؛ audit+event على عملية الـbackfill نفسها.

### 3.2 الأحداث المرفوضة (فجوة Audit/Event حية)

**المشكلة المرصودة بالدليل:** أثناء الاختبارات الحية، الـvalidator يرفض حدثين لنقص حقول إلزامية ويسجل ValidationError في الخلفية:

- `umrah.invoice.generated` — ناقص invoiceId / pilgrimId / total.
- `umrah.commission.calculated` — ناقص commissionId / agentId / amount / period.

أي أن **العملية تنجح لكن حدثها لا يصل** — نجاح وهمي على ركن الـEvent تحديداً: أي مستهلك (إشعارات، تقارير حدثية، تكاملات) أعمى عن هذه العمليات.

**المعالجة (مهمة U-03):** مواءمة حمولة `emitEvent` مع schema الحدثين (أو تصحيح الـschema إن كان هو المنحرف — يُحسم بمراجعة كتالوج الأحداث). 

**معيار القبول:** دورة E2E كاملة (استيراد→فوترة→عمولة) بلا أي ValidationError في سجل الأحداث، وحمولة كل حدث تحمل المعرّفات الفعلية.

### 3.3 تعميم العقد الرباعي على بقية العمليات

الاستيراد هو النمط المرجعي؛ م1 يفحص بقية عمليات الكتابة في العمرة على نفس العقد:

| العملية | فحص م1 |
|---------|--------|
| توليد فاتورة بيع | counters (سطور/مبالغ) ✓ قائمة في الـresponse؛ الحدث فيه الفجوة 3.2؛ الأثر مرئي (الفاتورة في القائمة + JE في الدفتر) |
| احتساب عمولة | نفس فجوة الحدث 3.2؛ الأثر: صف الاحتساب + القيد |
| تسجيل دفعة + توزيعها | counters التوزيع؟ حدث؟ أثر على رصيد الوكيل؟ |
| إنشاء مخالفة/غرامة | حدث+audit؟ أثرها على العمولة (سياسة blockCommission) مفعّل؟ |
| تحديث الحالة اليومي (run-daily-status) | يعيد counters (arrivedUpdated/overstayedUpdated/departedUpdated) ✓ + event ✓ — نمط سليم |
| محرك الغرامات (run-penalty-engine) | يعيد result كاملاً + audit + event ✓ — نمط سليم |
| الربط الجماعي لغير المربوط | linkedCount + audit + event ✓ — نمط سليم |

## 4. قاعدة تصميم ملزمة للمستقبل

أي endpoint كتابة جديد في العمرة لا يُقبل في المراجعة إلا إذا أعاد counters صريحة، وسجّل audit، وبثّ حدثاً صالح الحقول، وكان أثره قابلاً للإشارة إليه في شاشة محددة. تُضاف هذه القاعدة لقالب مراجعة PRات العمرة (الملف 12 §«الممنوع»).

## 5. معيار قبول المحور كاملاً

- صفر صفوف يتيمة غير ظاهرة في شاشة الاسترداد (قديمة وجديدة).
- صفر أحداث مرفوضة في دورة E2E.
- جدول م1 لعمليات الكتابة كلها مستوفياً العقد الرباعي أو له مهمة إصلاح معتمدة.
