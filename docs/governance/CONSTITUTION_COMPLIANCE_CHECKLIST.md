# قائمة فحص الالتزام بدستور غيث

> تستخدم هذه القائمة قبل أي PR جوهري، خصوصاً ما يمس: المسارات، المالية، الصلاحيات، الكيانات، الواجهات، أو عمليات الحفظ.
> لا تعتمد كبديل للاختبارات؛ هي بوابة مراجعة.

---

## 1) فحص الرؤية

- [ ] هل تخدم الميزة العميل أو المستخدم في مساحة عمل موحدة؟
- [ ] هل تمنع التنقل غير الضروري بين صفحات كثيرة؟
- [ ] هل كل حفظ ينتج أثراً تلقائياً واضحاً؟
- [ ] هل ظهر للمستخدم الأثر المتوقع قبل الحفظ إن كانت العملية مؤثرة؟

## 2) فحص المسار والملكية

- [ ] تم تحديد المسار القائد.
- [ ] تم تحديد المسارات المساندة.
- [ ] لا يوجد نقل قرار أو سياسة من القائد إلى المساند.
- [ ] لا توجد كتابة عابرة لمسار آخر بلا Service Contract.
- [ ] الكيان له مالك واحد في `ENTITY_OWNERSHIP_MATRIX.md`.
- [ ] لا يوجد كيان تابع بلا جذر.

## 3) فحص البيانات والنطاق

- [ ] كل قراءة مفلترة بـ `companyId` من scope.
- [ ] كل قراءة تراعي `deletedAt IS NULL` عند اللزوم.
- [ ] كل كتابة تأخذ `companyId` و`branchId` من `req.scope` لا من body.
- [ ] كل جدول تشغيلي يحتوي Owner Context مناسباً.
- [ ] لا يوجد Physical Delete لبيانات تشغيلية/مالية/تعاقدية مستخدمة.
- [ ] كل تاريخ مستند منفصل عن `createdAt`.

## 4) فحص الصلاحيات

- [ ] صلاحيات القراءة/الإنشاء/التعديل/الحذف/الاعتماد محددة.
- [ ] الدور مرتبط بتعيين نشط عند الحاجة.
- [ ] الأزرار تظهر حسب الصلاحية + حالة السجل.
- [ ] لا يوجد endpoint يعتمد على `authMiddleware` فقط إذا كان يحتاج permission gate.
- [ ] لا يوجد تجاوز لنطاق الفرع/الشركة.

## 5) فحص الأحداث والتدقيق

- [ ] كل POST/PUT/PATCH/DELETE ينشئ Audit Log.
- [ ] كل عملية معتبرة تنتج Event.
- [ ] اسم الحدث موثق في `EVENT_CATALOG.md` أو أضيف إليه.
- [ ] الحدث يحتوي `companyId`, `branchId`, `actorId`, `entityType`, `entityId`.
- [ ] العملية قابلة للتتبع: actor + متى + لماذا + diff + أين.

## 6) فحص المالية

- [ ] لا توجد كتابة مباشرة إلى GL خارج المحرك المالي.
- [ ] لا يوجد account code literal بدلاً من mapping.
- [ ] لا يوجد ترحيل على حساب أب `allowPosting=false`.
- [ ] أي أثر مالي يمر عبر mapping وfinancial engine.
- [ ] الفترة المالية مفتوحة قبل الترحيل.
- [ ] التعديل على الفاتورة الصادرة يتم عبر amendment chain فقط.
- [ ] أي أثر VAT يستخدم tax code/helper لا حساباً يدوياً.

## 7) فحص الواجهة

- [ ] لا يوجد منطق أعمال جوهري داخل React component.
- [ ] الحقول المرتبطة بكيانات تستخدم بحثاً ذكياً أو اختياراً متسلسلاً.
- [ ] يظهر سياق الكيان عند اختياره، لا الاسم فقط.
- [ ] توجد تحذيرات سياقية عند التعارضات.
- [ ] لا توجد modal تعامل كصفحة مستقلة لكيان رئيسي.
- [ ] الصفحة مسجلة في visibility/navigation حسب الحوكمة.

## 8) فحص الاعتمادات والطلبات

- [ ] الحالة الانتقالية واضحة: draft/submitted/approved/posted/closed... إلخ.
- [ ] الانتقالات الممنوعة مرفوضة server-side.
- [ ] الاعتماد يمر عبر workflow/approval engine عند الحاجة.
- [ ] الرفض/الإرجاع/الإحالة تسجل سبباً وتاريخاً ومستخدمًا.

## 9) فحص التفعيل والاشتراك

- [ ] الميزة لا تظهر إذا المسار غير مفعل.
- [ ] الخدمة المساندة تظهر داخل سياق المسار القائد لا كقسم مستقل لغير المخول.
- [ ] التفعيل الجزئي لا ينقل القرار للمساند.

## 10) فحص التسليم

- [ ] تم تحديث Blueprint المسار إن تغير التدفق.
- [ ] تم تحديث Event Catalog إن أضيف حدث.
- [ ] تم تحديث Service Contract Catalog إن أضيف تكامل بين مسارين.
- [ ] تم تحديث RBAC Permission Catalog إن أضيفت صلاحية.
- [ ] تم تحديث tests/guards/smokes ذات الصلة.
- [ ] وصف PR يحتوي Constitution Review.

---

## حكم القبول

- أي بند مالي أحمر = رفض.
- أي بند نطاق/صلاحية أحمر = رفض.
- أي كتابة بلا audit/event = رفض.
- أي كيان بلا مالك أو جذر = رفض.
- أي نقص توثيق بسيط يمكن قبوله فقط إذا سجل كـ Follow-up واضح ومحدد.
