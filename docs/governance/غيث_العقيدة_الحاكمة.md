# غيث — العقيدة الحاكمة

> **هذا الملف هو السلطة الأعلى على كل قرار معماري أو تنفيذي في النظام.**
> أي تعارض بين هذا الملف وأي ملف خطة أو audit أو slice doc — يَغلب هذا الملف.
> أي مكتبة جديدة، endpoint جديد، أو سلوك جديد — يُقاس بالمبادئ الـ12 أدناه قبل التنفيذ.

---

## §0 — لمن يقرأ هذا الملف

- **المُلّاك (Owners):** هنا تجد ما الذي يجوز للنظام يفعله تلقائياً بدون رجوع إليك، وما الذي يجب أن يطلب إذناً صريحاً.
- **المطوّرون والوكلاء (Claude/Cursor/etc):** هنا تجد القواعد التي تنزل تحتها كل مهمة، وكل سؤال "هل أنفذ هذا؟" يرجع لإجابة هنا.
- **المراجعون والمُدققون:** هنا تجد الحدود التي إذا تعدّاها PR — يُرفض حتى ولو كان CI أخضر.

## §1 — المبادئ الـ12 (Permanent Hard Rails)

### 1️⃣ صفر JE خارج المحرك المالي
- كل قيد محاسبي (Journal Entry) يمر عبر `lib/businessHelpers.createGuardedJournalEntry` أو الواجهة الأعلى `lib/engines/financialEngine.postSalesInvoice` (لما تنزل SLICE-B).
- ممنوع `INSERT INTO journal_entries` المباشر من أي route خارج هذا المسار.
- **الحارس:** `audit-domain-boundaries.mjs` يفشل CI لو شفت كتابة GL من routes غير المالية.

### 2️⃣ صفر إنشاء عميل / فتح ذمم / ربط صامت
- إنشاء `clients` row يحتاج طلب صريح من المستخدم. ممنوع side-effect من مسار آخر (مثلاً: استيراد معتمر يفتح client رياضياً).
- فتح AR account للوكيل/العميل الجديد يحتاج Owner Decision موثقة في `docs/governance/.../findings/...`.
- **الحارس:** كل مسار كتابة يجب أن يكون له `audit:wiring` يدعمه ولا يخفي إنشاء جانبي.

### 3️⃣ ممنوع تعديل فاتورة صادرة
- بعد ما الفاتورة تصير `status NOT IN ('draft','void')` — لا يُسمح بـ`UPDATE umrah_sales_invoices SET total = ...`.
- التعديل الوحيد المسموح: `amendment_chain` (مرجعها migration 240).
- **الحارس:** `check-sql-ambiguity` + يوجد smoke per-table يثبت absence of UPDATE على issued status.

### 4️⃣ ممنوع mapping محاسبي مكوّد
- كل حساب محاسبي يأتي من `accounting_mappings` عبر `getAccountCodeFromMapping(companyId, operation, ...)`.
- ممنوع تجد في الكود literal مثل `"2120"` أو `"5240"` بدون callback لـ`getAccountCodeFromMapping`.
- **الحارس:** smoke umrahCommissionPayrollCaptureContract pins هذا النمط.

### 5️⃣ ممنوع default flip بدون Owner Decision
- خاصة في: notifications policies، tax codes، الأدوار الافتراضية، الفئات المحاسبية، أوضاع الفوترة (`commission_via_hr`).
- لو لازم تغيّر default، يجب:
  1. توثيق `Owner Decision` في الـPR description.
  2. ربط بـIssue على Github.
  3. smoke يثبت السلوك الجديد ولا يكسر القديم.

### 6️⃣ ممنوع bulk silent (تعديل جماعي صامت)
- migrations لا تحوي `UPDATE … SET` على بيانات الإنتاج بدون filter محدّد ومسوّغ موثق.
- backfill يحتاج migration منفصلة، مع `@rollback` annotation.
- **الحارس:** `check-migration-policy.mjs` يفشل لو شفت migration بدون rollback comment.

### 7️⃣ ممنوع override لـ branch protection
- لا تشغل `--no-verify`، ولا `--force` على main، ولا `gh pr merge --admin`.
- إذا CI أحمر — تَفهَم السبب وتعالجه، لا تتجاوزه.
- **الحارس:** الـtreadmill (guard → rebase → guard → merge) هو الطريقة الوحيدة المعتمدة.

### 8️⃣ كل migration: additive + nullable + IF NOT EXISTS + @rollback
- `ADD COLUMN IF NOT EXISTS … NULL` (لا NOT NULL، لا DEFAULT، لا FK).
- backfill في migration منفصلة لاحقاً.
- `@rollback: ALTER TABLE … DROP COLUMN IF EXISTS …;` في تعليق أعلى الملف.
- **الحارس:** `check-migration-policy.mjs` + `check-duplicate-migrations.mjs`.

### 9️⃣ كل قراءة: tenant-scoped + soft-delete aware
- كل `SELECT FROM <table>` يلزمه `WHERE "companyId" = $1 AND "deletedAt" IS NULL`.
- في JOIN: نفس القاعدة على كل alias.
- **الحارس:** `check-ghost-rows.mjs` + `check-sql-ambiguity.mjs` يضمنان هذا.

### 🔟 كل كتابة: audit + event + companyId
- كل route كتابة (POST/PUT/PATCH/DELETE) يلزمه:
  1. `createAuditLog({ companyId, userId, action, entity, entityId, ... })`.
  2. `emitEvent({ companyId, branchId, userId, action: "<domain>.<entity>.<verb>", ... })`.
  3. `scope.companyId` يُؤخذ من `req.scope` (لا يأتي من body).
- **الحارس:** `check:audit-coverage` (#2418 لما يصلح) سيفرض هذا.

### 1️⃣1️⃣ ممنوع إدخال بدون فرع محدد أو دور محدد
- كل INSERT في جدول تشغيلي يأخذ `branchId` من `req.scope.branchId` (المستخرج من session).
- لو المستخدم له صلاحية على عدة فروع، يلزمه يختار فرعاً صريحاً قبل الإدخال (UI gate + server gate).
- الأدوار: لا role grant بدون `assignmentId` نشط في `employee_assignments`.

### 1️⃣2️⃣ التواريخ: تاريخ الإنشاء مفصول عن تاريخ المستند
- `createdAt`: آلي 100%، `DEFAULT now()` في schema، لا يُدخله المستخدم.
- `documentDate` / `invoiceDate` / `transactionDate` / `entryDate`: المستخدم يدخله، يُعرض ويُسأل عنه.
- ممنوع استخدام `createdAt` كتاريخ مالي. التقارير المالية تفلتر على `documentDate` فقط.
- **الحارس:** smoke per-domain يثبت الفصل (الفلاتر تستخدم `documentDate`، الترتيب يستخدم `createdAt`).

## §2 — ما يجوز للنظام يفعله تلقائياً (Auto-Allowed)

| الفعل | الحال | الشرط |
|---|---|---|
| تعبئة `createdAt` | ✅ نعم | DEFAULT now() في كل migration |
| تعبئة `companyId` من `req.scope` | ✅ نعم | لا يُقبل من body |
| تعبئة `branchId` من `req.scope` | ✅ نعم | إذا المستخدم محدد على فرع واحد |
| تعبئة `createdBy` من `req.scope.userId` | ✅ نعم | session-derived |
| تعبئة `updatedBy` + `updatedAt` على UPDATE | ✅ نعم | إجباري |
| إصدار رقم تسلسلي (`numberingService.issueNumber`) | ✅ نعم | فقط عبر الخدمة المركزية، ممنوع `nextval` مباشر |
| استخراج VAT من `taxCode` (`computeTaxFromTaxCode`) | ✅ نعم | فقط عبر الـhelper |
| حل حساب محاسبي من `accounting_mappings` | ✅ نعم | فقط عبر `getAccountCodeFromMapping` |
| إنشاء `audit_log` row | ✅ نعم | إجباري على كل كتابة |
| إنشاء `event` row | ✅ نعم | إجباري على كل كتابة |
| إنشاء notification داخلي | ✅ نعم | فقط بعد التحقق من opt-out + policy enabled |
| اقتراح تصنيف ذكي (suggestion) | ✅ نعم | الاقتراح يُعرض، المستخدم يقبل/يرفض |

## §3 — ما يطلب إذناً صريحاً (Owner-Gated)

| الفعل | السبب | كيف يُطلب الإذن |
|---|---|---|
| إنشاء `clients` row من side-effect | إعادة فتح نقاش "الربط الصامت" | Owner Decision في PR description |
| فتح AR account جديد | يغير الموازنة | Owner Decision موثق |
| `UPDATE` على فاتورة `status='issued'` | يخالف §3 | يستخدم amendment_chain فقط |
| تفعيل `BILL-MAIN P4/P5` | يغير طرف الفوترة من الفرعي للرئيسي | Owner Decision + migration plan |
| تفعيل `commission_via_hr=false` | يغير حساب الـCR من salary_payable إلى commission_payable | system_settings flag + Owner toggle |
| flip default في policies catalog | يؤثر على tenants existing | issue + smoke |
| تشغيل `postSalesInvoice` SLICE-B (engine wiring) | يحوّل القراءة إلى كتابة GL | Owner Decision + sequence migration |
| تفعيل auto-routing لمصاريف المركبة (purchase/maintenance/fuel) | يكتب GL تلقائياً | Owner Decision + per-category mapping |

## §4 — نمذجة الطلبات (Request Modeling)

كل طلب (Request) في النظام — سواء كان فاتورة، عمولة، صيانة مركبة، عقد إيجار، استلام بضاعة — يَخدُم نفس النموذج:

### 4.1 طبقات الطلب الـ4

```
┌─────────────────────────────────────────────────────┐
│ 1. Operational Layer (المستخدم يدخلها)             │
│    - moduleKey: "umrah" / "fleet" / "properties"   │
│    - entityKey: "sales_invoice" / "maintenance"    │
│    - documentDate (يُسأل المستخدم)                  │
│    - lines: [...]                                   │
│    - notes                                          │
├─────────────────────────────────────────────────────┤
│ 2. Auto-Inferred Layer (النظام يستخرجها)           │
│    - createdAt = now()                              │
│    - companyId = scope.companyId                    │
│    - branchId = scope.branchId                      │
│    - createdBy = scope.userId                       │
├─────────────────────────────────────────────────────┤
│ 3. Service Layer (تستخرج عبر خدمة مركزية)          │
│    - number = numberingService.issueNumber()        │
│    - tax = computeTaxFromTaxCode()                  │
│    - account = getAccountCodeFromMapping()          │
│    - period = checkFinancialPeriodOpen()            │
├─────────────────────────────────────────────────────┤
│ 4. Financial Outputs (المحرك يخرجها)               │
│    - journalEntryId                                 │
│    - arAccountCode + revenueAccountCode             │
│    - postingStatus: posted/deferred/failed         │
└─────────────────────────────────────────────────────┘
```

### 4.2 قاعدة "ممنوع التخمين"

- كل حقل في طبقة 1 يجب أن يأتي من user input أو اختيار قائمة. ممنوع توليده تلقائياً.
- كل حقل في طبقتين 2/3/4 ممنوع المستخدم يدخله. يُحسب آلياً.
- لو طبقة عبرت حدّها (مثلاً المستخدم أدخل `createdAt`) — يُرفض الطلب بـ422.

### 4.3 الفرز الذكي (Smart Routing)

عند إنشاء طلب صرف/قبض/مصروف، تظهر للمستخدم قائمتان متعاقبتان:

1. **القائمة الأولى — جهة العملية:**
   - عميل / مورد / موظف / مركبة / عقار / معتمر / مستودع / مشروع / إداري عام

2. **القائمة الثانية — نوع العملية حسب الجهة:**
   - مركبة → شراء / صيانة / وقود / تأمين / بيع
   - عقار → إيجار صادر / إيجار وارد / صيانة / مرافق
   - معتمر → فاتورة بيع / استرداد / غرامة / دفعة
   - موظف → راتب / عمولة / سُلفة / مكافأة
   - … (كل جهة تعرّف قائمتها)

3. **بعد الاختيار، النظام يستحضر:**
   - mapping محاسبي من `accounting_mappings.operation` المطابقة
   - النموذج المطلوب (الحقول الإضافية حسب النوع)
   - تاريخ المستند، الفرع، المستخدم — كلها جاهزة

### 4.4 معالجة الملف المكرر / الجزئي

عند استيراد ملف (Excel، CSV، JSON):

| الحالة | السلوك |
|---|---|
| الصف كاملاً مطابق لصف موجود (hash يطابق) | يُرفض الصف بدون توقيف الباقي + يُعرض في "duplicates" tab |
| الصف فيه partial match (مثلاً نفس passport, اسم مختلف) | يُعرض للمستخدم: "هل هذا نفس المعتمر؟" + لا يُكتب حتى يجاوب |
| الصف جديد كلياً | يُكتب بعد validation |
| الصف صحيح لكن نقصه FK (مثلاً seasonId غير موجود) | يُكتب في `unlinked` بشرط الـmandatory fields موجودة، ويُعرض في `/import/:batchId/unlinked` للاستكمال اليدوي |

**الحارس:** كل importer (umrah, fleet, properties) يلزمه:
- hash field على الصف (sha256 of canonical keys) لمنع التكرار
- `import_batches` row يربط الـbatch
- `unlinked` flow بدل الـfail-fast

## §5 — المكتبات المشتركة (Shared Libraries) — الترتيب الإلزامي

ترتيب الاستدعاء صارم: المكتبة الأعلى لا تستدعي مكتبة أسفل منها مباشرة، فقط عبر الطبقة التي تسبقها.

```
Level 0 — Foundation (لا تعتمد على شيء)
  ├─ lib/rawdb.ts            ← قاعدة البيانات (raw query + transactions)
  ├─ lib/logger.ts           ← logging
  ├─ lib/errorHandler.ts     ← error classes
  └─ lib/settings.ts         ← system_settings read/write

Level 1 — Cross-cutting Services (يعتمد على Level 0)
  ├─ lib/numberingService.ts ← إصدار الأرقام التسلسلية
  ├─ lib/taxCodes.ts         ← حساب الضريبة من الكود
  ├─ lib/businessHelpers.ts  ← createGuardedJournalEntry, getAccountCodeFromMapping, createNotification
  ├─ lib/eventBus.ts         ← emitEvent
  └─ lib/auditLog.ts         ← createAuditLog

Level 2 — Domain Engines (يعتمد على Level 0/1 فقط)
  ├─ lib/engines/financialEngine.ts   ← postSalesInvoice, postPurchase, etc
  ├─ lib/engines/hrEngine.ts          ← payroll, leave, contracts
  ├─ lib/engines/legalEngine.ts       ← cases, sessions
  ├─ lib/engines/warehouseEngine.ts   ← movements, stock
  ├─ lib/engines/supportEngine.ts     ← tickets
  └─ lib/umrahInvoicingEngine.ts      ← (سينضم لـ financial بعد FIN-P4-SLICE-C)

Level 3 — Print/Notification/Calendar (يعتمد على Level 0/1)
  ├─ lib/print/templateResolver.ts   ← BESPOKE_PRESETS + universalFallback
  ├─ lib/print/dataLoader.ts         ← load<Entity> functions
  ├─ lib/umrahInternalNotifications.ts ← in-app notifications
  ├─ lib/cronScheduler.ts            ← all crons
  └─ lib/calendar/*                  ← unified calendar API

Level 4 — Routes (يعتمد على Level 0-3)
  └─ routes/*.ts                     ← HTTP endpoints + RBAC + auditing
```

**القواعد:**
- ممنوع import من Level أعلى إلى Level أدنى.
- ممنوع route يستدعي route آخر مباشرة — يستدعي محرك Level 2 فقط.
- كل محرك مجاله جدوله المملوك. ممنوع `hrEngine` يكتب على `umrah_pilgrims`.

## §6 — الأنظمة المركزية (Single Source of Truth)

### 6.1 نظام الترقيم — `lib/numberingService.ts`

- كل رقم تسلسلي رسمي (فاتورة، عقد، عمولة، طلب صرف، عرض، تذكرة، عملية مركبة، …) يصدر عبر `issueNumber({moduleKey, entityKey, companyId, ...})`.
- ممنوع `nextval` مباشر في routes.
- ممنوع `generateTimeRef`, `generateRef`, `random` كتمويه لرقم رسمي.
- ممنوع `COUNT(*) + 1` أو `MAX(...) + 1` ك pattern.
- **الحارس:** `audit-numbering-coverage.mjs` + `audit-numbering-service-bypass.mjs` + `audit-numbering-schemes-vs-callers.mjs`.

### 6.2 نظام التوحيد للمكتبات — رصد + منع التشعب

- لكل خدمة عرضية: مكتبة واحدة فقط مركزية، لا two implementations.
- مثال: حساب الضريبة فقط `computeTaxFromTaxCode`، لا helper ثاني في `umrahInvoicingEngine`.
- **الحارس:** code review manual + كل مكتبة جديدة تحتاج audit doc يبرر سبب وجودها (لا تكرار).

### 6.3 نظام الطباعة — `lib/print/templateResolver.ts`

- كل قالب طباعة عبر `BESPOKE_PRESETS[entityType]?.() ?? universalFallback(entityType)`.
- كل entity يطبع يجب أن يكون له:
  1. `case "<entity>"` في `dataLoader.ts`
  2. `<entity>: () => build<X>Preset()` في `BESPOKE_PRESETS`
  3. `<entity>: "<Arabic title>"` في `ARABIC_TITLES`
- presetKey نموذج: `<entity>_classic`.
- seed id نطاق سالب يبدأ من -58 ويُعطى للقوالب الجديدة بترتيب لا يتعارض.
- **الحارس:** `umrahPrintTemplatesCoverageSmoke` + `printEngine.test.ts` consistency check.

### 6.4 التقويم الموحد — `lib/calendar/*`

- مكتبة موحّدة لكل أنواع المناسبات: فرع تشغيلي، رحلة، إجازة موظف، قسط، صيانة، …
- API موحّد:
  ```
  GET /calendar/events?from=&to=&domain=&entityId=
  POST /calendar/events
  ```
- ممنوع كل domain يكتب route تقويم خاص. كل domain يصدر events لمكتبة التقويم.
- **الفلاتر القياسية:** الفرع، الشركة، المستخدم، النوع.

### 6.5 مكتبة الأدوار والصلاحيات — `lib/rbac/*`

- كل صلاحية تُحدد بـ`{ feature: string, action: "view"|"list"|"create"|"update"|"delete"|"approve" }`.
- ممنوع `if (user.role === 'admin')` مباشر في routes.
- يُستخدم: `authorize({ feature, action })` كـmiddleware.
- المرجع: `rbac/featureCatalog.ts` يعرّف كل feature/action.
- **الحارس:** `audit-domain-routes.mjs` + check authorize coverage.

### 6.6 نظام السلوكيات التشغيلية — Branch + Role Gating

- ممنوع `POST /any-write` بدون `branchId` في `req.scope`.
- لو المستخدم له فروع متعددة، الـUI يفرض اختياره قبل الفتح.
- لو الرول غير محدد على `assignmentId` نشط → 403.
- **الحارس:** `audit-branch-scope.mjs` (مقترح).

## §7 — الفرق بين تاريخ الإنشاء وتاريخ المستند

| الحقل | المصدر | الاستخدام | يَظهر للمستخدم؟ |
|---|---|---|---|
| `createdAt` | آلي 100% — `DEFAULT now()` | ترتيب الـrows في القوائم، tracking | يظهر في "متى أُدخل النظام" |
| `documentDate` | المستخدم يدخله | الفلاتر المالية، فترات المحاسبة | الحقل الأساسي للمستخدم |
| `dueDate` | المستخدم يدخله | تواريخ الاستحقاق | الفلاتر التشغيلية |
| `updatedAt` | آلي 100% — `DEFAULT now()` ON UPDATE | tracking آخر تعديل | "آخر تعديل" |
| `deletedAt` | آلي — يُحدد عند `soft delete` | soft-delete filter | لا |

**قاعدة الذهب:** التقارير المالية تفلتر على `documentDate` ولا تستخدم `createdAt`. التقارير التشغيلية (آخر استيراد، آخر تحديث) تستخدم `createdAt` / `updatedAt`.

- المستوردات تحوي `hash` row في `umrah_pilgrims`.
- لو الصف بنفس `passportNumber + companyId` موجود → يُرفض ويُعرض في "duplicates" tab.
- لو نقصه `groupId` → يُكتب في `umrah_pilgrims` بدون groupId، يظهر في `/umrah/import/:batchId/unlinked`.
- لو الـbatch فيه صفوف صحيحة + صفوف مكررة + صفوف ناقصة:
  - الصحيحة → تنزل
  - المكررة → تظهر في "duplicates" (لا يُكسر الـbatch)
  - الناقصة → تنزل بـunlinked status

**ما ينقص (المرحلة القادمة):**
- import_batches summary row يحفظ الإحصاء (3 inserted, 2 dup, 1 unlinked).
- إعادة محاولة per-row من واجهة الاستيراد.

## §9 — قاعدة "أبسط ما يمكن"

- ممنوع abstraction قبل ما يكون فيه 3 use-cases فعلية.
- ممنوع helper function قبل ما يكون فيه 3 callers.
- ممنوع feature flag قبل ما يكون فيه قرار تشغيلي يحتاجه.
- ممنوع config option قبل ما يكون فيه tenant يطلبها.

## §10 — قائمة المراجعة قبل دمج أي PR

كل PR يجب أن يجيب على هذه الأسئلة في الـdescription:

1. **هل يخرق أحد المبادئ الـ12؟** → لا/نعم (ومتى).
2. **هل يحتاج Owner Decision؟** → لا / نعم + لينك للقرار.
3. **هل migration additive + nullable + @rollback؟** → نعم.
4. **هل smoke pins فشلت في كل من: alias regress + tenant scope + write absence؟** → نعم.
5. **هل CI أخضر بدون override؟** → نعم.
6. **هل أي default يتغيّر؟** → لا / نعم + sjbeer for tenants.
7. **هل أي endpoint جديد يكتب؟ هل audit + event مغطّيان؟** → نعم.
8. **هل أي column جديد في جدول fact مهم؟ هل صورة الـschema (`db/schema_pre.sql`) محدّثة؟** → نعم.

## §11 — متى يتوقف الوكيل التلقائي

- لو PR فيه conflict مع `main` يحتاج فهم semantic → يتوقف ويسأل.
- لو slice يحتاج owner decision → يتوقف ويفتح Issue/Comment.
- لو CI أحمر لسبب غير معروف بعد 2 محاولات → يتوقف ويعرض اللوغ.
- لو رأى نمطاً مكرراً في الكود يطلب refactor عريض → يتوقف ويقترح، لا ينفذ.
