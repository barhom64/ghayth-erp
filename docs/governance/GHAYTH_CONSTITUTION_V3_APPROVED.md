# دستور غيث — المرجع الأعلى (طبقة 0) · v3 — معتمد ومجمّد

> المرجع النهائي لنظام غيث. أي كود أو قرار يخالفه يُرفض. المجلس `ghayth-review` يفحص كل عمل عليه قبل أي اعتماد أو دمج.
> هذه وثيقة مبادئ مجمّدة؛ لا تُضاف إليها تفاصيل تشغيلية. مواصفات كل مسار في وثائق منفصلة.

---

## 1) الهوية والرؤية

- **النظام:** منصة غيث — نظام تشغيل مؤسسي متكامل `ERP` لمجموعة الدور الحديثة وشركاتها.

**الرؤية الذهبية:** العميل مركز كل شيء. الموظف يدخل ويرى كل ما عليه في **صفحة واحدة** ويتصرف. **كل عملية تنشئ سلسلة أفعال تلقائية** — لا حفظ بسيط. الموظف الواحد قد يكون مسجلاً في أكثر من شركة وفرع بمسمى مختلف. أي تغيير لا يخدم الرؤية = خارج الرؤية = مرفوض.

## 2) مساحة العمل الموحدة

عند الدخول، يرى المستخدم مساحة عمل موحدة تعرض كل مهامه واعتماداته وتنبيهاته وإجراءاته من كافة المسارات وفق صلاحياته، دون تنقل بين عشرات الصفحات. هذا جوهر الرؤية وليس تحسيناً اختيارياً.

## 3) المبادئ الحاكمة السبعة

1. الموظف شخص واحد وله عدة تعيينات: `employees` للشخص و`employeeAssignments` للتعيين.
2. ما يتبع الشخص لا يساوي ما يتبع التعيين.
3. العميل مركز كل شيء: ملف 360° + بوابة.
4. كل استعلام يتفلتر حسب الصلاحيات.
5. كل حفظ يطلق أفعالاً تلقائية.
6. الإعدادات تورث: نظام ← شركة ← فرع، والأقرب يفوز.
7. البناء بالترتيب. الصرامة قبل السرعة.

---

## 4) استقلال المسارات

كل مسار وحدة مستقلة ذات حدود خدمة واضحة: قابلة للتشغيل والاشتراك منفردة بمنطقها وإعداداتها وصلاحياتها وتقاريرها وتدقيقها وأحداثها. التكامل عبر عقود خدمة معتمدة فقط، دون نسخ منطق أعمال مسار إلى آخر.

## 5) المسار القائد والمسار المساند

القائد يملك القرار والسياسة ويقود الإجراءات. المساند يقدم خدمة محددة فقط: لا قرار، لا سياسة. مثال: HR قائد في الرواتب؛ المالية مساند تنشئ القيد وترحله؛ الوثائق تحفظ؛ الإشعارات ترسل. ممنوع نقل السياسة أو القرار بين القائد والمساند.

## 6) قفل الحدود الخدمية

لا يعدل مسار بيانات أو سياسات أو حالات مسار آخر إلا عبر واجهات خدمة معتمدة بصلاحية ومجال واضحين، مع تدقيق إلزامي على كل إنشاء أو تعديل أو حذف أو اعتماد أو إحالة. أي كتابة عابرة للنطاقات = خلل معماري جسيم يرفض.

## 7) التفعيل الجزئي الذكي

تفعيل مسار قائد يفعّل المساندة تلقائياً كخدمات فقط: محدودة، مضبوطة الصلاحيات، بلا منطق قرار، قابلة للتعطيل. ممنوع أن يصبح المساند اعتماداً جوهرياً يمنع استقلال القائد.

## 8) المنهجيات المستقلة للمسارات

لكل مسار منهجيته وحالاته وأحداثه وتقاريره وصلاحياته. التطبيع فقط في نقاط الالتقاء: التدقيق، الأحداث، الصلاحيات، الإعدادات، القيود، لا في جوهر المسار.

## 9) مصدر الحقيقة الواحد

لكل كيان مالك بيانات واحد فقط. يمنع إنشاء نسخ مكررة للبيانات في مسارات أخرى؛ المسارات المستفيدة تقرأ أو تتكامل عبر العقود المعتمدة.

## 10) مبدأ الأحداث

كل عملية تشغيلية معتبرة تنتج حدثاً `Domain Event` موثقاً قابلاً للتتبع. تعتمد المسارات على الأحداث والعقود لا على القراءة المباشرة لحالة مسار آخر متى أمكن. لا إجراء مؤسسي مهم بلا Event.

## 11) الكيانات الجذرية المعتمدة

الجذور المؤسسية: العميل، الموظف، الشركة، الفرع، العقار، المشروع، المركبة، المورد، العقد، القضية، المخزن. أي كيان جديد يحدد صراحة: هل هو جذر أم تابع لجذر قائم؟ ولا يوجد كيان تابع بلا جذر.

## 12) فصل الواجهة عن منطق الأعمال

الواجهات تعرض البيانات وتجمع المدخلات فقط. منطق الأعمال والسياسات والتحققات والأحداث في طبقة الخدمات والنطاقات. يمنع وضع قواعد الأعمال الجوهرية داخل مكونات الواجهة.

## 13) العزل المؤسسي متعدد الشركات

جميع الكيانات والعمليات والتقارير تعمل ضمن بيئة متعددة الشركات والفروع. يمنع أي تصميم يفترض شركة واحدة. العزل بين الشركات افتراضي وإجباري.

---

## 14) أدوار الصلاحيات

الأدوار تتبع التعيين لا الشخص: المالك، مدير النظام، مدير الفرع، HR، المدير المالي، مدير القسم، موظف. نفس الشخص قد تختلف صلاحياته بين شركة وأخرى بحسب تعيينه.

## 15) القواعد المعمارية الملزمة

1. لا صفحات منبثقة كصفحات؛ كل كيان رئيسي له صفحة أم.
2. لا كيان تابع بدون جذر.
3. لا إدخال حر في حقل مرتبط بكيان؛ يستخدم بحث ذكي أو اختيار متسلسل أو قيمة مقترحة.
4. الاختيار المتسلسل إلزامي.
5. الأثر المتوقع قبل الحفظ.
6. التحذيرات السياقية إلزامية.
7. السياق عند اختيار كيان إلزامي؛ لا الاسم فقط.

## 16) المعمارية المالية

دفتر أستاذ عام ثابت `GL` + دفاتر فرعية ديناميكية `Subledger` + أبعاد تحليلية. يمنع تضخيم دليل الحسابات أو استخدام الحسابات الرئيسية في التشغيل اليومي. تتبع المركبات والسائقين والمشاريع والعقارات عبر الأبعاد التحليلية لا حسابات GL منفصلة.

## 17) منع الترحيل على الحسابات الأب

الحساب الأب `allowPosting=false` لا يظهر للمستخدم ولا يقبل الترحيل. الترحيل على الحسابات الورقية فقط. أي fallback صامت إلى حساب أب أو غير مخصص = رفض فوري.

## 18) منع الحذف الفيزيائي

الأصل التعطيل أو الأرشفة أو الإلغاء المنطقي. يمنع الحذف الفيزيائي للبيانات التشغيلية أو التعاقدية أو المالية بعد استخدامها. أي حذف استثنائي موثق ومراجع وقابل للتدقيق.

## 19) تصنيف البيانات

1. مرجعية ثابتة لا تمسح أبداً.
2. إعدادات تفعل أو تعطل ولا تحذف.
3. تشغيلية.
4. تعاقدية.

## 20) القواعد العشر الملزمة

1. لا تبني خارج الرؤية أو المواصفات.
2. لا جدول بلا مبرر معتمد.
3. لا صفحة غير مربوطة.
4. كل حفظ يمر بالتحققات.
5. كل `SELECT` يمر بفلتر الصلاحيات.
6. كل جدول تشغيلي يحتوي مرجع الملكية المناسب `Owner Context`.
7. لا تصف الشكل؛ صف البيانات والتدفق.
8. لا تقفز بين المراحل.
9. كل إجراء له أثر متتبع: actor + متى + لماذا + diff + أين.
10. كل تسليم يشمل: مرجع المواصفات + الاستعلامات + التحققات + الأفعال التلقائية + الإشعارات + الاختبار.

## 21) ما يوقف ويصعد

تغييرات الدفتر المحاسبي أو معمارية المسارات أو الكيانات الجذرية المعتمدة تخضع لسياسة الاعتماد: إجماع المجلس + اختبارات assertion خضراء + سجل قرار. أي عمل خارج الرؤية أو يخرق بنداً من هذا الدستور يرفض ولا يدمج.
