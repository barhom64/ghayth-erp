## 0. الأساس الموجود (مثبت) — النموذج مقصود لا مكرّر

✅ النظام يفصل عمدًا (موثّق في `migrations/274`):

| الكيان | المعنى | الحقول | الدليل |
|--------|--------|--------|--------|
| `job_titles` | **الهوية المهنية** (محامي/محاسب/سائق) | `name`, `category`, `companyId` | `012_job_titles.sql` |
| `positions` | **المنصب الإداري** (مدير قسم/مشرف/نائب) | `positionKey`, `level` هرمي | `274_org_model_foundation.sql` |
| `categoryKey` | **فئة سياسة الحضور** (driver=30s, field=300s, admin=0) | على الموظف | PR-3 (#2077) |
| `employee_assignments.positionId` | ربط التعيين بالمنصب | FK→positions | `274` |
| `employees.jobTitle` + `employee_assignments.jobTitle` | **نصّ legacy غير مُطبَّع** 🔴 | text | `schema/index.ts:63` |

🏛 **ADR-HR-01 (محسوم اتجاهه):** يُبقى التمييز `job_titles`(مهني) × `positions`(إداري)، **ويُطبَّع العمودان النصّيان** ليرجعا إلى `job_titles.id` (FK). القالب يُبنى على المثلّث: **(job_title.category + position.level + categoryKey)**.

## 1. مفهوم Job Activation Profile

لكل (مسمى مهني × فئة تشغيلية) **profile** يحدّد المتطلبات تلقائيًا — بدل «عهدة/مركبة إن وجدت».

| البُعد | المصدر | مثال |
|--------|--------|------|
| الدور المقترح | `job_titles.category` → defaultRoleKey | محاسب→`finance` مقيّد |
| نطاق الصلاحية الافتراضي | position.level + الفئة | مدير قسم→department |
| سياسة الحضور | `categoryKey` | سائق→GPS 30s |
| الوثائق المطلوبة | profile | سائق→رخصة قيادة |
| حساب مستخدم؟ بريد داخلي؟ | profile | إداري→نعم |
| عهدة عينية/نقدية؟ | profile (طلب خدمة مستودع) | سائق→جهاز/شريحة |
| مركبة؟ | profile (طلب خدمة أسطول) | سائق→نعم |
| تدريب/اعتماد؟ | profile | — |
| ربط مشروع/مركز تكلفة؟ | profile | سائق→نعم |
| من يعتمد كل جزء؟ | جدول §3 | — |

## 2. أمثلة القوالب الإلزامية — ✅

### Driver (سائق)
- رخصة قيادة (وثائق) · سياسة حضور ميدانية/GPS (categoryKey=driver) · **طلب** تخصيص مركبة (أسطول) · عهدة جهاز/شريحة عند اللزوم (مستودع) · مركز تكلفة/مشروع · صلاحيات ميدانية محدودة (scope=self).

### Accountant (محاسب)
- صلاحية مالية مقيّدة · **لا مركبة · لا GPS** · ربط مركز تكلفة إن لزم · **لا رؤية للتحقيقات إلا grant صريح**.

### Branch Manager (مدير فرع)
- نطاق **فرع** · إدارة موظفي الفرع · **لا رواتب إلا grant** · اعتماد إجازات/حضور ضمن الفرع.

## 3. جدول المسار الخادم المسؤول عن كل متطلب — القاعدة الحاكمة

| المتطلب | المسار المالك | عقد الاستدعاء |
|---------|---------------|----------------|
| الحساب + الصلاحيات + النطاق | RBAC/admin (HR يربط) | grant عند التعيين |
| البريد الداخلي + PBX | الاتصالات | طلب توفير |
| العهدة العينية/النقدية | **المستودع** | طلب صرف + وثيقة استلام |
| المركبة | **الأسطول** | طلب تخصيص (لا إنشاء) |
| مركز التكلفة/المشروع | المالية | ربط |
| الوثائق/الرخص | الوثائق | تحقق |
| سياسة الحضور | HR | تطبيق categoryKey |

🚩 **ممنوع (تأكيد #2223):** نقل منطق الأسطول/المستودع إلى HR · عهدة تلقائية بلا سياسة ووثيقة · إنشاء مركبة من HR (HR يطلب خدمة فقط).

## 4. آلية توليد المهام من Profile

عند `quick-activate` (HR-REV-3): يُقرأ profile الـ(job_title+category) → تُولَّد بنود الخطة، كلٌّ بـ**مالك + deadline + سبب + إلزامي/اختياري + serviceType** → تُكتب في `onboarding_tasks` (موسّع) → إشعار لكل جهة.

---

## 5. أثر تغيير Profile

🏛 تغيير profile يطبَّق على **الموظفين الجدد فقط** افتراضيًا؛ ترحيل الموظفين القائمين يتطلب **أمر ترحيل صريح** (لا تغيير صامت لمتطلبات موظف مفعّل).
