## 2. الأنماط الإيجابية (Strengths)

### 2.1 مكتبة UI موحدة
المستودع يعتمد مكتبات داخلية واضحة:
- `@workspace/ui-core` — PageShell, FormShell, DataTable, PageStatusBadge
- `@workspace/entity-kit` — DetailPageLayout, EntityComments, EntityTimeline
- `@workspace/workflow-kit` — ApprovalActions
- `@workspace/report-kit` — PrintLayout

**الأثر**: تطوير صفحة جديدة يستغرق أقل وقت، والـUX يبقى متسقاً عبر الوحدات.

### 2.2 typed errors + Arabic-first
الـbackend يرمي:
```typescript
throw new ValidationError("اسم المستأجر مطلوب", { field: "name", fix: "أدخل الاسم الكامل" });
```

الـclient `apiFetch` يحوّل هذا لـtoast عربي مع شرح. **88%** من الأخطاء بالعربية.

### 2.3 PageStateWrapper موحد
حالة Loading/Error/Empty لكل صفحة تمر عبر component واحد `<PageStateWrapper>` يفهم `ApiError.code` ويعرض رسالة محددة:
- 401: "يرجى تسجيل الدخول"
- 403: "ليس لديك صلاحية الوصول"
- 404: "البيانات غير موجودة"
- 5xx: "حدث خطأ في الخادم"

**أثره**: المستخدم يفهم سبب الفشل بدل رسالة عامة.

### 2.4 صلاحيات في الواجهة (`<GuardedButton>`)
كل زر حساس يستخدم `<GuardedButton perm="properties.tenants:update">` فيختفي تلقائياً إذا الموظف بدون صلاحية. **لا يحتاج المستخدم رؤية أزرار لا يستطيع استخدامها.**

## 3. نقاط ضعف محتملة (تحتاج اختبار يدوي للتأكيد)

### 3.1 نسبة استخدام الـcanonical components منخفضة

| Layout | الاستخدام | الحالة |
|---|---|---|
| DetailPageLayout | 78/578 | 🟡 13% |
| FormShell | 85/578 | 🟡 15% |

**الإحتمال**: 80%+ من الصفحات تستخدم نمط مخصص بدلاً من الـlayouts الموحدة. قد يعني:
- (a) تنوع لـreasons معقولة (Dashboard pages مثلاً)
- (b) ديون فنية وعدم اتساق UX

**يحتاج**: مراجعة من designer.

### 3.2 سايدبار بـ517 إدخالاً
ضخامة الـsidebar قد تجعل العثور على الميزة صعباً. مهام جديدة:
- هل التصنيف منطقي؟
- هل توجد بحث في الـsidebar؟
- هل ترتيب الأهمية واضح؟

**يحتاج**: اختبار يدوي مع مستخدم جديد.

### 3.3 1407 typed error — لكن 161 منها (12%) ليست عربية
بعض الأخطاء النادرة قد تظهر بالإنجليزية للمستخدم العربي. حتى ولو نادرة، فهي مشكلة UX.

**اقتراح**: scan كل `throw new ValidationError("...")` يكتشف الإنجليزية ويصدر قائمة fix-list.

### 3.4 لا يوجد Onboarding Tour
الفحص لم يكتشف:
- `react-joyride` أو شبيهه
- صفحة "أول مرة تستخدم النظام"
- Tutorials مدمجة

**الأثر المحتمل**: مستخدم جديد قد يحتاج تدريب مفصل. (يحتاج اختبار يدوي للتأكيد).

## 4. أنماط محددة تستحق التركيز في الـPlaywright

عند بناء اختبارات Playwright، يجب التحقق من:

| السيناريو | لماذا |
|---|---|
| **تسجيل دخول → لوحة معلومات** | golden path (موجود في `e2e/tests/auth.spec.ts` ✓) |
| **إنشاء عميل → ظهوره في القائمة → الذهاب لتفاصيله** | CRUD أساسي |
| **زر مخفي بدون صلاحية** | اختبار `GuardedButton` |
| **رسالة خطأ بالعربية على failed submit** | اختبار i18n |
| **PageStateWrapper يظهر "حدث خطأ"** | اختبار error UX |
| **DataTable يدعم البحث + التصفية + الفرز** | اختبار قوائم |
| **DetailPageLayout يعرض تابات صحيحة** | اختبار navigation |
| **FormShell لا يسمح بحفظ بدون حقول إلزامية** | اختبار validation |
