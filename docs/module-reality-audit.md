# تدقيق واقعية الوحدات — نظام غيث

> تاريخ التدقيق: 2026-06 · المنهج: فحص الكود الفعلي (backend + جداول + واجهة) لا الملخّصات.
> سؤال التدقيق: هل وحدات النظام **تشغيلية فعلية** أم **قشور ببيانات وهمية**؟

## الخلاصة التنفيذية

**لا توجد بيانات وهمية (ملفّقة) في أي مكان بالنظام.** مُسحت ~120 مساره خلفية: كلّ وحدة رئيسية لها backend ضخم باستعلامات SQL حقيقية، وواجهة تجلب عبر API فعلي، بلا مصفوفات mock. أسوأ ما وُجد ليس «بيانات وهمية» بل **«ميزة غير مُهيّأة ترجع فارغًا وتقول ذلك بصراحة»** — وهذا لا يخالف المادة 14 من الدستور.

التمييز الجوهري: **«فارغ» ≠ «وهمي».** الجدول قد يكون فارغًا عندك (لا بيانات أُدخلت) لكنه مبنيّ وفعّال. لكشف المملوء من الفارغ في قاعدتك الحيّة، شغّل `scripts/module-usage-report.sql`.

## المسح الشامل (~120 مساره خلفية)

الغالبية الساحقة مساراتٌ ضخمة باستعلامات حقيقية، أمثلة على العمق:
`hr` (742 استعلامًا) · `fleet` (370) · `properties` (330) · `umrah-entities` (440) · `finance-invoices` (229) · `finance-purchase` (183) · `projects` (185) · `bi` (149) · `warehouse` (151).

## الحدود الحقيقية الوحيدة (صادقة، ليست وهمًا)

هذه **ليست بيانات وهمية**، بل ميزات لم تُهيّأ/تُكمل بعد، تُصرّح بذلك:

1. **تكاملات حكومية/مصرفية غير مُهيّأة** (`routes/wiring-stubs.ts`): من 23 endpoint فيه، 10 تستعلم فعلًا، والباقي يرجع فارغًا **مع ملاحظة صادقة** (مثل `{ data: [], note: "Mudad integration not configured" }`) — **صفر بيانات ملفّقة**. الميزات المعنيّة:
   - مدد/حماية الأجور WPS: `/saudi/mudad/settlements`, `/saudi/wps/credentials`.
   - قواعد التسعير: `/pricing/rules`, `/pricing/resolve`.
   - الفوترة الإلكترونية: `/zatca/missing-tax-numbers`, `/zatca/pause-history`.
   - الاستخراج الضوئي: `/ocr/extractions/:id/confirm|reject`.
2. **نظاما موافقة متعايشان** (ملاحظة معمارية — **لا فجوة تنفيذ**): محرك سير العمل العام **موجود ويعمل ويُستعمل**: `lib/workflowEngine.ts` (1042 سطرًا) يُنفّذ الدورة الكاملة (submit/approve/reject/refer/escalate/return + تقدّم الخطوات) ويُحدّث حالة الكيان الفعلي (إجازات/خطابات/قيود/…)، ويغذّيه **7 مواضع حقيقية** (السلف، المشتريات، الخروج، الإضافي، طلبات HR). يتعايش معه `approval_chains` (الإجازات والسلاسل الديناميكية). توحيدهما قرار معماري — لا بناء، فالمحرك قائم.
   > تصحيح: نسخة سابقة من هذا التقرير وصفت سير العمل خطأً بأنه «بلا منفّذ موحّد». الصواب: المنفّذ موجود ومُستعمَل، والمسألة الحقيقية هي تعايش نظامَي موافقة.
3. **ميزة الـLLM**: الكود حقيقي وموصول، لكن لا تعمل إلا بضبط `AI_INTEGRATIONS_ANTHROPIC_API_KEY` و`..._BASE_URL`؛ بدونها ترجع «AI غير مفعل» بصدق (التحليلات غير الـLLM تعمل دائمًا).

## كيف تكشف المملوء من الفارغ في قاعدتك

```bash
psql "$DATABASE_URL" -f scripts/module-usage-report.sql
```
يعرض كل جدول وعدد سجلاته (للقراءة فقط). الجداول ذات العدد 0 = مبنيّة لكن لم تُدخَل بياناتها.

## قرارات مطلوبة من إبراهيم

هذه تحتاج قرارك/بياناتك (لم تُنفَّذ من تلقاء الوكيل وفق الدستور):

- [ ] **تفعيل LLM؟** ضبط مفتاح Anthropic لتشغيل التصنيف/التلخيص/الترجمة الذكية.
- [ ] **تكاملات مدد/WPS/ZATCA**: هل نُكمل تنفيذها (تحتاج اعتماد مزوّد + بيانات اعتماد)؟ أم تبقى فارغة حتى الحاجة؟
- [ ] **توحيد نظامَي الموافقة** (قرار معماري): `workflowEngine` (قائم ومُستعمَل من 7 مواضع) مقابل `approval_chains` (الإجازات/السلاسل الديناميكية) — هل نوحّدهما في مسار واحد؟ **لا حاجة لبناء محرك جديد** — كلاهما يعمل.
