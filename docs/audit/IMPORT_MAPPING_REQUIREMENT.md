# متطلّب عام مؤجَّل — معاينة الاستيراد وربط الأعمدة

> **النوع:** متطلّب تصميم عام (standing design rule) — مُلتقَط أثناء تدقيق الجرد، **مؤجَّل التنفيذ**.
> **التاريخ:** 2026-05-22 · **المستودع:** `barhom64/ghayth-erp`
> **القرار:** أمر المالك — هذه قاعدة عامة تُسجَّل الآن ولا تُنفَّذ ضمن موجة إصلاح العيوب الحاجبة الحالية.

---

## 1. القاعدة

**كل ميزة استيراد في نظام غيث — حالية أو مستقبلية — يجب أن توفّر للمستخدم خطوتين قبل تنفيذ الاستيراد:**

1. **معاينة (Preview):** عرض عيّنة من صفوف الملف المرفوع قبل الإدراج، ليرى المستخدم ما سيُستورَد.
2. **ربط الأعمدة (Column Mapping):** واجهة تربط **أعمدة النظام** بـ**أعمدة الملف المرفق** يدويًا — لا يُفترَض تطابق الأسماء أو الترتيب. المستخدم يختار: «عمود النظام X ← عمود الملف Y».

النموذج المرجعي: تجربة الاستيراد في **نظام دفترة (Daftra)** وما يماثله — رفع الملف، ثم شاشة ربط أعمدة، ثم معاينة، ثم تأكيد.

**الغاية:** جعل الاستيراد أسهل وأكثر عملية — يقبل ملفات بأي ترتيب/تسمية أعمدة دون أن يُجبَر المستخدم على إعادة تهيئة ملفه ليطابق قالبًا صارمًا.

## 2. النطاق

تنطبق القاعدة على **أي مسار فيه استيراد**، ومنها الموجود حاليًا:

- استيراد العمرة — `import-wizard.tsx` / `umrahImportEngine` (استيراد ملفات NUSK).
- الاستيراد العام لكيانات ERP — `genericImportEngine` / `routes/import.ts`.
- أي استيراد يُضاف مستقبلًا (موظفون، عملاء، منتجات…).

## 3. حالة التنفيذ

**مؤجَّل.** لا يدخل ضمن موجة إصلاح العيوب الـ20 الحاجبة الجارية. يُعامَل كبند تطوير منفصل (تحسين تجربة، لا إصلاح عيب) يُجدوَل لاحقًا.

## 4. ملاحظة للتنفيذ لاحقًا

عند جدولته: يُفضَّل مكوّن واجهة موحَّد لربط الأعمدة + المعاينة يُعاد استخدامه عبر كل مسارات الاستيراد (لا تكرار لكل مسار) — اتّساقًا مع ملاحظات التحليل العرضي حول تجنّب تكرار الأنماط. الواجهة الخلفية لـ`import-wizard` (التي تدعم خطوة معاينة جزئيًا) نقطة بداية.
