# مخطط حزمة المشاريع — نظام غيث (Projects Package Blueprint)

> **رجعيّ — مُوثَّق مما هو مبنيّ ومدموج في `main` ومُتحقَّق حيًّا** (#1594). يحدّد:
> نموذج البيانات، السلسلة التشغيلية، عقود القائد/الخادم، وملكية المهام.
> القاعدة: المشاريع **قائد في التكلفة والمراحل والتطوير**؛ المالية **خادم** عبر
> المحرّك والأحداث (لا يكتب مسار المشروع فاتورةً أو قيدًا مباشرةً).

---

## ١) نموذج البيانات (جداول مجال المشاريع)

| الجدول | الدور |
|--------|-------|
| `projects` | رأس المشروع: عميل، **budget**، spentAmount، managerId، الحالة، النوع |
| `project_phases` | المراحل (دورة حياة: pending→in_progress→completed) — تحمل `companyId`/`updatedAt` لعقد lifecycleEngine (migration 290) |
| `project_tasks` | مهام المشروع (assigneeId، phaseId، priority، status، ساعات مقدّرة/فعلية) |
| `project_task_dependencies` | اعتماديات المهام (المسار الحرج) |
| `project_milestones` | المعالم/المستخلصات |
| `project_costs` | التكاليف المتجمّعة (projectId، amount، category) → تغذّي WIP والربحية |
| `project_boq_items` | بنود الفوترة (BOQ): `aggregate` إجمالي بالمتر / `custom` مخصّص — كمية×سعر (migration 287) |
| `development_units` | وحدات التطوير الداخلي للبيع — أساس تكلفة بحصة المساحة (migration 288) |
| `project_risks` / `project_resources` | المخاطر والموارد |

**ربط المالية (بلا تكرار ماستر):** كل مشروع يحصل تلقائيًّا على **مركز تكلفة**
(`cost_centers.linkedEntityType='project'`) **متداخل تحت مركز تكلفة الفرع**، ويحمل
**ميزانية المشروع كتخصيص** (`allocatedAmount`) — فتقارير الانحراف (مخصّص مقابل
مستخدَم) تعمل، وتتزامن عند تعديل الميزانية.

## ٢) السلسلة التشغيلية (مُثبتة حيًّا — `scripts/verify-projects-journey.sh`، 18/18)

```
مشروع (عميل + ميزانية)
  → مركز تكلفة تلقائي يحمل الميزانية كتخصيص            [variance-ready]
  → تكلفة مشروع   → قيد WIP (مدين project_wip / دائن نقد)        [متوازن]
  → تعديل الميزانية → إعادة مزامنة تخصيص مركز التكلفة
  → بنود BOQ      → فوترة → فاتورة عميل واحدة، سطر لكل بند، البنود billed
  → وحدة تطوير    → بيع → قيد WIP→COGS [متوازن] + فاتورة بيع، ربح = سعر − أساس تكلفة
  → مرحلة         → in_progress → complete → فاتورة مستخلص (ميزانية × وزن المرحلة)
```

كل أثر محاسبي **متوازن**؛ منع الفوترة/البيع المزدوج بمطالبة ذرّية؛ بيع الوحدة
ذرّي (فشل القيد يُرجع الوسم).

**مفاتيح GL (عبر `accounting_mappings` — قرار مالية لا تخمين، migration 289):**
`project_wip` · `project_cost_cash` · `project_cost_transfer` · `dev_unit_cogs`.

## ٣) عقود القائد/الخادم (لا نسخ منطق المالية داخل المسار)

| المجال | الدور تجاه المشاريع |
|--------|---------------------|
| **المشاريع** | **قائد** في: التكلفة (WIP)، المراحل/المهام/المسار الحرج، وحدات التطوير، بنود BOQ |
| **المالية** | **خادم**: كل فاتورة عبر حدث `project.invoice.requested` (معالِج `eventListeners` ينشئ الفاتورة + الأسطر)؛ كل قيد عبر `projectsEngine` → `financialEngine.postJournalEntry`. **مسار المشروع لا يكتب `invoices`/`journal_*` مباشرةً** (يحرسه `audit:domain-boundaries`). |
| **CRM** | العميل (clientId) — لا اسم نصّي |
| **HR** | الفريق = موظفون (`project_tasks.assigneeId`) |
| **العقارات** | وحدات التطوير منفصلة عن `property_units` الإيجارية (نموذج للبيع بأساس تكلفة) |

## ٤) ملكية المهام — العقد (لا تداخل صلاحيات)

النظام يفصل **نوعين** من المهام بوضوح على مستوى البيانات والمسار والصلاحية:

| | العمل العام | مهام المشروع |
|---|---|---|
| **الجدول** | `tasks` | `project_tasks` |
| **المسار الخادم** | `routes/tasks.ts` | داخل `routes/projects.ts` (`POST /projects/:id/tasks`، `PATCH /projects/tasks/:taskId`) |
| **صلاحية RBAC** | `feature: "tasks"` | `feature: "projects.tasks"` |
| **module** | operations | projects |
| **المُنشِئ التلقائي** | `rulesEngine`/`proactiveEngine`/legal/properties (متابعات) | المستخدم داخل صفحة المشروع |

**القائد:** العمليات (Operations) تملك `tasks`؛ المشاريع تملك `project_tasks`.
لا تداخل: جدولان، مساران، خاصيّتا RBAC منفصلتان.

### ✅ تصحيح تسمية الواجهة (تمّ — كان `/projects/tasks` يضلّل)
سابقًا: مسار الصفحة `/projects/tasks` + عنصر القائمة **«مهام المشاريع»** + تبويب
**«المهام»** في تبويبات المشاريع كانت **تعرض صفحة العمل العام** (`pages/tasks.tsx`
→ جدول `tasks`)، بينما مهام المشروع الحقيقية تُدار داخل `project-detail.tsx` عبر
واجهة `/projects/tasks/:taskId` الصحيحة (جدول `project_tasks`).

**التصحيح المُطبَّق (تسموي/تنقّلي فقط — البيانات والصلاحيات كانت سليمة):** أُزيلت
المؤشّرات المضلّلة الثلاثة من نطاق المشاريع: مسار الصفحة `/projects/tasks`
(`miscRoutes.tsx`)، وعنصر «مهام المشاريع» (`navigation.registry.ts`)، وتبويب
«المهام» (`projects-tabs-nav.tsx`). **صفحة العمل العام تبقى على `/tasks`** (module
operations)، ومهام المشروع تبقى في صفحة المشروع. لا فقدان وظيفة.

## ٥) قرار المالك المُطبَّق

**الفوترة بالبند (BOQ) لا T&M بالساعات** — كما قرّر المالك. لا فوترة بالساعات
في النظام؛ الفوترة عبر بنود (كمية×سعر) + مستخلصات المعالم + بيع وحدات التطوير.

### ✅ معيار #1959 (نجاح بحساب غير-admin) — مُحقَّق + إصلاح جذري
الرحلة الآن **تُهيّئ ذاتيًّا حساب `projects_manager` (non-admin) وتعمل به**، مُثبتةً
أنّ كامل السلسلة — حتى الترحيلات المحاسبية الخادمية (تكلفة→WIP، بيع→COGS، فوترة
المستخلص) — تنجح لدور غير المالك، لا للمالك فقط.

أثناء هذا التحقّق ظهر **عطل #1959 جذريّ على مستوى التطبيق**: في `routes/index.ts`
كانت موجِّهات النقل السبع مُركَّبة **بلا مسار** خلف `requireModule("fleet")`
(`router.use(requireModule("fleet"), …, transportRouter)`)، فالحارس بلا مسار يعمل
لكل طلب لاحق ويرفض **كل مستخدم غير-owner** من **كل** قسم مُركَّب بعده
(المشاريع/CRM/القانونية/العقارات/…) — المالك يتجاوز `requireModule` فيمرّ، فيبدو
«admin ينجح، non-admin يفشل». الإصلاح: حارس **مشروط بالمسار بلا تركيب-مسار**
(`transportPathGate`) يحرس `/transport`+`/fleet` فقط دون تجريد البادئة (فـ
`requireGuards` يقرأ `req.path` الحقيقي)؛ كل مسارات النقل السبعة تحت هاتين
البادئتين (مُتحقَّق: 56 `/transport` + 14 `/fleet`، صفر غيرهما). مُثبت حيًّا:
non-admin يصل للمشاريع الآن، ومستخدم بلا fleet ما زال يُرفض على `/transport`،
والمالك يصل للنقل (لا انحدار).
