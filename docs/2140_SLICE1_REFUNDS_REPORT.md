# تقرير الشريحة 1 — طلبات استرداد العمرة (#2170، دُمجت `ca548fad`)

_وفق قالب `docs/2140_SLICE_PROGRAM.md` — أُعد بأثر رجعي بعد اعتماد البروتوكول._

## 1. الخدمة التي كانت ناقصة

دورة استرداد مالية كاملة للمعتمر أو الوكيل (إلغاء رحلة، ازدواج تحصيل…)
بُنيت خلفياً بآلة حالات وتدقيق وأحداث (`umrahRefundWorkflow.ts`) **ولم تكن
لها أي واجهة** — المشغّل لا يستطيع تقديم طلب ولا اعتماده ولا صرفه إلا بأدوات
تقنية. أعلى بند في تصنيف «الخدمات الناقصة».

## 2. الـAPIs المغطاة (6/6)

| الفعل | المسار | الاستخدام في الواجهة |
|---|---|---|
| GET | `/umrah/refund-requests` | القائمة (بأسماء المعتمر/الوكيل والجواز) |
| POST | `/umrah/refund-requests` | نموذج التقديم السريع |
| POST | `…/:id/approve` | زر «موافقة» |
| POST | `…/:id/reject` | حوار «رفض» بسبب إلزامي |
| POST | `…/:id/pay` | حوار «صرف»: مبلغ تسوية + خزينة + مرجع |
| POST | `…/:id/close` | زر «إغلاق» |

## 3. آلة الحالات والانتقالات

```
مقدَّم (requested) → موافَق عليه | مرفوض | ملغى
موافَق عليه (approved) → مدفوع | ملغى
مدفوع (paid) → مغلق
مرفوض / مغلق / ملغى → نهائي
```
أزرار الصف تُرسم من مرآة أمامية (`UMRAH_REFUND_NEXT`) **واختبار التثبيت
يقارنها بجدول الخادم `REFUND_TRANSITIONS` حالةً بحالة** فلا انجراف صامت.
الخادم يعيد التحقق على كل POST عبر `canTransition` — الواجهة مجاملة UX لا بوابة.

## 4. الرسائل العربية الظاهرة (عينة محققة حياً)

- نجاح: «قُدِّم طلب الاسترداد» · «تمت الموافقة…» · «صُرف مبلغ الاسترداد» · «أُغلق…» · «رُفض…»
- رفض منضبط من الخادم: «لا يمكن صرف طلب بحالة "requested"» (CONFLICT) ·
  «لا يمكن إغلاق طلب بحالة "closed"» · «المعتمر غير موجود في النظام»
- تحقق مدخلات الواجهة قبل الإرسال: «أدخل رقم المعتمر أو رقم الوكيل» ·
  «سبب الرفض مطلوب» · «اختر حساب الخزينة» · «مرجع الدفع مطلوب»

## 6. الأثر على قاعدة البيانات (محقق بـpsql)

- صف في `umrah_refund_requests` لكل طلب؛ الانتقالات تحدّث
  status/approvedBy/rejectedBy+reason/settledAmount/treasuryId/paymentReference/paidAt.
- **سجل تدقيق**: صف `audit_logs` (entity=umrah_refund_requests) عند الإنشاء — محقق.
- **أحداث**: `umrah.refund.requested/approved/rejected/paid/closed` تُبث لكل انتقال.
- **القيود المحاسبية**: خطوة الصرف **لا تولّد قيد GL حالياً** — لا ترحيل داخل
  معالج العمرة (وهذا صحيح معمارياً: العمرة قائد قرار الاسترداد) ولا يوجد بعدُ
  مستهلك مالي للحدث يرحّل القيد. **فجوة موثقة تحتاج قرار منتج/مالية**: بناء
  خدمة مالية تستهلك `umrah.refund.paid` وترحّل (دائن خزينة / مدين ذمم الوكيل
  أو حساب الاستردادات) وفق عقد خدمة — لم تُبنَ داخل هذه الشريحة عمداً التزاماً
  بحدّ «أي أثر مالي لاحق يبقى خدمة من المالية».

## 7. أدلة التشغيل الحي (تنصيب نظيف، عبر HTTP)

| خطوة | النتيجة |
|---|---|
| تقديم (معتمر + 1500 − استقطاع 100) | `id=1 status=requested` |
| صرف قبل الموافقة | **رُفض**: CONFLICT «لا يمكن صرف طلب بحالة "requested"» |
| موافقة → صرف (خزينة حقيقية + «حوالة-7791») → إغلاق | `ok` × 3 |
| إغلاق مكرر | **رُفض**: «لا يمكن إغلاق طلب بحالة "closed"» |
| رفض بلا سبب | **رُفض**: VALIDATION |
| رفض بسبب «المبلغ سُدد سابقاً» | `ok` |
| القائمة النهائية | `#1 … closed (مصروف=1400.00، مرجع=حوالة-7791)` · `#2 … rejected` |

**العزل (محقق حياً)**: جلسة شركة 1 ترى **0 صف** من استردادات شركة 2؛ محاولة
`POST …/1/approve` عابرة للشركات → **404 «طلب الاسترداد غير موجود»** (لا
تسريب وجود). **الصلاحيات**: لم يتوفر مستخدم مبذور بلا صلاحية عمرة في بيئة
التحقق لاختبار سلبي حي؛ البوابة `authorize()` هي وسيطة الترخيص الموحدة
المغطاة باختبارات الوحدة القائمة، وبوابة الواجهة `GuardedButton` تخفي
الأزرار لمن لا يملك الصلاحية — يُستكمل الاختبار السلبي الحي ضمن مراجعة
العمرة #2141 حيث سيُجهز مستخدم محدود الدور.

## 8. ما لم يُعمل عمداً ولماذا

| البند | السبب | القرار لمن |
|---|---|---|
| ترحيل GL للصرف | حدّ معماري: خدمة مالية مستقلة تستهلك الحدث بعقد | منتج + مالية |
| حالة «ملغى» (cancelled) كزر | لا endpoint إلغاء في الخادم (الحالة معرفة في الآلة فقط) | خلفية أولاً |
| ربط salesInvoiceId/nuskInvoiceId في النموذج | يتطلب منتقي فواتير؛ الحقول اختيارية في العقد ولم تُعرَض لتجنب نموذج مزدحم — تُضاف عند الحاجة التشغيلية | منتج |
| صفحة تفاصيل مستقلة للطلب | الدورة كلها أزرار صف + حوارات تكفيها القائمة؛ تفاصيل أعمق ضمن #2141 | — |
