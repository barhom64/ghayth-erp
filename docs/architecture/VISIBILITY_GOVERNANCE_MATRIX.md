# مصفوفة حوكمة الظهور — VISIBILITY_GOVERNANCE_MATRIX

> المرحلة 2 — **Ghaith Operating Foundation** (#1418) · 2026-05-29 · فرع `claude/ghaith-foundation-audit-wdIUf`
> **المبدأ:** المستخدم يرى ما يخصه فقط. الظهور دالة في (الدور + النطاق + التفعيل + حالة السجل).
> **يُبنى على:** `PAGE_VISIBILITY_INVENTORY.md` + `MENU_GOVERNANCE.md` (المرحلة 1).

---

## 1. معادلة الظهور المعتمدة

```
صفحة تَظهر ⇔ مسجَّلة(registry) ∧ مُفعَّلة(feature/subscription) ∧ مخوّلة(feature.view ضمن النطاق) ∧ ناضجة(أو مخوّل للتجريبي)
زر يَظهر  ⇔ مخوّل(feature.action ضمن النطاق) ∧ حالة السجل تسمح بالإجراء
```

الطبقات الموجودة اليوم: `canAccessModule` + `canAccessSubPage` + `minRoleLevel` (+ `registry`). الطبقتان الناقصتان: **التفعيل** (VIS-002) و**النضج/حالة السجل** (VIS-004/005).

---

## 2. مصفوفة "من يرى ماذا" (نماذج الأدوار من #1413)

| الدور | يرى | لا يرى |
|---|---|---|
| موظف عادي | بوابة الموظف (طلباتي/معلوماتي)، مهامه | المالية، العمرة، النقليات، الإعدادات، الإدارة |
| محاسب (إنشاء) | المالية ضمن صلاحيته فقط | اعتماد ما لا يملكه، HR، إعدادات النظام |
| مدير فرع مكة | بيانات فرع مكة فقط | جدة، الرياض (نطاق branch) |
| مسؤول العمرة | العمرة وخدماتها فقط | إعدادات مالية متقدمة (إلا بدور مالي) |
| مدير مالي (شركة) | كل المالية | تعديل سياسات HR |
| مدير HR | كل HR ضمن النطاق | المالية المتقدمة |
| مشاهد تقارير | تقارير ضمن صلاحيته (view فقط) | أي إجراء create/approve |

> الأساس موجود: `canAccessModule` يطابق المالك (`ENTITY_OWNERSHIP_MATRIX`)، و`scope` في RBAC يقيّد الفرع/الإدارة.

---

## 3. مصفوفة حوكمة الظهور (الطبقة × الموجود × الفجوة)

| طبقة الظهور | الموجود | الفجوة | القرار |
|---|---|---|---|
| المالك (module) | `canAccessModule` | umrah بلا بوابة (VIS-001) | يُطبَّق على umrah/misc |
| النطاق (scope) | `scope` في RBAC + `buildScopeFilter` | تبنّي scoping غير موحّد (FND-013) | يُعمَّم `buildScopedWhere` |
| الميزة الدقيقة | `canAccessSubPage` (خرائط ثابتة) | لا يُشتق من featureCatalog (VIS-003/MENU-005) | يُطوَّر للاشتقاق |
| التفعيل/الاشتراك | **لا شيء** | VIS-002/MENU-003 | **يُبنى** (`VISIBILITY_ENGINE_SPEC`, `SERVICE_MODE_VISIBILITY`) |
| النضج | **لا شيء** | VIS-004 | يُبنى واسم بسيط |
| حالة السجل (أزرار) | يدوي مبعثر | VIS-005 | يُوحَّد (`ACTION_VISIBILITY_RULES`) |

---

## 4. قواعد الحوكمة

1. **افتراض الإغلاق:** الصفحة/العنصر مخفي حتى يثبت التخويل (module + feature + نطاق).
2. **الخادم داخل السياق:** الخدمة المساندة لا تظهر كقسم مستقل لغير المخوّل (`SERVICE_MODE_VISIBILITY`).
3. **لا ناقص في الإنتاج:** الصفحات غير الجاهزة تُخفى بواسم نضج، لا تُعرض ثم تكسر.
4. **الأزرار حسب الصلاحية + الحالة:** زر الاعتماد يظهر فقط لمن يملك `approve` وعندما الحالة `submitted`.
5. **التوجيه = القائمة:** كلاهما يستهلك نفس دوال `AppContext` (مؤكَّد متّسق في المرحلة 1).

## 5. القرارات

- **المصفوفة تُعتمَد** كعقد ظهور؛ تنفيذها يبدأ بإصلاح VIS-001 (umrah) كأولوية، ثم بناء طبقة التفعيل.
- **التفصيل الهندسي** في مرحلة 4: `VISIBILITY_ENGINE_SPEC`, `NAVIGATION_CONTEXT_CONTRACT`, `ACTION_VISIBILITY_RULES`, `SERVICE_MODE_VISIBILITY`.
