# تطور نموذج الأطراف — PARTY_MODEL_EVOLUTION

> المرحلة 6 — **Ghaith Operating Foundation** (#1418) · 2026-05-29 · فرع `claude/ghaith-foundation-audit-wdIUf`
> **المبدأ:** بناء اليوم بطريقة لا تكسر الغد. توحيد "الطرف" (شخص/منظمة) مستقبلًا دون كسر الموجود.

---

## 1. الوضع الحالي

أطراف غيث اليوم منفصلة: `employees`، `clients`، `suppliers`، `umrah_agents`، `property_owners`، `property_tenants`. كل منها جدول مستقل بحقول متشابهة (اسم/جوال/هوية/IBAN).

## 2. الخطر المستقبلي

نفس الشخص قد يكون موظفًا وعميلًا ومالك عقار → تكرار بيانات وتعارض. أي توسّع يضاعف التكرار.

## 3. مسار التطور (دون كسر)

```
المرحلة الحالية: جداول منفصلة (تبقى كما هي)
        ▼
المرحلة 1 (غير كاسرة): جدول party موحّد اختياري + party_id على الجداول الموجودة (nullable)
        ▼
المرحلة 2: الأدوار التجارية (party_role: employee/client/supplier/owner/agent) تشير لـ party واحد
        ▼
المرحلة 3: توحيد الحقول المشتركة (هوية/جوال/IBAN) في party، مع إبقاء الخاص بكل جدول
```

## 4. القواعد

1. **لا كسر:** الجداول الحالية تبقى؛ `party_id` يُضاف nullable ويُملأ تدريجيًا.
2. **الحقول الحساسة** (هوية/IBAN) تخضع لسياسة حقلية موحّدة (`rbac_field_policies`).
3. **التوحيد اختياري** — لا يُفرض دفعة واحدة.

## 5. القرار

- **لا تنفيذ الآن** — هذا مسار تطور موثّق. أي ميزة جديدة تتعامل مع "طرف" تراعي هذا الاتجاه (تضيف `party_id` إن أمكن) لتجنّب دين تقني.
