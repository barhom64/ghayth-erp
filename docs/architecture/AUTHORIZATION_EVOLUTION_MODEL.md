# تطور نموذج التفويض — AUTHORIZATION_EVOLUTION_MODEL

> المرحلة 6 — **Ghaith Operating Foundation** (#1418) · 2026-05-29 · فرع `claude/ghaith-foundation-audit-wdIUf`
> **المبدأ:** يتحمّل النظام أدوارًا متعددة وصلاحيات مؤقتة/موسمية وتفويضًا دون كسر RBAC v2.

---

## 1. الوضع الحالي (قوي)

RBAC v2 يدعم: تعدد الأدوار، النطاق، الشروط (ABAC)، حدود الاعتماد، الحقول الحساسة، فصل المهام، **صلاحيات مؤقتة** (`rbac_user_grants.expires_at`)، شجرة أدوار (وراثة).

## 2. ما يحتاج تطورًا

| الحاجة | الموجود | التطور |
|---|---|---|
| صلاحيات مؤقتة | `rbac_user_grants.expires_at` ✅ | تُفعَّل في الواجهة (منح بانتهاء) |
| صلاحيات موسمية (عمرة) | `conditions` (season) ✅ | شرط موسم في الواجهة |
| التفويض (delegation) | جزئي (`/hr/delegations`) | نموذج تفويض موحّد: أ يفوّض ب لمدة، بنطاق محدد، مع تدقيق |
| الدور النشط في التدقيق | مفقود ❌ | RBAC-001 (أولوية) |
| توحيد الكتالوج | مكرر | RBAC-003 |

## 3. نموذج التفويض المقترح (دون كسر)

```
delegation (from_user, to_user, scope, feature_set, from_date, to_date, reason)
   → يُترجَم إلى rbac_user_grants مؤقتة على المفوَّض إليه (expires_at = to_date)
   → كل فعل بالتفويض يُسجَّل: "نُفّذ بتفويض من X"
```

يستفيد من البنية الموجودة (`rbac_user_grants`) — لا جدول صلاحيات جديد.

## 4. القواعد

1. **التفويض مؤقت ومُدقَّق** — ينتهي تلقائيًا، ويُسجَّل المفوِّض والمفوَّض إليه.
2. **لا يتجاوز التفويض صلاحية المفوِّض** — لا يمنح ما لا يملك.
3. **يُبنى فوق `rbac_user_grants`** لا نظام موازٍ.

## 5. القرار

- **يُطوَّر التفويض الموحّد** فوق `rbac_user_grants` + تفعيل المؤقت/الموسمي في الواجهة — مسار تطور بعد المراحل الحرجة.
