# تطور سياسة الاعتماد — APPROVAL_POLICY_EVOLUTION

> المرحلة 6 — **Ghaith Operating Foundation** (#1418) · 2026-05-29 · فرع `claude/ghaith-foundation-audit-wdIUf`
> **المبدأ:** يتحمّل النظام اعتمادًا متعدد المستويات وحدودًا وتصعيدًا دون كسر مركز القرار.

---

## 1. الوضع الحالي

`approval_chains` + `approval_chain_steps` (بسيط/متعدد المراحل/توقيع) + `rbac_approval_limits` (حد المبلغ + اعتماد ثنائي) + `rbac_sod_rules` (فصل المهام). مركز القرار `approval_actions` موحّد.

## 2. ما يحتاج تطورًا

| الحاجة | الموجود | التطور |
|---|---|---|
| اعتماد متعدد المستويات | `approval_chain_steps` ✅ | واجهة بناء السلاسل لغير التقني |
| حدود ديناميكية بالمبلغ | `rbac_approval_limits` ✅ | توجيه السلسلة حسب المبلغ (مبلغ أكبر → مستوى أعلى) |
| اعتماد ثنائي (dual control) | `requires_dual_control` ✅ | تفعيله في التدفّق |
| تصعيد عند التأخّر | جزئي (SLA دعم) | تعميم (`SLA_ESCALATION_MODEL`) |
| تفويض الاعتماد | جزئي | `AUTHORIZATION_EVOLUTION_MODEL` |

## 3. نموذج التطور

```
طلب اعتماد → السلسلة تُختار حسب (الميزة + المبلغ + النطاق)
   → المستوى 1 (حد منخفض) → المستوى 2 (حد أعلى) → ... 
   → تجاوز الموعد في مستوى → تصعيد تلقائي للمستوى التالي (SLA)
   → كل قرار يُسجَّل في approval_actions بالصفة
```

## 4. القواعد

1. **مركز واحد:** كل المستويات عبر `approval_chains` + `approval_actions` — لا اعتماد لكل مسار.
2. **الحد يوجّه المستوى** — لا كود مبلغ مُرمَّز في كل مسار.
3. **التصعيد عبر SLA الموحّد**، لا منطق تصعيد لكل مسار.

## 5. القرار

- **يُطوَّر** توجيه السلاسل بالمبلغ + تفعيل الاعتماد الثنائي + واجهة بناء السلاسل — فوق الموجود، مسار تطور.
