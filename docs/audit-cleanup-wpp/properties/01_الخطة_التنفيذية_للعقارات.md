# 01 — الخطة التنفيذية لمسار العقارات

> **النطاق:** العقارات فقط. لا يشمل المستودع ولا المشاريع (وكلاء مستقلون لكلٍّ منهما). لا يشمل بقية المسارات إلا بقدر التكامل المحاسبي المباشر (المحرّك المالي خادم؛ العقارات قائد في العقد والمالك والوحدة).
> **المرجع:** خطة نظام غيث · جرد العقارات الموحّد · حالة `main` الفعلية بعد الموجة المنجزة · فلسفة المحرّكات المركزية (`propertiesEngine` / `financialEngine` / `lifecycleEngine` / `partyService`).

---

## 1) العقيدة الحاكمة لمسار العقارات

أربعة أشقاق نشاط، محورا مصدر العقد، مالية خادم، حالة وحدة عبر `lifecycleEngine`، أطراف عبر `partyService`:

| المحور | المضمون |
|---|---|
| **النشاط** | إيجار سكني / إيجار تجاري (ضريبة 15%) / بيع / إدارة بعمولة |
| **مصدر العقد** | إيجار (قراءة عبر `EjarContractReader`) أو يدوي |
| **القيد** | عبر `propertiesEngine` فقط → `financialEngine.postJournalEntry`. **ممنوع** كتابة `journal_entries` من شاشة عقار. |
| **الحسابات** | تُحلّ عبر `accounting_mappings` (قابل للتحكم بدون كود). الـfallbacks في المحرّك للسلامة الانتقالية فقط. |
| **الأبعاد** | `propertyId / contractId / clientId / unitId` على كل سطر قيد. |
| **حالة الوحدة** | `STATE_MACHINES.property_units` في `lifecycleEngine.ts` مصدراً واحداً. الـPATCH يشتقّ منه. |
| **الأطراف** | `partyService.registerEntityParty` عند إنشاء مالك/مستأجر. لا تكرار ماستر. |

## 5) قواعد القبول لكل موجة

- **بنية أساسية:** PR صغير، فرع منفصل، عقود قائد/خادم محترمة.
- **القيد:** اختبار يُقفل سطور القيد فعلياً (`vi.hoisted` يلتقط `lines[]`)، DR=CR، حسابات عبر `resolveAccountCode`.
- **التحقّق الحيّ:** كل b-part يمدّد `verify-property-*-journey.sh` ويعمل ضد `db:provision-agent` بـnon-admin (door@door.sa).
- **الترحيل:** هيكل `MIGRATION_POLICY` (`@rollback:`, idempotent, additive افتراضياً).
- **القابلية للتحكم:** كل ثابت (معدّل، حساب، نسبة عمولة) من إعدادات قابلة للعدّ، لا مغروس في الكود.
- **guard:** أخضر بدون override.
