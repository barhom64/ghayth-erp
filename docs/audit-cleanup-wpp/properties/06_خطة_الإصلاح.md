# 06 — خطة الإصلاح لمسار العقارات

> كل بند: المشكلة (مشاهدة فعلية) → السبب → الإصلاح (PR صغير) → معيار القبول → الحالة.

## P0 — قبض إيجار مكرّر (P0-1)

**المشكلة:** مساران مستقلان لتسجيل دفعة إيجار يصدران قيداً مختلفاً.

- `POST /payments/:id/pay` → جدول `rent_payments` → `propertiesEngine.postRentRevenueGL` (DR 1131 ذمم / CR 4121 إيراد) — **بدون سند قبض**.
- `POST /contracts/:id/schedule/:installmentId/pay` → جدول `contract_payment_schedule` → `postInstallmentPaymentGL` (DR 1111 نقد / CR 4121 إيراد) — **مع `issueNumber("lease_receipt")`**.

**السبب:** بناء متراكم بدون توحيد. الأول حدث (accrual)، الثاني نقد (cash basis) — نتيجتان محاسبيتان مختلفتان لنفس الفعل.

**الإصلاح:**
1. اعتماد مسار الذمم (accrual): استحقاق → ذمم مدينة → عند الدفع نقد + إغلاق ذمم + سند قبض.
2. حذف `postInstallmentPaymentGL` أو إعادة استخدامه فقط لجدولة بـ`contract_payment_schedule` بدون قيد منفصل.
3. توحيد `rent_payments` و`contract_payment_schedule`: الأول للحركات النقدية، الثاني للجدولة فقط (no GL).
4. إصدار سند قبض في كلا المساران عبر `issueNumber("properties","lease_receipt")`.

**معيار القبول:** رحلة E2E تخلق جدولة → تدفع → تجد قيداً واحداً متوازناً 1131→4121 + سند قبض مرقّم + `rent_payments.receiptNumber` غير NULL.

**يتقاطع مع:** #2039 — انتظار دمجه ثم العمل.

**الحالة:** ❌ لم يبدأ.

## P0 — b-parts (الراوتر + الواجهة + الـmigrations)

**المشكلة:** المحرّكات الأربعة (VAT تجاري، بيع، إدارة، صيانة على المالك) موجودة في المحرّك ومُقفلة بعقد القيد، لكن **لا راوتر يستدعيها بعد** للأشقاق الجديدة.

**الإصلاح (بعد دمج الموجة):**

| PR | المحتوى | الترحيل المتوقّع |
|---|---|---|
| PR-4b | تجاري VAT: لا راوتر جديد، الراوتر القائم في `/payments/:id/pay` يقرأ `contractType` ويستدعي VAT split. رحلة E2E تجاري تؤكّد سطر 2131. | — |
| PR-5b | البيع: جدول `property_sales` + `POST /sales` + واجهة `sale-create.tsx` + رحلة E2E (DR 1132 / CR 1240 / CR/DR مكسب/خسارة). | جديد |
| PR-6b | الإدارة: عمود `rental_contracts.commissionPct` + راوتر تحصيل + كشف المالك (يقرأ من `journal_lines` لا جدول منفصل) + رحلة E2E. | جديد |
| PR-7b | الصيانة: قرار التوجيه يقرأ `property_buildings.ownerType` → دالة المحرّك الصحيحة + رحلة E2E تجاري + فاتورة ضريبية للمالك. | — |

**معيار القبول:** كل b-part يمدّد `verify-property-*-journey.sh` ضدّ `db:provision-agent`، يقرأ `journal_lines` فعلياً، يؤكّد:
- DR=CR
- الحسابات الصحيحة عبر mappings
- الأبعاد كاملة
- idempotency (ref + sourceKey فريدان لكل اتجاه)

**الحالة:** ❌ لم تبدأ (تنتظر دمج الموجة).

## P0 — تكامل المالك مع PartyService (P0-2)

**المشكلة:** قبل #2061 لم يكن أي owner/tenant مُنشأ من الراوتر يُربط بـ`parties` master. الـbackfill الدوري كان يلتقطهم، لكن البيانات تبقى مفرّقة بين تشغيلتَي backfill.

**الإصلاح:** #2061 — `POST /properties/owners` و`POST /properties/tenants` يستدعيان `registerEntityParty` بـfire-and-forget بعد INSERT.

**تحقّق حيّ:** ربط فوري، idempotent، dedup عبر nationalId يربط نفس الطرف بدوريّ owner + tenant معاً.

**الحالة:** ✅ منجز (#2061).

## P0 — ودائع الضمان في كشف العميل (P0-3)

**المشكلة:** `postSecurityDepositGL` يكتب القيد صحيحاً (DR cash / CR liability 2170)، لكن كشف العميل لا يعرض الوديعة. مشغّل يسوّي مستأجر متروك يرى كشفاً نظيفاً والشركة تحتجز آلاف الريالات.

**الإصلاح:** #2055 — كتلة `securityDeposits` منفصلة في استجابة `customer-statement`، لا تُدخل في حركات AR (لأنها التزام لا ذمم).

**تحقّق حيّ:** seeded client → tenant → contract → deposit (held + refunded) → الكشف يعرض المحجوزة فقط، يستبعد المُعادة.

**الحالة:** ✅ منجز (#2055).

## P0 — عيب refund ref collision + 5 مخصصات حسابات ناقصة

**المشكلة (مكتشفة من رحلة E2E حيّة):**
1. `postSecurityDepositGL` يستخدم `ref=JE-DEP-${id}` لكلا الاستلام والاسترداد → ثاني نداء يفجّر `uniq_journal_entries_ref` → 502.
2. خمس operationTypes (`property_maintenance_expense / _payable, vat_output, property_building_asset, property_building_purchase_cash`) بـfallbacks (6400/2100/2200/1520/1100) **غير موجودة كحسابات postable في COA الديار**. الرحلة تتعطّل عند الصيانة بـValidationError "الحساب غير موجود".

**الإصلاح:** #2072 —
1. `ref` أُلحق به `R` للاستلام / `F` للاسترداد.
2. ترحيل 300 يبذر المخصصات الخمس (5610 / 2150 / 2131 / 1240 / 1111) — idempotent، يلتزم بـCOA postable.

**تحقّق حيّ:** الرحلة تصل لـ18/18 نجاح.

**الحالة:** ✅ منجز (#2072).

## P1 — بوابات المالك والمستأجر

**المشكلة:** لا واجهة مخصّصة. المالك يرى كل شيء كأنه مشغّل، أو لا يرى شيئاً.

**الإصلاح المقترح:**
- `permission-gate` بـrole=owner يقصر العرض على عقاراته فقط (filter بـ`property_owners.id = scope.linkedOwnerId`).
- صفحة `/portal/owner` تعرض: قائمة عقاراته + كشف عمولتنا (يُحسب من `journal_lines` accountCode='2150' clientId=ownerId) + الدفعات.
- بوابة `/portal/tenant`: عقده، دفعاته، بلاغ صيانة 12 خطوة.

**معيار القبول:** non-admin login بحساب مالك يرى عقاراته فقط، non-admin login بحساب مستأجر يرى عقده فقط؛ كلاهما يقدّم بلاغ/طلب.

**الحالة:** ❌ لم يبدأ.

## P1 — بلاغ صيانة 12 خطوة

**المشكلة:** المستأجر اليوم لا يفتح بلاغاً صراحة — البلاغ يفتحه المشغّل بعد اتصال.

**الإصلاح:** نموذج 12 خطوة (الفئة، الوصف، الموقع، تاريخ بدء المشكلة، الصور قبل، الأولوية، التوقيت المفضّل للزيارة، …) → `POST /maintenance-requests` بـ`source='tenant_portal'`.

**معيار القبول:** بلاغ يصل ويظهر للمشغّل بـSLA حسب الأولوية، والمستأجر يتلقّى رقم البلاغ في إشعار.

**الحالة:** ❌ لم يبدأ.

## P1 — تصعيد الإيجار المتأخر للقانونية

**المشكلة:** `POST /late-rent/escalate` يصعّد محلياً بـ6 مراحل، لكن **لا تكامل صريح** مع `legal_contracts` في المرحلة السادسة.

**الإصلاح:** المرحلة 6 تُطلق `requestLegalCaseCreation` (موجود في `propertiesEngine`) → ينشئ قضية إخلاء في وحدة القانونية بـrefType='rental_contract'.

**معيار القبول:** عقد متأخر ≥90 يوماً → bot الجدولة يطلق التصعيد → قضية قانونية تظهر في `/legal/cases` بمرجع للعقد.

**الحالة:** جزئي — التكامل موجود في المحرّك لكن غير مُستدعى.
