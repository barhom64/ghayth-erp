# 13 — المخططات التسلسلية للأشقاق الأربعة (ملحق لـ#2088)

> أربعة فروع نشاط، كل واحد عملية متكاملة UI → Route → Engine → Financial Engine → GL. المخططات بصيغة [Mermaid](https://mermaid.js.org/) — يعرضها GitHub تلقائياً في معاينة الـMarkdown.
> **الأطراف:** المشغّل (Operator) · الواجهة (UI) · راوتر العقارات (`properties.ts`) · المحرّك (`propertiesEngine`) · المحرّك المالي (`financialEngine`) · المخصصات (`accounting_mappings`) · الإعدادات (`system_settings`) · دفتر الأستاذ (`journal_entries / journal_lines`).
> **القاعدة الذهبية المرئية:** كل سهم يصل لـ`journal_entries` يمرّ عبر `propertiesEngine` ثم `financialEngine`. لا اختصارات.

---

## 1) إيجار سكني (`residential_rent`) — قبض دفعة

**العملية:** المشغّل يسجّل قبض إيجار سكني. لا VAT.

```mermaid
sequenceDiagram
    autonumber
    actor OP as المشغّل
    participant UI as واجهة /payments/:id/pay
    participant R as Route (properties.ts:1970)
    participant E as propertiesEngine
    participant F as financialEngine
    participant CM as accounting_mappings
    participant GL as journal_entries + journal_lines

    OP->>UI: paidAmount = 3000
    UI->>R: POST /properties/payments/:id/pay
    R->>R: SELECT … FOR UPDATE rent_payments + contractType
    R->>R: contractType=residential_rent → vatAmount=0
    R->>E: postRentRevenueGL({ id, contractId, propertyId, amount:3000, tenantId })
    par resolveAccountCode على ٣ مفاتيح
        E->>F: resolveAccountCode(rent_receivable, debit, "1200")
        F->>CM: SELECT debitAccountCode FROM accounting_mappings
        CM-->>F: "1131"
        F-->>E: "1131"
    and
        E->>F: resolveAccountCode(rent_revenue, credit, "4100")
        F-->>E: "4121"
    and
        E->>F: resolveAccountCode(vat_output, credit, "2200")
        F-->>E: "2131"
    end
    Note over E: vatAmount=0 → سطر VAT يُحذف
    E->>F: postJournalEntry(ref=JE-RENT-{id}, lines=[<br/>DR 1131 = 3000,<br/>CR 4121 = 3000<br/>])
    F->>F: تأكيد guardTable+guardId (idempotency)
    F->>GL: INSERT journal_entry + 2 journal_lines
    GL-->>F: journalId
    F-->>E: { journalId }
    E-->>R: { journalId }
    R->>R: UPDATE rent_payments SET status='paid', paidDate=…
    R-->>UI: 200 { status: 'paid' }
    UI-->>OP: ✅ تمّ التحصيل
```

**ما تؤكّده رحلة E2E:** عدد سطور القيد = 2، DR=CR=3000، السطر الدائن بحساب 4121 (موجود فعلاً في `verify-property-rent-journey.sh`).

## 2) إيجار تجاري (`commercial_rent`) — قبض دفعة بـVAT (شقّ #2039 b-part)

**العملية:** نفس العملية الأولى، لكن المشغّل يدفع gross (شامل VAT)، والـRoute يقسم.

```mermaid
sequenceDiagram
    autonumber
    actor OP as المشغّل
    participant UI as واجهة /payments/:id/pay
    participant R as Route (properties.ts:1970)
    participant S as system_settings
    participant E as propertiesEngine
    participant F as financialEngine
    participant GL as journal_entries + journal_lines

    OP->>UI: paidAmount = 1150 (شامل VAT)
    UI->>R: POST /properties/payments/:id/pay
    R->>R: SELECT … contractType + companyId
    R->>R: contractType=commercial_rent → يحتاج VAT
    R->>S: getCompanyVatRate(companyId)
    S-->>R: 15 (من system_settings.vat_rate)
    R->>R: net = 1150 / 1.15 = 1000<br/>vat = 1150 - 1000 = 150
    R->>E: postRentRevenueGL({ amount:1000, vatAmount:150 })
    E->>F: resolveAccountCode × 3
    F-->>E: 1131 / 4121 / 2131
    E->>F: postJournalEntry(lines=[<br/>DR 1131 = 1150 (gross),<br/>CR 4121 = 1000 (net),<br/>CR 2131 = 150 (VAT)<br/>])
    F->>GL: INSERT journal_entry + 3 journal_lines
    GL-->>F: journalId
    F-->>E: { journalId }
    E-->>R: { journalId }
    R-->>UI: 200 { status: 'paid' }

    Note over OP,GL: قابل للتحكم بالكامل — معدّل VAT من settings،<br/>حسابات من mappings، لا ثوابت في الكود.
```

**معيار القبول للـb-part:** رحلة E2E تجاري تجد:
- 3 سطور قيد (لا 2).
- مجموع DR = 1150، CR = 1150.
- السطر `2131` بقيمة 150 موجود.

## 3) بيع عقار (`sale`) — استبعاد أصل + مكسب/خسارة (شقّ #2042 b-part)

**العملية:** المشغّل يسجّل بيع مبنى. الأصل يخرج بقيمته الدفترية، النقد يدخل بسعر البيع، الفرق مكسب أو خسارة.

```mermaid
sequenceDiagram
    autonumber
    actor OP as المشغّل
    participant UI as واجهة sale-create.tsx
    participant R as Route POST /sales (جديد)
    participant E as propertiesEngine.postSaleGL
    participant F as financialEngine
    participant GL as journal_entries + journal_lines

    OP->>UI: salePrice=1,725,000 (gross)،<br/>bookValue=1,000,000،<br/>vatAmount=225,000
    UI->>R: POST /properties/sales
    R->>R: INSERT property_sales (sale.id)
    R->>E: postSaleGL({<br/>id, propertyId, buyerId,<br/>salePrice:1725000,<br/>bookValue:1000000,<br/>vatAmount:225000<br/>})
    E->>E: net = 1725000 - 225000 = 1500000<br/>gainOrLoss = 1500000 - 1000000 = +500000<br/>→ مكسب
    par resolveAccountCode × 5
        E->>F: property_sale_receivable / 1130
        E->>F: property_building_asset / 1520
        E->>F: vat_output / 2200
        E->>F: property_sale_gain / 4910
        E->>F: property_sale_loss / 6910
    end
    Note over E: gainOrLoss > 0 → سطر loss يُحذف،<br/>سطر gain يُضاف
    E->>F: postJournalEntry(lines=[<br/>DR receivable = 1,725,000 (gross),<br/>CR building_asset = 1,000,000 (book),<br/>CR vat_output = 225,000,<br/>CR sale_gain = 500,000<br/>])
    F->>F: guardTable=property_sales, guardId=sale.id<br/>(idempotency)
    F->>GL: INSERT 4 سطور
    GL-->>F: journalId
    F-->>E: { journalId }
    E-->>R: { journalId }
    R->>R: UPDATE property_units SET status='sold' (lifecycleEngine)
    R-->>UI: 200 { saleId, journalId }

    Note over OP,GL: مكسب XOR خسارة — لا يجتمعان أبداً.
```

**معيار القبول للـb-part:** رحلة E2E بيع تؤكّد:
- 4 سطور إن وُجدت VAT، 3 سطور بلا VAT.
- DR=CR على كل الأشكال.
- لا سطرَي مكسب وخسارة معاً.
- `property_units.status` يتحوّل إلى `sold` عبر `applyTransition`.

## 4) إدارة بعمولة (`management`) — التحصيل والكشف (شقّ #2043 b-part)

**العملية:** المستأجر يدفع، نقتطع عمولتنا، الباقي مستحق للمالك.

```mermaid
sequenceDiagram
    autonumber
    actor T as المستأجر
    actor OP as المشغّل
    participant UI as واجهة /payments/:id/pay
    participant R as Route (يقرأ commissionPct)
    participant E as propertiesEngine.postManagementCollectionGL
    participant F as financialEngine
    participant GL as journal_entries + journal_lines
    participant ST as صفحة /properties-owner-statement

    T->>OP: يدفع 10,000 إيجار
    OP->>UI: paidAmount=10000
    UI->>R: POST /properties/payments/:id/pay
    R->>R: SELECT contract + ownerId + commissionPct
    R->>R: commission = 10000 × 8% = 800<br/>ownerShare = 10000 - 800 = 9200
    R->>E: postManagementCollectionGL({<br/>rentAmount:10000,<br/>commissionAmount:800,<br/>ownerId, tenantId<br/>})
    par resolveAccountCode × 3
        E->>F: property_cash / 1100
        E->>F: property_owner_payable / 2150
        E->>F: property_management_commission / 4130
    end
    E->>F: postJournalEntry(lines=[<br/>DR cash = 10,000 (clientId=tenant),<br/>CR owner_payable = 9,200 (clientId=owner),<br/>CR commission = 800<br/>])
    F->>GL: INSERT 3 سطور
    GL-->>F: journalId
    F-->>E: { journalId }
    E-->>R: { journalId }
    R-->>UI: 200

    Note over GL,ST: كشف المالك لاحقاً يقرأ مباشرة<br/>من journal_lines بدون جدول وسيط

    ST->>GL: SELECT SUM(credit) FROM journal_lines<br/>WHERE accountCode='2150' AND clientId=ownerId<br/>AND period BETWEEN …
    GL-->>ST: 9,200 (للشهر)
    ST-->>OP: كشف يوقّعه المالك
```

**معيار القبول للـb-part:**
- 3 سطور قيد، dimensions صحيحة (ownerId على سطر liability فقط، tenantId على cash).
- `commission = 0` يحذف سطر الـcommission (لا zero-amount).
- كشف المالك = aggregate من `journal_lines` لا جدول منفصل.

## 5) صيانة على المالك (`management` + ownerType=external) — شقّ #2044 b-part

**العملية:** صيانة في عقار نديره لمالك خارجي. التكلفة عليه (لا علينا)، فاتورة ضريبية له.

```mermaid
sequenceDiagram
    autonumber
    actor TC as فنّي
    participant UI as تفاصيل بلاغ صيانة
    participant R as Route /complete (properties.ts:2521)
    participant B as property_buildings
    participant E as propertiesEngine
    participant F as financialEngine
    participant GL as journal_entries + journal_lines
    participant FIN as وحدة المالية (event listener)

    TC->>UI: إغلاق البلاغ + تكلفة 4000 + VAT 600
    UI->>R: POST /maintenance-requests/:id/complete
    R->>B: SELECT ownerType, ownerId FROM property_buildings
    B-->>R: ownerType='external', ownerId=200
    R->>R: قرار التوجيه:<br/>external → owner-billing
    R->>E: postMaintenanceOwnerBillingGL({<br/>id, propertyId, ownerId:200,<br/>totalCost:4000, vatAmount:600<br/>})
    par resolveAccountCode × 3
        E->>F: property_owner_receivable / 1140
        E->>F: property_maintenance_payable / 2160
        E->>F: vat_output / 2200
    end
    E->>F: postJournalEntry(<br/>sourceKey='property:maintenance_owner:{id}',<br/>lines=[<br/>DR owner_receivable = 4,600 (gross, clientId=owner),<br/>CR maintenance_payable = 4,000 (net),<br/>CR vat_output = 600<br/>])
    F->>GL: INSERT 3 سطور
    F-->>E: { journalId }
    E->>FIN: emit('property.invoice.requested', {<br/>clientId:200,<br/>subtotal:4000, vatAmount:600, total:4600<br/>})
    Note over FIN: مستمع وحدة المالية ينشئ فاتورة draft
    E-->>R: { journalId }
    R-->>UI: 200 { status:'completed', invoiceRequested:true }

    Note over R,GL: إن كان ownerType='internal' (شركتنا)،<br/>التوجيه يذهب لـpostMaintenanceExpenseGL القديمة<br/>(DR maintenance_expense / CR payable) بدون فاتورة للمالك.
```

**معيار القبول للـb-part:**
- قرار التوجيه يقرأ `ownerType` فعلاً.
- `sourceKey` مختلف بين المسارَين (`property:maintenance:` vs `property:maintenance_owner:`) — فإعادة تصنيف بلاغ بعد إغلاقه تكتب JE جديداً بدون اصطدام idempotency.
- فاتورة المالك تُنشأ في وحدة المالية عبر event.

## 6) الأبعاد الموحّدة عبر الأربعة

كل سطر قيد ينتجه أي شقّ من الأربعة يحمل:

| البعد | المعنى | لمن |
|---|---|---|
| `propertyId` | معرّف المبنى (`property_buildings.id`) | كل السطور |
| `contractId` | معرّف العقد المرتبط | حيث ينطبق |
| `unitId` | معرّف الوحدة (`property_units.id`) | الصيانة والإيجار |
| `clientId` | المستأجر / المشتري / المالك | حسب الاتجاه: المستأجر على receivable الإيجار، المالك على owner_payable، المشتري على receivable البيع |

**النتيجة:** تقرير 360 لأي طرف (مالك، مستأجر، مشترٍ) يستخرج كل حركاته من `journal_lines` بـSELECT واحد filter بـ`clientId`.
