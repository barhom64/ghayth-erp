# [P0] جرد وتنظيم وإصلاح مسار **المستودع** — نظام غيث

> **النطاق:** المستودع فقط (المشاريع والعقارات لوكلائهما — لا تُلمس ملفاتهما). المالية تُذكَر **بقدر الارتباط المحاسبي** فقط.
> **المصدر:** مولَّد من الكود الفعلي على `main` ومُتحقَّق حيًّا (`verify-warehouse-journey.sh` = 25/25 بحساب غير-admin) + اختبارَي تكامل dynamic. لا افتراض.
> **العقيدة الحاكمة:** المستودع يدير **الكميات والسجلّات**؛ **التقييم/COGS في المالية عبر المحرك**؛ كل منطق حالة/ترحيل/سياسة عبر محرك مركزي لا داخل المسار؛ لا تكرار ماستر بيانات.

يغطّي هذا الملف الأقسام الـ12 المطلوبة للمستودع.

---

## 0. الخلاصة التنفيذية
مسار المستودع **مكتمل وظيفيًا ومُثبَت** بعد 10 PRs (#2014، #2020، #2037، #2040، #2053، #2063، #2065، #2075 + harness #2005 + migration 299). **صفر wiring-stubs مستودعية**. الأثر المحاسبي يمرّ حصرًا عبر `warehouseEngine.postMovementGL` (المالية خادم).

**المتبقّي ليس بناءً جديدًا بل «وصل seams» وتنظيم:** أبرزها **انفصال تتبّع الدفعات (lots) عن مسار استهلاك الحركة** (الحركات تخصم `warehouse_stock_batches` لا `warehouse_stock_lots`)، و**ازدواج المورد/المُورّد** (`suppliers` مستودع مقابل vendors مالية)، و**موضع التقارير** (داخل تبويب «المتقدّم» لا تحت سطح تقارير قانوني)، و**ازدواج مفهوم الجرد** (`inventory_counts` العادي مقابل `warehouse_cycle_counts` الدوري).

## 2. الجرد التفصيلي (من الكود)

### 2.1 الصفحات (FE) — 14 مسارًا مسجّلًا في `miscRoutes.tsx`
| الصفحة | الملف | السطح |
|--------|------|-------|
| لوحة المستودع (4 تبويبات: منتجات/حركات/فئات/موردون) | `pages/warehouse.tsx` | D |
| المتقدّم (جرد دوري/دفعات/تسلسلات/ABC/تقارير) | `pages/warehouse-advanced.tsx` | D+G (مختلط — §8) |
| الجرد (جلسات + تسجيل) | `pages/warehouse/inventory-count.tsx` | C/D |
| إنشاء منتج/حركة/فئة | `pages/create/warehouse*` | C |
| تفاصيل منتج/حركة/فئة/مورد | `pages/details/warehouse-*-detail.tsx` (×4) | D (360) |

### 2.2 الحركات
أنواع: `in · out · return · transfer_in · transfer_out · adjustment_in · adjustment_out`. استلام/صرف بـFIFO على `warehouse_stock_batches` + WA cost + GL. تحويل ساقان بلا GL. سياسات: `warehouse.require_movement_reference` · `warehouse.auto_purchase_request_on_min_stock`. معاينة الأثر `POST /movements/impact-preview` (قراءة بحتة).

### 2.3 الجرد والتسويات
- عادي `inventory_counts`: جلسة → تسجيل فرق → اعتماد (lifecycle) → تسوية ذرّية + GL.
- دوري `warehouse_cycle_counts`: pending→in_progress→reviewed→approved→post (ترقيم مركزي + variance GL + ختم `adjustmentJournalEntryId` idempotent) + خطط + cron.

### 2.4 الموردون
`suppliers` + CRUD، **مربوط وقت-الإنشاء بـParty master** (`registerEntityParty`، migration 249؛ ضمن `PARTY_SOURCES`).

### 2.5 المخزون المتقدّم
الدفعات `warehouse_stock_lots` (QC pending→approved/rejected؛ استدعاء بسبب) · التسلسلات `warehouse_stock_serials` (سجلّ + PATCH حالة) · ABC `product_abc_classification` (Pareto كسول مع cache) · 3 تقارير.

### 2.6 الصلاحيات
مفتاحان: `warehouse.inventory` (43 موضعًا) · `warehouse.transfers` (5، فيها `approve`). المستودع لا يقيّم.

### 2.7 المحركات المركزية
`postMovementGL` ×3 · `issueStock` · `issueNumber` ×5 · `lifecycleEngine.applyTransition` ×8 · `registerEntityParty` ×2. لا ترحيل/حالة خارج المحرك (guard: `audit:boundaries`/`numbering-coverage`/`event-bus` خضراء).

### 2.8 المطبوعات
لا قوالب طباعة مخصّصة للمستودع؛ CSV عبر المحرك الموحّد فقط (§7).

## 4. التكرارات
- مورد/مُورّد (C2) — أوضح تكرار ماستر (مُخفَّف بالربط بالـParty).
- batches/lots (C4) — تكرار آلية تتبّع الدفعة.
- سطحا تعديل المنتج: `warehouse-product-detail` (كمية/أسعار) مقابل بطاقة المحاسبة في `finance/product-catalog` (حسابات/أبعاد) لنفس `warehouse_products` — **ليس تكرار بيانات** بل تكرار سطح تحرير (مقبول إن وُثّق التقسيم: المستودع=الكمية، المالية=التوجيه).

## 5. العناصر القديمة/الميتة
- صفر wiring-stubs مستودعية (أُزيلت كلها).
- `warehouse_stock_lots.qualityControlStatus` الافتراضي `'approved'` (للـbackfill) بينما المسار الجديد يبدأ `'pending'` — ليس ميتًا لكن يلزم توثيقه.
- لا تقارير/صفحات يتيمة (`audit:routes`/`audit:wiring`: 0 يتيم).

## 7. القوالب والمطبوعات
الحالي: لا قالب مخصّص؛ CSV فقط. المطلوب (P2): قوالب عبر المحرك الموحّد لـ«سند حركة/صرف» و«محضر جرد (عادي+دوري)» و«بطاقة دفعة (QC+صلاحية)»، تُسجَّل في `/reports/print-log`. لا منطق طباعة داخل المسار.

## 8. القوائم والصفحات
`warehouse-advanced.tsx` يخلط C/D (جرد دوري/دفعات) مع G (تقارير) — مخالف «التقرير في G فقط». المطلوب (P2): فصل التقارير لسطح `/warehouse/reports`؛ إبقاء الإدارة في سطحها.

## 9. الصلاحيات
الحالي: مفتاحان + الاعتماد على `approve`. المطلوب (P2): **التكلفة/التقييم خلف `permission-gate`** (حساس — للمالية قراءةً)؛ فصل صلاحية «اعتماد الجرد/QC» عن «إنشاء الحركة» عند الحاجة لنطاق أدق.

## 10. معايير القبول
أساس مقفل → سياق المدخِل → نوع حركة + مرجع صحيحان → صنف مخزني (لا خدمة) → **lot صحيح عند التتبّع (F1)** → محرك/مكوّن مُعاد استخدامه → معاينة أثر → **المستودع لا يقيّم** → ترقيم مركزي → **رحلة حيّة بحساب non-admin** + `build/guard` خضراء بلا override.

## 11. قائمة المهام التنفيذية
1. **F1+F5** (P1): صرف lot-tracked بـFEFO + منع المستدعاة + مصدر حقيقة واحد، مع فصل في الرحلة الحيّة.
2. **F2** (P1): توحيد قراءة المورد عبر Party + منع تكرار الرقم الضريبي.
3. **F3** (P2): سطح `/warehouse/reports` ونقل التقارير الثلاثة.
4. **F7** (P2): قوالب الطباعة.
5. **F4** (P2): وثيقة مصالحة الجردين.
6. **F9** (P2): تكلفة/تقييم خلف `permission-gate` + تدقيق صلاحية الاعتماد.

## 12. التوصية النهائية والأولويات
**ابدأ بـF1+F5 (وصل lots بالحركة)** — الـseam الوحيد ذو الأثر التشغيلي الحقيقي (FEFO/recall/صلاحية لا تُنفَّذ على الاستهلاك اليوم)، والبنية جاهزة فلا محرك جديد. ثم **F2** لإغلاق ازدواج الماستر. الباقي تنظيمي P2 لا يحجب الإنتاج.

> **خارج نطاق المستودع** (لوكلاء آخرين/قرار المالك): المشاريع، العقارات، رحلاتهما، إعداد required-check، و«جرد النظام الشامل» لمهمة لاحقة. هذا الملف محصور في المستودع فقط.
