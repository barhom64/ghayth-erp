> هذه **مراجعة** (جرد + تصنيف + سبب + أثر) — **لا تغيير سلوك**. النتائج مثبَّتة باختبار

## 1) طبقات الحماية (ثلاث بوّابات متتابعة)

| الطبقة | الموضع | الأثر |
|---|---|---|
| **بوّابة الوحدة** `requireModule("operations")` | `routes/index.ts:438` | لا يصل راوتر المشاريع إلا لمن تمنحه `ROLE_DEFAULT_MODULES` وحدة `operations`. غيرهم: 403 قبل أي endpoint. |
| **بوّابة الميزة** `authorize({ feature, action })` | كل endpoint في `routes/projects.ts` | كل مسار يُصرَّح مقابل `projects.list` أو `projects.tasks` بفعلٍ محدّد (view/list/create/update/delete). المالك يمرّ دائمًا (short-circuit). |
| **بوّابة النطاق الصفّي** `isFullAccess` + `assertProjectAccess` + مرشّحات `managerId` | داخل المعالجات | تقييد صفّي: من ليس owner/GM يرى/يعدّل فقط ما يملك صلته به. |

`isFullAccess(scope) = scope.isOwner || OWNER_GM_ROLES.includes(scope.role)` (`projects.ts:364`).

## 2) مصفوفة الصلاحيات لكل دور

| الدور | وحدة operations؟ | عرض/قائمة | إنشاء مشروع | تعديل/حذف مشروع | النطاق الصفّي |
|---|---|---|---|---|---|
| **owner** | ✓ | كل المشاريع | ✓ | ✓ | بلا تقييد |
| **general_manager** (OWNER_GM_ROLES) | ✓ | كل المشاريع | ✓ | ✓ | على مستوى الشركة |
| **projects_manager** | ✓ | **مشاريعه فقط** (`managerId = employeeId`) | ✓ (يُسنَد مديرًا تلقائيًّا) | ✓ **لمشاريعه فقط** | `managerId = employeeId` |
| **employee** | ✓ (self-service) | مشاريعه أو ما له مهمّة فيه | ✗ | ✗ | `managerId = employeeId` أو مهمّة مُسندة |
| بقية المدراء (hr/finance/fleet/property/warehouse/legal/crm/bi/support/branch) | ✗ | ✗ | ✗ | ✗ | يُرفض عند بوّابة الوحدة |

## 3) نقاط الإنفاذ في الكود (سبب + أثر)

| الموضع | السطر | الإنفاذ |
|---|---|---|
| قائمة المشاريع `GET /` | `projects.ts:343-348` | `projects_manager` → `p."managerId" = $` |
| قائمة المشاريع `GET /` | `projects.ts:349-354` | `employee` → `managerId = $ OR id IN (project_tasks WHERE assigneeId = $)` |
| إنشاء `POST /` | `projects.ts:414` | غير (full \| projects_manager) → `ForbiddenError` |
| إنشاء `POST /` | `projects.ts:462` | `projects_manager` يُسنَد مديرًا للمشروع (لا يفوّض خارج فريقه) |
| عرض `GET /:id` | `projects.ts:571-579` | تقييد `managerId` لمدير المشاريع، و(managerId أو مهمّة) للموظف |
| تعديل `PATCH /:id` | `projects.ts:642,647` | منع غير المخوّل + قصر مدير المشاريع على مشاريعه |
| حذف `DELETE /:id` | `projects.ts:771,776` | منع غير المخوّل + قصر مدير المشاريع على مشاريعه |
| كل تعديلات الموارد الفرعية (مراحل/مهام/معالم/مخاطر/تكاليف/BOQ/وحدات) | `assertProjectAccess` (≥15 موضعًا) | قصر صفّي قبل أي طفرة؛ خارج النطاق → 404 «غير مصرح بالوصول» |

`assertProjectAccess` (`projects.ts:374-393`) يطبّق نفس مرشّح `managerId`/المهمّة لغير full-access، فيمنع العبث عبر المشاريع.

## 4) النتائج

**لا تجاوز حرج في النطاق.** التصميم محكم: ثلاث بوّابات متتابعة + قصر صفّي على كل طفرة فرعية.

**ملاحظة واحدة (مقصودة، موثّقة لا مُغيَّرة):**
- في `GET /` و`GET /:id`، يرى الموظف المشاريع التي **يديرها** (`managerId = employeeId`) إضافةً لما له مهمّة فيه (`projects.ts:351,576`). هذا **مقصود**: موظف رُقّي لإدارة مشروع (دون تغيير دوره الأساسي) يجب أن يراه. لا يفتح إنشاء/حذف (ممنوعان للموظف). **القرار: إبقاؤه كما هو** — لا أثر أمني (الإنشاء/التعديل/الحذف محكومة بمرشّح المدير أو ممنوعة).

**فجوة سُدّت:** لم تكن هناك اختبارات RBAC مخصّصة للمشاريع. أُضيف `projectsRbacScope.test.ts` يثبّت العقد أعلاه (بوّابة الوحدة، تعريف full-access، مرشّحات القائمة/العرض/التعديل/الحذف، قصر الإنشاء، إسناد مدير المشاريع، إنفاذ `assertProjectAccess`، وتصريح كل endpoint).
