# مراجعة تخطيط الواجهة والتنقل وهندسة المعلومات — نظام غيث

> مراجعة مبنية على مسح فعلي للكود (4 مساحات استكشاف، 2026-06-23). تُقرأ مع `UX_ACCEPTANCE_GATE.md` و`DYNAMIC_ACTIONS_MATRIX.md` و`DESIGN_SYSTEM_GAPS.md`. الهدف توثيق الواقع وتحديد ما يحتاج إعادة ترتيب — لا إعادة هيكلة في هذه الجولة.

## 1. الخلاصة التنفيذية

طبقة التنقل وهندسة المعلومات في غيث **ناضجة ومحروسة آليًا**. التحسينات المتبقية محصورة وموثّقة وتحتاج غالبًا قرار مالك، لا إصلاحًا تقنيًا عاجلًا.

| المحور | الحالة | الدليل |
|---|---|---|
| مصدر تنقل واحد | ✅ | `components/layout/navigation.registry.ts` (432 مدخلًا، 8 أقسام) |
| تصفية حسب الصلاحية/الدور/الوحدة | ✅ | `sidebar-layout.tsx` (module → feature-flag → role-level → sub-page → perm → route-registry) |
| توحيد الأسماء الرسمية | ✅ جزئي | `navigation.canonical-map.ts` (seed، لم يكتمل) |
| حُرّاس التنقل | ✅ 6 خضراء | `gate:nav`, `gate:labels`, `gate:nav-titles`, `gate:tabs`, `gate:quick-actions`, `gate:page-actions` |
| ترتيب حسب منطق العمل | ✅ غالبًا | HR/المالية/الأسطول أُعيد تنظيمها لمجموعات دورة العمل |

## 2. هيكلة التنقل

- **السجل (Registry):** `navigation.registry.ts` — شجرة `NavSection[]` → `NavItem` بحقول `label/path/icon/module/minRoleLevel/perm/children`. دالة `getNavigationRegistry()` تسطّحها للحُرّاس.
- **خط التصفية** (`sidebar-layout.tsx`, `useFilteredNavSections`): يطبّق recursive walk؛ إذا فُقدت كل أبناء مجموعة تُحذف المجموعة. الترتيب من الأوسع (module) إلى الأضيق (تسجيل المسار).
- **التبويبات الأفقية** مرآة للسجل عبر `ModuleTabsNav` (حارس `gate:tabs`).
- **الإجراءات السريعة** في `sidebar-layout.tsx` (`pageQuickActions`, `resolveQuickActions`) — محروسة بـ `gate:quick-actions` (لا روابط redirect/ميتة).

## 3. هندسة المعلومات — ملاحظات

1. **إعادة التنظيم حسب دورة العمل تمّت فعلًا** (موثّق في تعليقات السجل):
   - HR: 17 مدخلًا مسطّحًا → 9 مجموعات (الموظف → النشاط → الطلبات → المخالفات → الأداء → الرواتب → التقارير → الإعدادات).
   - المالية: → 13 مجموعة حسب الدورة. الأسطول: 38 → 7 مجموعات.
   - **لا حذف مسارات** — تجميع فقط (لا أيتام، لا روابط ميتة).
2. **هيمنة المالية:** 207 مسارًا (~36% من 570). يعكس تعقيد المجال؛ مرشّح لمراجعة تجميع فرعية مستقبلية (ليس عيبًا بذاته).
3. **ديون أسماء موثّقة** (`NAVIGATION_DUPLICATE_INVENTORY.md`) — **قرار مالك**:
   - أسماء متعددة لنفس المسار: `/work-inbox` (صندوق الأعمال / ما ينتظر إجراءاتي)، `/my-space` (مساحاتي/مساحتي)، `/hr/services` (3 أسماء)، `/bi` (3 أسماء).
   - الحل: اعتماد اسم رسمي واحد في `canonical-map` والباقي «أسماء بحث».

## 4. تخطيط الصفحة (Page Layout)

- **قشرة موحّدة:** `PageShell` (عنوان + إجراءات)، `DetailPageLayout`، `PageStateWrapper` (تحميل/فراغ/خطأ).
- **القاعدة المعمارية محترمة:** صفحة أم لكل كيان جذر، التفاصيل تبويبات/صفحات مرتبطة (لا منبثقات) — راجع الدستور (القسم 5).
- **فجوة تخطيط:** بعض صفحات التفاصيل تستخدم بدائل `<Table>` الخام بدل `DataTable` الموحّد (تفصيل في `DESIGN_SYSTEM_GAPS.md`).

## 5. ما يحتاج إعادة ترتيب أو تحسين (مدخلات التقرير النهائي)

| البند | المستوى | القرار |
|---|---|---|
| توحيد الأسماء المتعددة لنفس المسار في `canonical-map` | P2 | قرار مالك (اختيار الاسم الرسمي) |
| مراجعة تجميع فرعي لمسارات المالية (207) | P2 | قرار مالك (تحسين، لا عيب) |
| ترقية صفحات `<Table>` الخام إلى `DataTable` | P2 | دفعات لاحقة مملوكة للمسار |

> لا بند P0/P1 في طبقة التنقل وهندسة المعلومات — الأساس سليم ومحروس.
