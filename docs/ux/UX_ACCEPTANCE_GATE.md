# بوابة قبول تجربة المستخدم — نظام غيث

> هذه الوثيقة تعرّف معيار قبول تجربة المستخدم النهائي في نظام غيث. الهدف أن تصبح تجربة المستخدم قابلة للقياس والمنع قبل الدمج، لا مجرد رأي بصري بعد الإطلاق.

## 1. المبدأ الحاكم

لا يُقبل أي تطوير ينجح تقنيًا ويفشل تشغيليًا أمام المستخدم العربي. القبول النهائي لأي Pull Request أو مسار لا يكون إلا عند تحقق الآتي:

1. الرحلات الحرجة تعمل من البداية إلى النهاية.
2. الواجهة عربية أولًا واتجاهها RTL سليم.
3. لا توجد أخطاء Runtime ظاهرة أو Console/Page errors حقيقية.
4. لا توجد روابط مكسورة أو صفحات 404 في الرحلات الأساسية.
5. كل إجراء تشغيلي له أثر ظاهر: حالة، تقرير، سجل تدقيق، أو حدث.
6. لا يكسر التطوير حدود المسارات أو صلاحيات RBAC.
7. الأداء مقبول على سطح المكتب والجوال.

## 2. درجات القبول

| المستوى | الوصف | القرار |
|---|---|---|
| P0 | رحلة تشغيلية حرجة أو صلاحية أو أثر محاسبي/نظامي | يمنع الدمج فورًا |
| P1 | رحلة مهمة في مسار أساسي | يمنع إطلاق المسار، وقد يمنع الدمج حسب الأثر |
| P2 | تحسين قابلية استخدام أو وضوح | يسمح مع Issue مربوط |
| P3 | ملاحظة شكلية لا تعطل العمل | يسمح مع تحسين لاحق |

## 3. الحد الأدنى للدرجة

يعتمد التقييم من 100 نقطة:

| المحور | الوزن |
|---|---:|
| نجاح الرحلات الحقيقية | 40 |
| الأداء والتحميل والاستجابة | 20 |
| العربية و RTL والجوال | 15 |
| منع الأخطاء وسهولة التعافي | 10 |
| الوصول Accessibility | 10 |
| وضوح الأثر Audit/Event/Report | 5 |

**حد قبول المسار النهائي: 85/100.**

ولا يُقبل أي مسار مهما كانت درجته إذا وُجد أحد الآتي:

- فشل رحلة P0.
- كسر صلاحية أو ظهور بيانات خارج النطاق.
- إجراء بلا أثر قابل للتتبع.
- نص إنجليزي ظاهر في صفحة تشغيلية عربية أساسية.
- خطأ Runtime مؤثر في صفحة تشغيلية.

## 4. تعريف الرحلة المقبولة

الرحلة المقبولة هي التي يستطيع المستخدم إنجازها دون معرفة تقنية ودون الرجوع للمطور، وتشمل:

1. الوصول للصفحة من القائمة أو الرابط.
2. فهم عنوان الصفحة وغرضها.
3. تعبئة النموذج أو اتخاذ القرار.
4. رؤية نتيجة واضحة بعد الحفظ أو الاعتماد.
5. ظهور الأثر في القائمة أو التقرير أو السجل.
6. احترام صلاحيات المسار القائد والمسارات الخادمة.

## 5. ربط البوابة بـ GitHub

يجب أن تكون الفحوص التالية مرشحة كـ Required checks قبل الدمج إلى `main`:

- `build`
- `guard`
- `e2e`
- `ux-gate`
- `arabic-rtl-guard`
- `accessibility`
- `lighthouse`

> ضبط Branch Protection يتم من إعدادات المستودع بعد دمج هذا PR إذا لم تكن صلاحياته متاحة عبر الأداة.

## 6. القاعدة الخاصة بغيث

نظام غيث عربي أولًا، ومساراته مستقلة. لذلك لا يكفي أن تعمل الصفحة؛ بل يجب أن تعمل داخل حدود المسار:

- المسار القائد يقرر.
- المسار الخادم ينفذ خدمة محددة فقط.
- أي محاولة لتمرير منطق قرار لمسار خادم تعد فشلًا معماريًا وتجربة مستخدم خطرة.
- كل إجراء يجب أن يترك أثرًا قابلًا للتتبع: من فعل؟ متى؟ لماذا؟ ماذا تغير؟ أين؟

## 7. مخرجات كل PR

يجب أن يحتوي وصف PR أو تقرير CI على:

- قائمة الرحلات التي تم اختبارها.
- نتيجة العربية و RTL.
- نتيجة الجوال.
- أخطاء Console/Page errors إن وجدت.
- الصفحات البطيئة.
- مشكلات الوصول.
- الأثر التشغيلي لكل إجراء.
- الدرجة النهائية.
