# مصفوفة اختبارات تجربة المستخدم — نظام غيث

هذه المصفوفة تحدد الرحلات التي تُقاس في GitHub Actions وفي الاختبار اليدوي قبل اعتماد أي مسار.

## 1. مستويات الرحلات

| المستوى | التعريف | مثال | القرار عند الفشل |
|---|---|---|---|
| P0 | رحلة لا يعمل المسار بدونها | إنشاء موظف، إنشاء فاتورة، إغلاق رحلة | يمنع الدمج أو الإطلاق |
| P1 | رحلة مهمة لكنها ليست أصل تشغيل المسار | فلترة، تصدير، تفاصيل | يمنع الإطلاق النهائي |
| P2 | تحسين استخدام | ترتيب أعمدة، اختصار | Issue لاحق |
| P3 | ملاحظة شكلية | تباعد، نص مساعد | Issue لاحق |

## 3. صفحات الفحص الآلي الأساسية

هذه الصفحات يجب ألا تعطي 404 أو Runtime error، ويجب أن تعرض واجهة عربية RTL:

- `/`
- `/employees`
- `/employees/create`
- `/hr/leaves`
- `/hr/leaves/create`

> إذا تغيرت المسارات، يجب تحديث هذا الملف واختبارات `@ux-gate` في نفس PR.

## 4. تعريف النجاح اليدوي

كل رحلة يدوية تُنفذ بهذه الطريقة:

1. أعط المستخدم مهمة واحدة واضحة.
2. لا تشرح له مكان الزر.
3. سجّل الوقت من بداية المهمة حتى ظهور الأثر.
4. سجّل عدد مرات التردد أو الرجوع أو السؤال.
5. سجّل الخطأ الأول الذي وقع فيه.
6. اسأله بعد المهمة: هل فهمت النتيجة؟ وهل تعرف أين تجد الأثر لاحقًا؟

## 5. مؤشرات الاختبار اليدوي

| المؤشر | القبول |
|---|---|
| نجاح المهمة | 90% للرحلات P0 |
| الزمن | لا يتجاوز الزمن المرجعي المتفق عليه لكل رحلة |
| التردد | لا يزيد عن مرتين في الرحلة P0 |
| فهم الأثر | المستخدم يعرف أين ظهرت النتيجة |
| الرضا | SUS لا يقل عن 68، والهدف لغيث 85+ |

## 6. مبدأ عدم خداع القياس

يمنع اعتبار الرحلة ناجحة إذا:

- احتاج المستخدم إلى شرح خارجي.
- نجح الحفظ لكن لم يظهر الأثر.
- ظهرت رسالة نجاح كاذبة.
- كان الوصول للصفحة ممكنًا فقط بالرابط المباشر وليس من تجربة تشغيلية واضحة.
- كانت الصلاحية مفتوحة أكثر من اللازم.
