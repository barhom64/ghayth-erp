## 2. الحدود المعمارية (يُمنع أثناء #2140)

- نقل منطق أعمال بين المسارات، أو جعل مسار خادم قائداً.
- تغيير عقود API بلا ضرورة موثقة.
- واجهة شكلية فوق API غير مكتمل، أو إظهار خدمة ناقصة بلا حالة واضحة.
- نمط واجهة جديد بدل القالب/الشقيقة الموحدة.
- تحسينات جانبية خارج نطاق الشريحة.
- أي حذف APIs/صفحات (مؤجل لمهمة #2147).

## 6. اقتراح دمج الشرائح 2+3+4 (يحتاج قرارك قبل التنفيذ)

**الاقتراح**: شريحة واحدة «مستندات الموردين» بصفحة واحدة بثلاثة تبويبات.

**السبب**: الثلاثة مرايا AP متطابقة البنية (نفس نموذج الترحيل والتسلسل والجمهور
— موظف الذمم الدائنة)، ودورتها واحدة: شراء ← دفع مسبق/فوترة/إرجاع، واثنتان
منها تشتركان في نفس حوار «التطبيق على أمر شراء» حرفياً. فصلها = ثلاث صفحات
شبه متطابقة تخالف هدف الموجة الأولى (لا تشتت ولا تكرار).

**الأثر**: PR واحد (~600 سطر صفحة + ربط + اختبار) بدل ثلاثة.

**المخاطر**: حجم مراجعة أكبر لمرة واحدة؛ يقابلها أن كل تبويب مستقل وظيفياً
ويُختبر على حدة داخل التقرير نفسه.

**البديل إن رفضت**: ثلاث PRs بنفس الملف يتوسع تبويباً تبويباً.
