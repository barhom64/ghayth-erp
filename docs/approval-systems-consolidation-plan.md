# خطة توحيد نظامَي الموافقة — مقترح بانتظار قرار إبراهيم

> **الحالة: مقترح للقرار — لم يُنفَّذ.** تغيير معماري يمسّ تدفّقات موافقة حيّة،
> فالدستور يوجب التوقّف وعرض الخطة قبل أي تنفيذ. هذا تحليل قراءة-فقط.

## المشكلة (مُتحقَّق منها بالكود)

في الـbackend **نظاما موافقة متعايشان**، وبعض أنواع الطلبات **تُعالَج في الاثنين معًا**:

| نوع الطلب | `initiateApprovalChain` (ب) | `submitWorkflow` (أ) | الملف |
|---|---|---|---|
| سلفة موظف | ✓ سطر 419 | ✓ سطر 429 | `hr-loans.ts` |
| وقت إضافي | ✓ سطر 402 | ✓ سطر 412 | `hr-overtime.ts` |
| نهاية خدمة | ✓ سطر 411 | ✓ سطر 421 | `hr-exit.ts` |
| طلب شراء | ✓ سطر 576 | ✓ سطر 590 | `finance-purchase.ts` |

أي السلف/الإضافي/الخروج/المشتريات تُنشئ **سجلَّي موافقة** في نظامين منفصلين بلا نقطة وصل — خطر عدم تزامن + ازدواج صندوق الوارد.

- **النظام أ — `workflowEngine`** (`workflow_*`): تعريفات ثابتة متعدّدة الخطوات، يدعم **التصعيد/الإحالة/الإعادة/SLA**. مُغذّى من 6 ملفات.
- **النظام ب — `approval_chains`** (`approval_*`): سلاسل ديناميكية **حسب المبلغ** (tiered)، **بلا** تصعيد/إحالة/إعادة. مُغذّى من 8 ملفات.

## مقارنة القدرات

| القدرة | workflowEngine (أ) | approval_chains (ب) |
|---|---|---|
| موافقة/رفض، متعدّد خطوات، تدقيق، إشعارات | ✓ | ✓ |
| تصعيد / إحالة / إعادة للطالب | ✓ | ✗ |
| SLA + اعتماد تلقائي عند التأخّر | ✓ | جزئي |
| توجيه حسب المبلغ (tiered) | ✗ | ✓ |

## التوصية

**التوحيد على `workflowEngine`** (الأكمل وظيفيًّا)، مع **إضافة توجيه حسب المبلغ** إليه (العتبات `minAmount/maxAmount` على `workflow_definitions`) لاستيعاب ميزة النظام ب الوحيدة المتقدّمة. ثم إيقاف الكتابة المزدوجة تدريجيًّا.

## أكبر 3 مخاطر (تتطلّب احترازات)

1. **هجرة الطلبات الجارية**: طلبات معلّقة الآن في `approval_requests` قد تفقد سياق خطوتها عند التحويل → يلزم سكربت هجرة + تحقّق + أرشفة، وفترة كتابة-مزدوجة.
2. **كسر تدفّق حيّ**: طلب في منتصف سلسلة قد لا يتطابق مع تعريف workflow الجديد → اختبار cutover على staging بسيناريوهات حقيقية + تخطيط خطوات 1:1.
3. **تضارب الصلاحيات**: `approval_authorities` (حسب المبلغ) قد لا تُحترم في `workflow_definitions` → دمج فحص الصلاحية في `approveWorkflow()`.

## المسار المقترح (دفعات، كلٌّ باعتماد)

> كلٌّ يحتاج migrations و/أو يمسّ تدفّقات حيّة ⟶ **اعتماد إبراهيم + مجلس ghayth-review قبل كل دفعة.**

1. توسعة workflowEngine بتوجيه المبلغ (migration: أعمدة عتبات).
2. سكربت هجرة تعريفات `approval_chains → workflow_definitions` (اختبار على dev).
3. كتابة مزدوجة مؤقّتة + تحقّق تطابق (shadow mode).
4. هجرة الطلبات المعلّقة دفعةً دفعة.
5. Cutover: جعل workflowEngine المسار الوحيد + مراقبة 48 ساعة.
6. أرشفة `approval_*` + توثيق الإيقاف.
