# تدفّق الإنشاء السريع للمستخدم — USER_QUICK_CREATE_FLOW

> المرحلة 3 — تنفيذ **#1413 §6** · 2026-05-29 · فرع `claude/ghaith-foundation-audit-wdIUf`
> **الهدف:** زر واحد "إنشاء موظف + حساب مستخدم + أدوار" بخطوة سهلة لغير التقني — يعالج RBAC-002.
> **يُبنى على:** النقاط الموجودة (`POST /employees`, `POST /admin/users`, `POST /admin/user-roles`) — يغلّفها بمعاملة ذرّية، لا يستبدلها.

---

## 1. المشكلة الحالية

الإنشاء اليوم ثلاث نقاط منفصلة (موظف → مستخدم → إسناد دور) بلا معاملة موحّدة ولا واجهة واحدة → تجربة صعبة، وخطر حالة جزئية (موظف بلا حساب/حساب بلا دور).

## 2. التدفّق المقترح (واجهة واحدة)

```
[إنشاء موظف + حساب مستخدم]  (زر واضح بالعربية)
   1. الاسم        2. الجوال      3. البريد
   4. الشركة       5. الفرع       6. الإدارة       7. المنصب
   8. الأدوار (اختيار قالب جاهز أو أكثر — متعدد)
   9. النطاق لكل دور (self/branch/company…)
  10. دعوة بالبريد  أو  كلمة مرور مؤقتة
        │
        ▼
  POST /admin/onboard  (معاملة ذرّية واحدة — جديدة)
        │
        ├─ INSERT employee + employee_assignment
        ├─ INSERT user (مرتبط بالموظف)
        ├─ INSERT rbac_user_roles[] (دور لكل اختيار + نطاق)
        ├─ (دعوة بريد أو كلمة مرور مؤقتة)
        └─ createAuditLog("user.onboarded", بالصفة)  + event "user.created"
        │
        ▼
   ملخص واضح: "هذا المستخدم يستطيع: … في النطاق: …"  (من Effective Permissions)
```

## 3. عقد النقطة الجديدة `POST /admin/onboard`

| العنصر | القيمة |
|---|---|
| الصلاحية | `admin.users:create` + `hr.employees:create` (أو دور onboarding) |
| المدخلات | بيانات الموظف + قائمة `{ roleKey, scope, branchId?, departmentId?, expiresAt? }` |
| الذرّية | معاملة واحدة — إما الكل أو لا شيء (لا حالة جزئية) |
| الأثر | موظف + حساب + أدوار + تدقيق + حدث + (دعوة/كلمة مرور) |
| التراجع | فشل أي خطوة → ROLLBACK كامل |

## 4. تجربة الأدوار لغير التقني

- **قوالب جاهزة:** اختيار "محاسب إنشاء" / "مدير فرع" / "مسؤول عمرة" (من `is_template` roles).
- **تخصيص عند الحاجة فقط:** بعد القالب، تعديل دقيق اختياري.
- **ملخص بالعربية:** "أحمد يستطيع: إنشاء مصروفات (فرع مكة)، عرض تقارير مالية (الشركة)، طلب إجازته."

## 5. القرارات

- **يُبنى `POST /admin/onboard`** كغلاف ذرّي فوق النقاط الموجودة (لا يستبدلها — تبقى للاستخدام المنفصل).
- **الواجهة** تستخدم `PageShell`/`FormShell` الموجودة + قوالب الأدوار.
- **اختبارات القبول:** `RBAC_MULTI_ROLE_ACCEPTANCE_TESTS` اختبار 1 و2 (دور واحد / 3 أدوار).
