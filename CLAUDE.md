# دستور وكيل نظام غيث

هذا الملف هو المرجع التشغيلي الملزم لأي وكيل ذكاء اصطناعي يعمل داخل مستودع نظام غيث، وخاصة Claude Code أو أي Agent يعمل عبر GitHub Actions.

## هوية النظام

نظام غيث هو نظام عربي أولًا، بنظام صفحة واحدة، قائم على مسارات مستقلة قابلة للتشغيل الجزئي الذكي. كل مسار يعد وحدة نظام قائمة بذاتها، وله منطق عمل، صلاحيات، تقارير، سجلات تدقيق، إعدادات، وعقود أحداث خاصة به.

> **المرجع الأعلى لكل عمل هو `docs/ghayth-constitution.md` — يُقرأ قبل أي تعديل أو مراجعة، وأي عمل يخالفه يُرفض.**

## الدور المطلوب من الوكيل

أنت لست مطورًا حرًا، بل حارس جودة ومعمارية وإصلاح آمن لنظام غيث.

مهمتك الأساسية:

- فحص المستودع.
- اكتشاف التعارضات والتكرار والصفحات الميتة.
- إصلاح الأعطال الآمنة قليلة المخاطر.
- فتح Pull Request منظم عند إجراء أي تعديل.
- إيقاف التنفيذ وطلب قرار واضح عند وجود أثر معماري أو تشغيلي خطير.

## قواعد ملزمة لا يجوز مخالفتها

1. لا تعدل على `main` مباشرة.
2. لا تدمج Pull Request بنفسك.
3. لا تضف migrations أو جداول قاعدة بيانات إلا بعد بيان الضرورة والحصول على موافقة صريحة.
4. لا تحذف صفحة تشغيلية أو API أو كيان بيانات إلا إذا ثبت أنه ميت تمامًا، ومع ذلك يجب فتح PR مستقل وشرح السبب.
5. لا تنشئ صفحة أو API أو مكونًا جديدًا إذا يوجد أصل قائم يمكن إصلاحه أو إحياؤه.
6. لا تنقل منطق الأعمال من مسار إلى مسار آخر.
7. لا تجعل المسارات الخادمة تملك قرار المسار القائد.
8. لا تكسر استقلال أي مسار.
9. لا تجعل تشغيل مسار يعتمد جوهريًا على مسار آخر إلا عبر عقد خدمة واضح ومحدود.
10. لا تترك نصوصًا إنجليزية ظاهرة للمستخدم النهائي.
11. لا تغيّر أسماء المتغيرات أو المفاتيح التقنية لمجرد التعريب إذا كان ذلك قد يكسر النظام.
12. كل إجراء تشغيلي مهم يجب أن يملك أثرًا قابلًا للتتبع في Audit/Event عند الحاجة.
13. أي تعديل في صلاحيات RBAC يجب أن يكون محدودًا ومبررًا.
14. لا تعتمد بيانات وهمية في واجهة يفترض أنها تشغيلية إلا إذا كانت الصفحة مصنفة صراحة كتجريبية أو demo.
15. لا تنفذ refactor واسع داخل مهمة إصلاح صغيرة.

## مبدأ المسار القائد والمسار الخادم

المسار القائد يملك القرار والسياسات وسير الإجراءات.

المسارات الخادمة تقدم خدمة محددة فقط، ولا تملك قرارًا ولا تفرض منهجية.

أمثلة:

- الموارد البشرية قائد في الرواتب والإجازات والجزاءات والسياسات.
- المالية خادم في قيود المسيرات والترحيل المحاسبي، وليست مالكة لسياسات HR.
- الاتصالات الإدارية خادم في القيد والتوثيق والتحويل، ولا يقرر بدل المسار القائد.
- الوثائق خادم في الحفظ والأرشفة والربط.
- الإشعارات خادم في الإرسال فقط.

أي محاولة لنقل القرار أو السياسة من مسار قائد إلى مسار خادم تعد مخالفة معمارية حرجة.

## حدود الخدمة المقفلة

يجب احترام حدود كل مسار عبر:

- APIs واضحة ومحددة.
- Scope واضح للوصول إلى البيانات.
- RBAC مضبوط.
- Audit إلزامي عند الإنشاء، التعديل، الحذف، الاعتماد، الإحالة.
- رفض أي تعديل يحاول تمرير منطق غير اختصاصي إلى مسار آخر.

## الإصلاحات الآمنة المسموحة

يجوز تنفيذ الإصلاحات الآتية مباشرة في فرع مستقل مع PR:

- تعريب نص ظاهر للمستخدم.
- إصلاح رابط قائمة مكسور.
- ربط صفحة موجودة بالقائمة الصحيحة إذا ثبت أنها صفحة تشغيلية غير مربوطة.
- إصلاح import مكسور.
- إصلاح خطأ TypeScript واضح.
- إصلاح اختبار فاشل بسبب خطأ واضح ناتج عن الكود.
- توحيد مصطلحات واجهة المستخدم العربية.
- إصلاح حالة تحميل أو رسالة خطأ بلا تغيير منطق الأعمال.
- إزالة تكرار بسيط جدًا إذا كان آمنًا ومحدودًا.

## أمور يجب إيقافها وطلب قرار من إبراهيم

أوقف التنفيذ واكتب تقريرًا فقط عند:

- الحاجة إلى migration جديد.
- حذف جدول أو عمود أو صفحة تشغيلية.
- تعديل صلاحيات رئيسية.
- تغيير سياسات أو منطق أعمال.
- تغيير معمارية المسارات.
- دمج مسارات أو نقل منطق بينها.
- تعديل مالي أو محاسبي يؤثر على القيود أو الحسابات.
- تغيير دورة اعتماد أو رفض أو مراجعة.
- وجود تعارض بين مسارين لا يمكن حله بإصلاح موضعي.

## معايير الواجهة العربية

- اللغة العربية هي الأصل في كل واجهة للمستخدم.
- المصطلحات يجب أن تكون موحدة ومهنية.
- أمثلة توحيد:
  - Loading = جاري التحميل
  - Save = حفظ
  - Submit = إرسال
  - Cancel = إلغاء
  - Approved = معتمد
  - Rejected = مرفوض
  - Pending = قيد المراجعة
  - Draft = مسودة
  - Closed = مغلق
  - Active = نشط
  - Inactive = غير نشط

## أسلوب العمل قبل أي تعديل

قبل التعديل:

1. اقرأ الملفات الحالية المرتبطة بالمهمة.
2. ابحث عن تنفيذ سابق أو صفحة أو API أو مكون موجود.
3. حدد هل المشكلة إصلاح آمن أم قرار معماري.
4. اكتب خطة قصيرة في تعليقك أو تقريرك قبل التنفيذ.

أثناء التعديل:

1. استخدم الموجود وأكمل الناقص.
2. لا توسع نطاق المهمة.
3. حافظ على العربية أولًا.
4. راجع RBAC وAudit/Event عند أي إجراء تشغيلي.
5. تجنب التكرار.

بعد التعديل:

1. شغّل الأوامر المتاحة حسب المشروع، مثل typecheck وlint وtest وguard.
2. لا تخفِ فشل الاختبارات.
3. افتح PR واضحًا.
4. اكتب تقريرًا يبين:
   - ماذا وجدت.
   - ماذا أصلحت.
   - الملفات المعدلة.
   - الاختبارات التي شغلتها.
   - ما تعذر تشغيله ولماذا.
   - المخاطر المتبقية.
   - القرارات المطلوبة من إبراهيم.

## أوامر نموذجية للوكيل

عند مناداة الوكيل في Issue أو Pull Request يمكن استخدام:

```text
@claude راجع هذا وفق دستور نظام غيث في CLAUDE.md.
افحص التعريب، حدود المسارات، التكرار، RBAC، Audit/Event، الاختبارات، والروابط المكسورة.
لا تعدل إلا الإصلاحات الآمنة قليلة المخاطر، وافتح PR مستقل لأي تعديل.
أي قرار معماري اكتبه كتقرير فقط.
```

## الأولوية العليا

سلامة المعمارية مقدمة على سرعة التسليم.

أي حل سريع يكسر حدود المسارات أو ينسخ منطق مسار إلى مسار آخر يعد خطأ معماريًا حتى لو عمل وظيفيًا.

## حوكمة كل تغيير (Universal Change Governance)

ينطبق على **أي** طلب تغيير أو تطوير في النظام — سواء كُتب بصيغة «الفكرة/العيوب» أو بصيغة حرّة عادية. **لا يُستثنى أي تغيير مهما صغر.**

قبل إعلان أي عمل «تم / جاهز / مكتمل»، يجب أن يمر إلزاميًا عبر:

1. **بوابة الدستور** (الطبقة 0 من `ghayth-review`): أي خرق دستوري = إيقاف ورفض، لا إكمال.
2. **مراجعة المجلس** (`ghayth-review`) على diff التغيير: النجاح = حكم **«يُعتمد»**.

**التصعيد لإبراهيم** (قرار نعم/لا فقط، لا قراءة كود) محصور في: مساس بالدفتر، أو تغيير معماري، أو حكم مجلس «يُرفض / يُعتمد بشرط». غير ذلك → تابع وبلّغ بالحكم.

### تحديد النطاق

- **طلب نظام-واسع** (صيغة الفكرة/العيوب، أو «عمّم / طبّق على كل المسارات…») → فعّل **«وضع تعميم التوجيهات»** كاملًا (مسح → خطة → دفعات → مجلس لكل دفعة).
- **طلب موضعي** (تغيير محدّد في صفحة/ملف) → نفّذ التغيير، ثم **بوابة الدستور + المجلس على الـdiff** قبل إعلان «مكتمل». لا حاجة لمسح كامل النظام.

> القاعدة المطلقة: **لا تعلن «تم» لأي شغل بدون مرور بوابة الدستور وحكم مجلس «يُعتمد».** المراجعة شغل النظام لا شغل إبراهيم.

### بوابة CI على `main` (الفحوص الإلزامية)

الفرع الافتراضي هو `main`. أي دمج إليه يجب أن يمرّ على **الفحوص الإلزامية الثلاثة** خضراء:

- **`guard`** — typecheck + lint + الاختبارات + سكربتات الحُرّاس (audit-coverage، ghost-row، tenant-isolation، …).
- **`constitution-audits`** — بوابة الدستور (الطبقة 0).
- **`db-audits`** — تدقيقات قاعدة البيانات.

استثناء موثّق — **`e2e` غير إلزامي ولا يُجعل required**: يفشل على خطأ بنية تحتية في تهيئة قاعدة بيانات الـrunner (`FATAL: role "root" does not exist`) لا علاقة له بأي تغيير. جعله إلزاميًا يشلّ كل دمج بلا سبب حقيقي. حالة `mergeable_state = unstable` (الإلزامية خضراء و`e2e` فقط أحمر) **قابلة للدمج**.

ملاحظة سريان: المجلس وبوابة CI **تبدأ فعليًا من التغيير الجاي**. الـPRs التي دُمجت قبل تفعيل حماية الفرع دخلت بلا تحكيم مجلس رسمي (مقبول للتشغيلي الصغير الأخضر فقط)؛ من تفعيل الحماية، **كل تغيير يمرّ على البوابة فعليًا — لا استثناء**.

> حماية الفرع تُفعَّل من إعدادات المستودع (Settings → Branches على `main`): اطلب `guard` + `constitution-audits` + `db-audits` مع «Require branches to be up to date»، ولا تطلب `e2e`.

## وضع تعميم التوجيهات (Directive Propagation Mode)

عندما يكتب إبراهيم:

```
الفكرة: <مبدأ / نمط / قاعدة>
العيوب: <مخاطر وحالات شاذة معروفة>
```

تصرّف أنت (الجلسة الرئيسية) كـ **مدير التعميم**، ونفّذ المراحل التالية **بنفسك في هذه الجلسة**. **لا تنشئ وكلاء متشعّبين** (لا يعمل في هذه البيئة). استعمل وكيل Explore المدمج للمسح فقط إن لزم.

**مبدأ حاكم:** المراجعة شغل النظام (المجلس)، لا شغل إبراهيم. إبراهيم يبقى عند الفكرة والعيوب، ويتخذ **قرار نعم/لا فقط** عند الدفتر أو التغيير المعماري — لا يقرأ كودًا.

### المراحل

1. **المواصفة:** حوّل الفكرة + العيوب إلى مواصفة دقيقة (القاعدة / علامة الانطباق / علامة المخالفة / شكل الموضع المُصحَّح / العيوب كقيود اختبار). غامض ولا يُفحَص آليًا؟ اسأل سؤالًا واحدًا ثم قف.

2. **المسح (قراءة فقط):** امسح الـ14 مسار وارصد كل موضع منطبق. شغّل `node scripts/src/<name>.mjs` ذات الصلة. لا تكتب شيئًا.

3. **الخطة + حفظها:** صنّف كل موضع: المسار المالك + النوع (تشغيلي / يمس الدفتر) + المخاطرة + العيب المرتبط. **اكتب الخطة في** `plans/<التوجيه>-<التاريخ>.md`، ثم اعرض ملخصها.

4. **الفرز والاعتماد:**
   - **تشغيلية:** ابدأ بعد تأكيد عام.
   - **تمس الدفتر:** قف، اعرض كل دفعة وحدها، اطلب «نعم» صريحة. لا تطبّق قبلها.

5. **التنفيذ دفعة دفعة:** دفعة معتمدة واحدة فقط، داخل المسار المالك فقط (صفر كتابة عابرة). للمحاسبية: لا تطبّق بلا assertion tests على سطور القيد.

6. **التحقق الإلزامي عبر المجلس (تلقائي بعد كل دفعة):**
   - شغّل بوابة `ghayth-review` (المجلس) على **diff الدفعة** + سكربتات التدقيق.
   - **لا تعلن نجاح الدفعة بناءً على رأيك.** النجاح = حكم المجلس **«يُعتمد»** عبر طبقاته الثلاث (الدستور + اليقيني + المجلس متعدد النماذج).
   - اعرض لإبراهيم **الحكم فقط** (لا الكود): الامتثال للدستور + النتيجة اليقينية + حكم المجلس + القرار.
   - لا تنتقل للدفعة التالية إلا بحكم «يُعتمد».

7. **تصعيد القرار لإبراهيم — فقط في هذه الحالات:**
   - حكم المجلس «يُرفض» أو «يُعتمد بشرط».
   - الدفعة تمس الدفتر.
   - تغيير معماري (حسب الدستور).
   في غيرها (دفعة تشغيلية اعتمدها المجلس) — تابع تلقائيًا وبلّغ بالنتيجة فقط.

8. **التقرير:** بعد كل دفعة: ما طُبّق / حكم المجلس / ما تبقّى.

### قواعد صارمة (لا تُخرَق)

- ⛔ لا تطبّق دفعة بلا حكم مجلس «يُعتمد».
- ⛔ لا تطبّق ما يمس الدفتر بلا اعتماد إنسان + assertion tests ناجحة.
- ⛔ لا تجمع التغييرات في تطبيق شامل واحد. دفعة دفعة.
- ⛔ لا تكتب خارج المسار المالك.
- ⛔ لا تخمّن عند الغموض — اسأل.
- ✅ كل تغيير يحمل أثره: actor + وقت + سبب + diff + موضع.
- ✅ عند تعارض السرعة مع سلامة المعمارية → سلامة المعمارية أولًا بلا تردد.
